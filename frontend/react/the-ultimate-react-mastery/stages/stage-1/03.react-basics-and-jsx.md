**JSX** (JavaScript XML) is a syntax extension for JavaScript that allows you to write HTML-like code in JavaScript files.

- JSX compiles to `React.createElement()` calls
- Browser doesn't understand JSX - needs to be transpiled (by Babel/Vite)

**Q6**: Discuss the performance implications of different conditional rendering approaches and when you would optimize.

<details>
<summary>Expected Answer</summary>

**Performance Considerations:**

**1. Ternary vs Logical AND:**

```javascript
// Ternary - always evaluates both branches
{
  condition ? <ComponentA /> : <ComponentB />;
}
// React creates elements for both, but only renders one

// Logical AND - short-circuits
{
  condition && <ComponentA />;
}
// Only creates element if condition is true
```

**Verdict**: `&&` is slightly more performant when only showing/hiding, but difference is negligible in most cases.

**2. Component mounting/unmounting:**

```javascript
// Option 1: Unmount/remount (loses state)
{
  showModal && <Modal />;
}

// Option 2: CSS visibility (keeps in DOM, preserves state)
<Modal style={{ display: showModal ? "block" : "none" }} />;
```

**When to use each:**

- **Unmount/remount**: Default choice, cleaner DOM, prevents memory leaks
- **CSS hide**: When state preservation is critical, frequent toggles, heavy initialization

**3. Early returns vs nested ternaries:**

```javascript
// âŒ Nested ternaries (hard to read, all evaluated)
{
  status === "loading" ? (
    <Spinner />
  ) : status === "error" ? (
    <Error />
  ) : status === "success" ? (
    <Data />
  ) : null;
}

// âœ… Early returns (clearer, potentially faster)
if (status === "loading") return <Spinner />;
if (status === "error") return <Error />;
if (status === "success") return <Data />;
return null;
```

**4. Object mapping for large switch cases:**

```javascript
// âŒ Switch statement (evaluates every case)
switch (type) {
  case "user":
    return <UserIcon />;
  case "admin":
    return <AdminIcon />;
  // ... 50 more cases
}

// âœ… Object lookup (O(1) access)
const ICONS = {
  user: <UserIcon />,
  admin: <AdminIcon />,
  // ... 50 more
};
return ICONS[type];
```

**When to optimize:**

- List with 1000+ items using complex conditions
- Conditions in hot paths (frequent re-renders)
- Heavy components being conditionally rendered
- Profiler shows conditional rendering as bottleneck

**When NOT to optimize:**

- Premature optimization (make it work first)
- Readability suffers significantly
- Real-world performance is acceptable
- No measurable impact from profiling

**Pro tip**: Use React DevTools Profiler to identify actual bottlenecks before optimizing.
