## 🤔 Kiểm tra đầu vào (5 phút)

<details>
<summary><b>1. Object reference equality là gì? <code>{} === {}</code> trả về gì?</b></summary>

**Reference equality** so sánh xem 2 biến có trỏ đến cùng 1 vùng nhớ trong heap hay không, thay vì so sánh nội dung bên trong.

```jsx
{} === {} // → false (2 object khác nhau trong memory)
const a = {}; const b = a; a === b // → true (cùng reference)

// React dùng Object.is() (tương tự ===) để detect changes
// → New object mỗi render = "state changed" = re-render
```

Đây là **root cause** của Context performance issue: mỗi render tạo `value={newObject}` → React nghĩ context đổi → tất cả consumers re-render.

</details>

<details>
<summary><b>2. useMemo và useCallback khác nhau như thế nào?</b></summary>

|                 | `useMemo`                                           | `useCallback`             |
| --------------- | --------------------------------------------------- | ------------------------- |
| **Cache**       | Giá trị (value)                                     | Function reference        |
| **Return**      | Kết quả của function                                | Function đã memoize       |
| **Dùng cho**    | Object, array, computed value                       | Event handlers, callbacks |
| **Tương đương** | `useCallback(fn, deps)` = `useMemo(() => fn, deps)` | —                         |

```jsx
// useMemo → cache object
const value = useMemo(() => ({ user, theme }), [user, theme]);

// useCallback → cache function
const handleClick = useCallback(() => { ... }, [deps]);
```

**Quy tắc:** Dùng `useMemo` cho objects/arrays trong context value, `useCallback` cho functions truyền vào context.

</details>

<details>
<summary><b>3. React.memo làm gì? Khi nào component re-render?</b></summary>

`React.memo` là **Higher-Order Component** ghi nhớ kết quả render, chỉ re-render khi props thay đổi.

```jsx
const MyComponent = memo(function MyComponent({ user }) {
  return <div>{user.name}</div>;
});

// Re-render khi: user prop thay đổi (shallow compare)
// KHÔNG re-render khi: parent re-render nhưng props giống cũ
```

**Shallow comparison:** `prevProps === nextProps` cho từng prop.

**⚠️ Lưu ý:** Context value là "hidden prop" — nếu context value đổi → `memo` không ngăn được re-render!

**Khi nào re-render?**

1. Props thay đổi (shallow compare fail)
2. Context value thay đổi (dù component không dùng context!)
3. Parent re-render + không có `memo`
4. Force update (`setState` trong component)

</details>

## 📖 PHẦN 1: GIỚI THIỆU KHÁI NIỆM (30 phút)

### 1.1 Vấn Đề Thực Tế

Context rất tiện, nhưng có performance cost:

```jsx
/**
 * ❌ PROBLEM: Context Performance Issue
 */

const AppContext = createContext();

function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');
  const [notifications, setNotifications] = useState([]);

  // ❌ NEW OBJECT mỗi render!
  const value = {
    user,
    setUser,
    theme,
    setTheme,
    notifications,
    setNotifications,
  };

  return <AppContext.Provider value={value}>{children}</AppContext.Provider>;
}

// Component chỉ cần user
function UserProfile() {
  const { user } = useContext(AppContext);

  console.log('UserProfile rendered!');

  return <div>{user?.name}</div>;
}

// Component chỉ cần theme
function ThemeToggle() {
  const { theme, setTheme } = useContext(AppContext);

  console.log('ThemeToggle rendered!');

  return (
    <button onClick={() => setTheme((t) => (t === 'light' ? 'dark' : 'light'))}>
      {theme}
    </button>
  );
}

// ❌ VẤN ĐỀ:
// - Change theme → UserProfile re-render (KHÔNG cần!)
// - Change user → ThemeToggle re-render (KHÔNG cần!)
// - Mỗi setState → TẤT CẢ consumers re-render
// - Value object mới mỗi render → Context change detection
```

**Root Causes:**

1. **Context = Broadcast Model (CORE ISSUE)**
   - Khi `value` đổi (reference !==), React re-render ALL consumers
   - Không có selective subscription (không như Redux selector)

2. **God Context (Design Issue)**
   - Nhiều domain (user, theme, todos…) trong 1 context
   - → mọi update lan ra toàn bộ tree

3. **Value Identity Instability (Secondary Issue)**
   - `{}` mới mỗi render → trigger context change ngay cả khi state không đổi
   - → gây re-render KHÔNG cần thiết

4. **Function Identity Instability (Minor Issue)**
   - Functions recreate mỗi render
   - → ảnh hưởng shallow compare / props, KHÔNG phải root cause của context re-render

❗ Không phải tất cả vấn đề đều ngang nhau:

- (1) Broadcast model → KHÔNG fix bằng memo
- (2) God context → FIX bằng architecture
- (3)(4) Identity → chỉ là optimization nhỏ

### 1.2 Giải Pháp

#### ❌ Tránh hiểu sai:

```md
❌ useMemo / useCallback KHÔNG giải quyết context re-render do state change
```

**3 Optimization Strategies:**

```jsx
/**
 * ✅ STRATEGY 1: Memoize Context Value (Defensive Optimization)
 *
 * 👉 Chỉ giải quyết:
 * - Re-render do parent render (không đổi state)
 *
 * ❌ KHÔNG giải quyết:
 * - Re-render khi context state thay đổi
 */

function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');

  const value = useMemo(
    () => ({ user, setUser, theme, setTheme }),
    [user, theme],
  );

  return <AppContext.Provider value={value}>{children}</AppContext.Provider>;
}

/**
 * ✅ STRATEGY 2: Context Splitting
 *
 * 👉 Giảm blast radius của update
 * 👉 Tách domain → isolate re-render
 */

const UserContext = createContext();
const ThemeContext = createContext();

function UserProvider({ children }) {
  const [user, setUser] = useState(null);
  const value = useMemo(() => ({ user, setUser }), [user]);

  return <UserContext.Provider value={value}>{children}</UserContext.Provider>;
}

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const value = useMemo(() => ({ theme, setTheme }), [theme]);

  return (
    <ThemeContext.Provider value={value}>
      {children}
    </ThemeContext.Provider>
  );
}

/**
 * ✅ RESULT:
 *
 * - Change theme → KHÔNG re-render consumer của UserContext
 * - Change user → KHÔNG affect ThemeContext
 *
 * 👉 Từ "broadcast toàn app" → "broadcast theo domain"
 */

// ✅ Bây giờ: Change theme → KHÔNG affect UserProfile!

/**
 * ✅ STRATEGY 3: State + Dispatch Splitting
 *
 * 👉 Tách read vs write
 * 👉 Dispatch KHÔNG bao giờ đổi → stable reference
 */

const StateContext = createContext();
const DispatchContext = createContext();

function CartProvider({ children }) {
  const [state, dispatch] = useReducer(cartReducer, initialState);

  return (
    <StateContext.Provider value={state}>
      <DispatchContext.Provider value={dispatch}>
        {children}
      </DispatchContext.Provider>
    </StateContext.Provider>
  );
}

function useCartState() {
  return useContext(StateContext);
}

function useCartDispatch() {
  return useContext(DispatchContext);
}

/**
 * ✅ RESULT:
 *
 * - Component chỉ dùng dispatch → KHÔNG re-render khi state change
 * - Component đọc state → vẫn re-render (expected)
 */

/**
 * ✅ STRATEGY 4: Context Selector (Advanced) - Dùng Dependencie `use-context-selector`
 *
 * 👉 Subscribe theo slice thay vì toàn bộ context
 */

const count = useContextSelector(AppContext, v => v.count);
```

```
🧠 Hierarchy of solutions:

1. Context Splitting → FIX kiến trúc (quan trọng nhất)
2. Selector pattern → FIX scalable performance
3. State/Dispatch split → tối ưu write-heavy
4. useMemo/useCallback → chỉ là defensive optimization

❗ Nếu chỉ dùng useMemo:
→ Bạn chưa giải quyết root problem
```

### 1.3 Mental Model

```
CONTEXT PERFORMANCE MODEL:

🔴 Core Behavior (Broadcast Model)

Provider { value: NEW_REFERENCE }
    ↓
ALL Consumers using this Context
    ↓
→ Re-render (NO selective subscription)

❗ Trigger:
- Bất kỳ state nào trong value thay đổi
- HOẶC value object bị tạo mới (reference !==)

--------------------------------------------------

❌ Unoptimized (Identity luôn thay đổi)

Provider { value: NEW_OBJECT mỗi render }
    ↓
Consumer A (user)          ← Re-render
Consumer B (theme)         ← Re-render
Consumer C (notifications) ← Re-render

→ Re-render cả khi KHÔNG có state change (wasted render)

--------------------------------------------------

🟡 useMemo (Defensive Optimization)

Provider { value: SAME_OBJECT nếu deps không đổi }
    ↓
Consumers ← KHÔNG re-render (nếu parent render nhưng state không đổi)

NHƯNG:

Provider { value: NEW_OBJECT khi state đổi }
    ↓
ALL Consumers ← VẪN re-render

❗ Kết luận:
useMemo chỉ:
- Fix re-render do parent
- KHÔNG fix broadcast khi state change

--------------------------------------------------

🟢 Context Splitting (Architectural Fix)

UserProvider
    ↓
Consumer A (user) ← Re-render khi user change

ThemeProvider
    ↓
Consumer B (theme) ← Re-render khi theme change

NotificationProvider
    ↓
Consumer C ← Re-render khi notifications change

→ Isolated re-renders theo domain
→ Giảm blast radius của update

--------------------------------------------------

🔵 Advanced: Selector Pattern (Scalable)

Provider { value: BIG_OBJECT }
    ↓
Consumer A subscribes to (user)
Consumer B subscribes to (theme)

→ Chỉ re-render khi slice thay đổi
→ Không cần split context

--------------------------------------------------

📡 Analogy (Corrected):

Single Context = 1 Broadcast Channel
→ Tất cả listeners nhận MỌI signal (dù không cần)

Split Contexts = Multiple Channels
→ Mỗi component subscribe đúng channel của nó

Selector Pattern = Smart Subscription
→ Nghe channel chung nhưng chỉ react khi “đúng nội dung mình cần”

--------------------------------------------------

🧠 Key Insight:

Context không phải:
→ fine-grained state management

Nó là:
→ broadcast mechanism với side-effect là re-render

--------------------------------------------------

🔥 Rule thực chiến:

- Ít state → 1 context OK
- Nhiều domain → PHẢI split
- High-frequency update → tránh Context (dùng external store)

```

### 1.4 Hiểu Lầm Phổ Biến

❌ "Luôn luôn optimize Context"

→ ✅ Premature optimization = sai hướng

- Context bản chất đã đơn giản và đủ nhanh cho đa số case
- Chỉ optimize khi:
  - Có nhiều consumers
  - Có re-render không cần thiết (measured, không đoán)
  - Hoặc state update tần suất cao

👉 Rule:
Measure → Identify bottleneck → THEN optimize

--------------------------------------------------

❌ "useMemo giải quyết mọi vấn đề"

→ ❌ Sai bản chất

- useMemo chỉ:
  ✔ giữ reference ổn định khi KHÔNG có state change
  ❌ KHÔNG ngăn re-render khi context value thay đổi

👉 Nếu gặp:
- Over-render do nhiều domain
→ phải dùng Context Splitting hoặc Selector pattern

--------------------------------------------------

❌ "Context chậm, không dùng production"

→ ❌ Sai

- Context không "chậm"
- Nó chỉ:
  → broadcast updates (design choice)

👉 Context phù hợp khi:
- State ít thay đổi
- Scope nhỏ (theme, auth, config…)

👉 Không phù hợp khi:
- High-frequency updates (real-time, drag-drop, animation…)
- Nhiều consumers cần fine-grained updates

--------------------------------------------------

❌ "Cứ optimize là tốt hơn"

→ ❌ Sai (Optimization có cost)

Cost bao gồm:
- Code phức tạp hơn (split nhiều context)
- Khó trace data flow
- Khó maintain / onboarding

👉 Trade-off:

| Approach | Pros | Cons |
|--------|------|------|
| Single Context | Simple | Over-render |
| Split Context | Better perf | More boilerplate |
| Selector pattern | Best perf | Complexity + lib |

👉 Rule:
Chọn giải pháp **đơn giản nhất mà đáp ứng được scale hiện tại**

--------------------------------------------------

🧠 Mental Model cần nhớ:

- Context = Dependency Injection + Broadcast
- Không phải full state management solution

👉 Nếu cần:
- Fine-grained subscription
- High scalability

→ Dùng:
- Zustand
- Redux Toolkit
- Jotai

--------------------------------------------------

🔥 Kết luận:

- Đừng optimize theo "best practice"
- Optimize theo:
  → scale
  → frequency update
  → số lượng consumer
- Context không có vấn đề về performance — vấn đề là bạn dùng nó cho bài toán nó không được thiết kế để giải.

### Demo 3: Context Splitting Pattern ⭐⭐⭐

```jsx
/**
 * 🎯 Perfect optimization: Split contexts
 * - CountContext
 * - NameContext
 * - Isolated re-renders
 */

import { createContext, useContext, useState, useMemo, memo } from 'react';

// ✅ SPLIT CONTEXTS
const CountContext = createContext();
const NameContext = createContext();

function CountProvider({ children }) {
  const [count, setCount] = useState(0);

  const value = useMemo(() => ({ count, setCount }), [count]);

  console.log('CountProvider rendered');

  return (
    <CountContext.Provider value={value}>{children}</CountContext.Provider>
  );
}

function NameProvider({ children }) {
  const [name, setName] = useState('');

  const value = useMemo(() => ({ name, setName }), [name]);

  console.log('NameProvider rendered');

  return <NameContext.Provider value={value}>{children}</NameContext.Provider>;
}

// Custom hooks
function useCount() {
  const context = useContext(CountContext);
  if (!context) throw new Error('useCount must be used within CountProvider');
  return context;
}

function useName() {
  const context = useContext(NameContext);
  if (!context) throw new Error('useName must be used within NameProvider');
  return context;
}

// Components
const Counter = memo(function Counter() {
  const { count, setCount } = useCount();

  console.log('Counter rendered');

  return (
    <div style={{ padding: '20px', border: '2px solid blue', margin: '10px' }}>
      <h3>Counter Component</h3>
      <p>Count: {count}</p>
      <button onClick={() => setCount((c) => c + 1)}>Increment</button>
      <p style={{ fontSize: '12px', color: '#666' }}>
        ✅ Only re-renders when count changes
      </p>
    </div>
  );
});

const NameInput = memo(function NameInput() {
  const { name, setName } = useName();

  console.log('NameInput rendered');

  return (
    <div style={{ padding: '20px', border: '2px solid green', margin: '10px' }}>
      <h3>Name Input Component</h3>
      <input
        value={name}
        onChange={(e) => setName(e.target.value)}
        placeholder='Enter name'
        style={{ padding: '5px' }}
      />
      <p>Name: {name}</p>
      <p style={{ fontSize: '12px', color: '#666' }}>
        ✅ Only re-renders when name changes
      </p>
    </div>
  );
});

// Component dùng CẢ 2 contexts
const Summary = memo(function Summary() {
  const { count } = useCount();
  const { name } = useName();

  console.log('Summary rendered');

  return (
    <div
      style={{ padding: '20px', border: '2px solid purple', margin: '10px' }}
    >
      <h3>Summary Component</h3>
      <p>Count: {count}</p>
      <p>Name: {name || '(empty)'}</p>
      <p style={{ fontSize: '12px', color: '#666' }}>
        ⚠️ Re-renders when EITHER count or name changes
      </p>
```

```jsx
 * Cart với State/Dispatch Splitting
 * Dispatch NEVER changes → Components dùng dispatch KHÔNG re-render
 */

// 1. Separate Contexts
const CartStateContext = createContext();
const CartDispatchContext = createContext();

// 2. Reducer
const cartReducer = (state, action) => {
  switch (action.type) {
    case 'ADD_ITEM': {
      const existing = state.items.find(
        (item) => item.id === action.payload.id,
      );

      if (existing) {
        return {
          ...state,
          items: state.items.map((item) =>
            item.id === action.payload.id
              ? { ...item, quantity: item.quantity + 1 }
              : item,
          ),
        };
      }

      return {
        ...state,
        items: [...state.items, { ...action.payload, quantity: 1 }],
      };
    }

    case 'REMOVE_ITEM':
      return {
        ...state,
        items: state.items.filter((item) => item.id !== action.payload),
      };

    case 'CLEAR':
      return { ...state, items: [] };

    default:
      return state;
  }
};

// 3. Provider
function CartProvider({ children }) {
  const [state, dispatch] = useReducer(cartReducer, { items: [] });

  // dispatch NEVER changes (stable reference)
  // state changes frequently

  return (
    <CartStateContext.Provider value={state}>
      <CartDispatchContext.Provider value={dispatch}>
        {children}
      </CartDispatchContext.Provider>
    </CartStateContext.Provider>
  );
}
```

```jsx
import {
  createContext,
  useContext,
  useRef,
  useSyncExternalStore,
  memo,
} from 'react';

/**
 * @typedef {Object} UserState
 * @property {string} name
 * @property {string} email
 * @property {number} age
 * @property {{ theme: string, language: string, notifications?: boolean }} preferences
 */

/**
 * @typedef {Object} UserStore
 * @property {() => UserState} getSnapshot - Get current state snapshot
 * @property {(cb: () => void) => () => void} subscribe - Subscribe to changes
 * @property {(updater: UserState | ((prev: UserState) => UserState)) => void} updateState
 */

const UserContext = createContext(/** @type {UserStore | null} */ (null));

/**
 * ✅ UserProvider (External Store Pattern)
 *
 * - Context chỉ giữ reference đến store (KHÔNG đổi)
 * - State nằm ngoài React (useRef)
 * - Manual subscription để control re-render
 *
 * @param {{ children: React.ReactNode }} props
 */
function UserProvider({ children }) {
  /** @type {React.MutableRefObject<UserState>} */
  const stateRef = useRef({
    name: 'John Doe',
    email: 'john@example.com',
    age: 30,
    preferences: {
      theme: 'light',
      language: 'en',
      notifications: true,
    },
  });

  /** @type {React.MutableRefObject<Set<() => void>>} */
  const listenersRef = useRef(new Set());

  /** @type {React.MutableRefObject<UserStore | null>} */
  const storeRef = useRef(null);

  // ✅ Create store ONCE (stable reference)
  if (!storeRef.current) {
    storeRef.current = {
      /**
       * Get latest state snapshot
       * @returns {UserState}
       */
      getSnapshot: () => stateRef.current,

      /**
       * Subscribe to store updates
       * @param {() => void} callback
       * @returns {() => void} unsubscribe
       */
      subscribe: (callback) => {
        listenersRef.current.add(callback);
        return () => listenersRef.current.delete(callback);
      },

      /**
       * Update state + notify listeners
       * @param {UserState | ((prev: UserState) => UserState)} updater
       */
      updateState: (updater) => {
        const next =
          typeof updater === 'function'
            ? updater(stateRef.current)
            : updater;

        stateRef.current = next;

        // Notify ALL listeners → selector decides who re-render
        listenersRef.current.forEach((cb) => cb());
      },
    };
  }

  return (
    <UserContext.Provider value={storeRef.current}>
      {children}
    </UserContext.Provider>
  );
}

/**
 * ✅ useUserSelector
 *
 * - Subscribe vào store
 * - Chỉ return selected slice
 * - React sẽ chỉ re-render nếu value thay đổi (Object.is)
 *
 * @template T
 * @param {(state: UserState) => T} selector
 * @returns {T}
 */
function useUserSelector(selector) {
  const store = useContext(UserContext);
  if (!store) throw new Error('useUserSelector must be used within Provider');

  return useSyncExternalStore(
    store.subscribe,
    () => selector(store.getSnapshot()),
    () => selector(store.getSnapshot()),
  );
}

/**
 * Hook để update state
 * @returns {UserStore['updateState']}
 */
function useUserUpdate() {
  const store = useContext(UserContext);
  if (!store) throw new Error('useUserUpdate must be used within Provider');
  return store.updateState;
}

/**
 * HOW IT WORKS:
 *
 * useSyncExternalStore:
 * - Hook chính thức của React 18 để kết nối với external store
 * - subscribe: đăng ký listener → được gọi khi store update
 * - getSnapshot: trả về state hiện tại (source of truth)
 * - React sẽ:
 *    1. Gọi getSnapshot()
 *    2. So sánh với lần trước bằng Object.is
 *    3. Nếu khác → trigger re-render component
 *
 * 👉 Quan trọng:
 * - React KHÔNG biết gì về state bên trong
 * - Nó chỉ dựa vào snapshot + subscription
 *
 * --------------------------------------------------
 *
 * useUserSelector:
 * - Nhận selector: (state) => selectedValue
 * - Mỗi lần store update:
 *    → chạy lại selector trên snapshot mới
 *    → nếu kết quả KHÁC (Object.is) → re-render
 *
 * 👉 Tức là:
 * - Không còn "subscribe toàn bộ state"
 * - Mà subscribe theo "kết quả của selector"
 *
 * --------------------------------------------------
 *
 * BENEFITS:
 * ✅ Fine-grained subscription (re-render theo slice)
 * ✅ Tránh broadcast của Context
 * ✅ Flexible: chọn bất kỳ phần nào của state
 * ✅ Không cần external library (dùng built-in React)
 *
 * --------------------------------------------------
 *
```

```jsx
// CONTEXTS (Split by domain)

const SearchContext = createContext();
const DispatchContext = createContext();

  todos: [],
  filter: 'all',
  searchQuery: '',
};

// ========================================
// REDUCER
// ========================================

function reducer(state, action) {
  switch (action.type) {
    case 'ADD_TODO': {
      const todo = {
        id: Date.now(),
        text: action.payload,
        completed: false,
      };
      return { ...state, todos: [...state.todos, todo] };
    }

    case 'TOGGLE_TODO': {
      const todos = state.todos.map((t) =>
        t.id === action.payload ? { ...t, completed: !t.completed } : t,
      );
      return { ...state, todos };
    }

    case 'DELETE_TODO': {
      return {
        ...state,
        todos: state.todos.filter((t) => t.id !== action.payload),
      };
    }

    case 'SET_FILTER':
      if (state.filter === action.payload) return state;
      return { ...state, filter: action.payload };

    case 'SET_SEARCH':
      if (state.searchQuery === action.payload) return state;
      return { ...state, searchQuery: action.payload };

    case 'BULK_ADD':
      return { ...state, todos: [...state.todos, ...action.payload] };

    default:
      return state;
  }
}

function useFilteredTodos() {

    if (filter === 'active') {
      result = result.filter((t) => !t.completed);
    } else if (filter === 'completed') {
      result = result.filter((t) => t.completed);
    }

    if (search) {
      result = result.filter((t) =>
        t.text.toLowerCase().includes(search.toLowerCase()),
      );
    }

    return result;
  }, [todos, filter, search]);

      completed,

 * PERFORMANCE ANALYSIS (Multi-Context Version):
 *
 * WITHOUT Optimizations (Single Context - Broadcast):
 * - Add todo → ALL components re-render (context value changed)
 * - Toggle todo → ALL components re-render
 * - Change filter → ALL components re-render
 * - Type in search → ALL components re-render
 * - Root cause: 1 context = 1 broadcast channel
 * - With 100+ todos: render storm, UI lag, wasted work
 *
 * WITH Optimizations (Multi-Context + Memoization):
 *
 * 👉 Add todo:
 * - Re-render: TodoList, Stats
 * - Reason: todos slice changed
 *
 * 👉 Toggle todo:
 * - Re-render: ONLY affected TodoItem + Stats
 * - Reason: structural sharing giữ reference các item khác
 *
 * 👉 Change filter:
 * - Re-render: FilterButtons, TodoList
 * - Reason: filter slice changed
 *
 * 👉 Type in search:
 * - Re-render: SearchBar, TodoList
 * - Reason: search slice changed
 *
 * 👉 Dispatch-only components:
 * - TodoInput, BulkAdd → KHÔNG re-render (trừ khi internal state change)
 *
 * 👉 Provider:
 * - TodoProvider luôn re-render (do useReducer)
 * - Nhưng KHÔNG broadcast toàn bộ nhờ split contexts
 *
 * RESULT:
 * - Re-render được isolate theo từng domain (todos / filter / search)
 * - Không còn cascade re-render toàn app
 * - Với 100+ todos: UI vẫn mượt, predictable
 *
 * KEY OPTIMIZATIONS:
 *
 * 1. Multi-Context Splitting (CRITICAL)
 *    - Mỗi slice state = 1 context riêng
 *    - Giảm phạm vi broadcast
 *
 * 2. State / Dispatch Split
 *    - dispatch stable reference
 *    - Component chỉ dispatch → không bị ảnh hưởng bởi state changes
 *
 * 3. useMemo (Derived Data)
 *    - Filter/search chỉ recompute khi dependencies đổi
 *    - Tránh O(n) mỗi render
 *
 * 4. React.memo
 *    - Skip render khi props không đổi
 *    - Cắt propagation từ parent
 *
 * 5. Custom Comparison (TodoItem)
 *    - Chỉ re-render khi chính item đó thay đổi
 *    - Enable fine-grained updates

 * LIMITATIONS:
 *
 * - Vẫn là coarse-grained theo slice (todos/filter/search)
```

### Bug 1: useMemo với Wrong Dependencies ⚠️

```jsx

 * 🐛 BUG: useMemo không work như expected
 *
 * 🎯 CHALLENGE: Tìm lỗi

  const [theme, setTheme] = useState('light');
```

**❓ QUESTIONS:**

2. Dependencies nào cần thêm?

```jsx
  [user, theme],
);

// 🔥 RULE (critical):
// - Context value PHẢI thay đổi khi state thay đổi
```

### Bug 2: React.memo không Work với Objects ⚠️

```jsx
//   prevProps.user === nextProps.user
// - → luôn FALSE (different reference)
// → Component re-render

// 🔥 KEY INSIGHT:
// React.memo KHÔNG compare value
// → chỉ compare reference

const user = { name: 'John', age: 30 };

  return prev.user.name === next.user.name;

// - Object / array / function trong render → ALWAYS new
```

```jsx
import {
  createContext,
  useContext,
  useReducer,
  useMemo,
  useCallback,
  memo,
  useRef,
  useEffect,
} from 'react';

// ========================================
// 1. SPLIT CONTEXTS (State & Dispatch)
// ========================================
const CartStateContext = createContext(null);
const CartDispatchContext = createContext(null);

// ========================================
// 2. REDUCER
// ========================================
const cartReducer = (state, action) => {
  switch (action.type) {
    case 'ADD_ITEM': {
      const exists = state.items.find((i) => i.id === action.payload.id);
      if (exists) {
        return {
          ...state,
          items: state.items.map((i) =>
            i.id === action.payload.id ? { ...i, qty: i.qty + 1 } : i,
          ),
        };
      }
      return {
        ...state,
        items: [...state.items, { ...action.payload, qty: 1 }],
      };
    }
    case 'REMOVE_ITEM':
      return {
        ...state,
        items: state.items.filter((i) => i.id !== action.payload),
      };
    case 'UPDATE_QTY':
      return {
        ...state,
        items: state.items.map((i) =>
          i.id === action.payload.id ? { ...i, qty: action.payload.qty } : i,
        ),
      };
    case 'CLEAR':
      return { ...state, items: [] };
    default:
      return state;
  }
};

// ========================================
// 3. PROVIDER
// ========================================
function CartProvider({ children }) {
  const [state, dispatch] = useReducer(cartReducer, { items: [] });

  // ✅ dispatch từ useReducer là stable reference → không cần useCallback
  return (
    <CartStateContext.Provider value={state}>
      <CartDispatchContext.Provider value={dispatch}>
        {children}
      </CartDispatchContext.Provider>
    </CartStateContext.Provider>
  );
}

  const ctx = useContext(CartStateContext);
  if (!ctx) throw new Error('useCartState must be used within CartProvider');
  return ctx;
}

function useCartDispatch() {
  const ctx = useContext(CartDispatchContext);
  if (!ctx) throw new Error('useCartDispatch must be used within CartProvider');
  return ctx;
}

// ✅ Selector hook: memoized derived state
function useCartTotals() {
  const { items } = useCartState();
  return useMemo(() => {
    const totalQty = items.reduce((sum, i) => sum + i.qty, 0);
    const totalPrice = items.reduce((sum, i) => sum + i.price * i.qty, 0);
    return { totalQty, totalPrice };
  }, [items]);
}

  });
  return countRef.current;
}
```

```jsx
function useVisibleMessages() {
  const { messages } = useMessageState();
  const [scrollTop, setScrollTop] = useState(0);
  const ITEM_HEIGHT = 60; // px
  const VISIBLE_COUNT = 50;
  const CONTAINER_HEIGHT = VISIBLE_COUNT * ITEM_HEIGHT;

  const { virtualItems, totalHeight, startIndex } = useMemo(() => {
    const totalH = messages.length * ITEM_HEIGHT;
    const startIdx = Math.max(0, Math.floor(scrollTop / ITEM_HEIGHT) - 5); // Buffer 5 items
    const endIdx = Math.min(messages.length, startIdx + VISIBLE_COUNT + 10); // Buffer 5 items
    const items = messages.slice(startIdx, endIdx).map((msg, idx) => ({
      ...msg,
      index: startIdx + idx,
      style: {
        position: 'absolute',
        top: (startIdx + idx) * ITEM_HEIGHT,
        height: ITEM_HEIGHT,
        left: 0,
        right: 0,
      },
    }));
    return { virtualItems: items, totalHeight: totalH, startIndex: startIdx };
  }, [messages, scrollTop]);

  return {
    virtualItems,
    totalHeight,
    startIndex,
    CONTAINER_HEIGHT,
    setScrollTop,
  };
}

          text: texts[Math.floor(Math.random() * texts.length)],
          sender: senders[Math.floor(Math.random() * senders.length)],
          timestamp: Date.now(),
        },
      });
    }, 2000); // Mỗi 2s có tin nhắn mới

    // Simulate user online/offline
    const userInterval = setInterval(() => {
      const id = Math.floor(Math.random() * 3) + 1;
      const isOnline = Math.random() > 0.5;
      userDispatch({
        type: isOnline ? 'USER_ONLINE' : 'USER_OFFLINE',
        payload: id,
      });
    }, 5000);

    return () => {
      clearInterval(interval);
      clearInterval(userInterval);
    };
  }, [msgDispatch, userDispatch, typingDispatch]);

  return null;
}

function AppProviders({ children }) {
  return (
    <MessageProvider>
      <UserProvider>
        <TypingProvider>{children}</TypingProvider>
      </UserProvider>
    </MessageProvider>
  );
}

function App() {
  return (
    <AppProviders>
      <div
        style={{
          maxWidth: '900px',
          margin: '20px auto',
          fontFamily: 'sans-serif',

          gridTemplateColumns: '1fr 200px',
          gap: '20px',
        }}
      >
        <Simulation />

        <div>
          <h2>💬 Real-time Chat</h2>
          <StatsBar />
          <MessageList />
          <TypingIndicator />
          <ChatInput />
        </div>

        <div>
          <UserList />
        </div>
      </div>
    </AppProviders>

}

export default App;
```

### Optimization Cost (Reality)

```jsx
// Simple Code:

// “Optimized” Code:

// ✅ RULE (correct):
// Optimize khi:

```

### Câu Hỏi Phỏng Vấn

**Junior:**

```
Q: Tại sao Context gây re-render issues?
A: 2 root causes:
   1. Value object recreation: {} mới mỗi render → React nghĩ context đổi
   2. Single context: Mọi state chung 1 context → change 1 state = re-render all
   → Tất cả consumers re-render dù không dùng state vừa đổi.
```

```
Q: useMemo giải quyết vấn đề gì trong Context?
A: Cache context value object. Chỉ tạo mới khi dependencies đổi.
   → Same reference → React không detect change → Không re-render consumers.
   Không giải quyết được: Consumer vẫn re-render nếu dùng field khác trong cùng value.
```

```
Q: React.memo làm gì? Giới hạn là gì?
A: Memoize component render. Chỉ re-render khi props thay đổi (shallow compare).
   Giới hạn: Không ngăn re-render khi context value đổi (context là "hidden prop").
   Không work với objects/arrays mới mỗi render làm props.
```

**Mid:**

```
Q: So sánh 3 strategies: useMemo vs Context Splitting vs Selector Pattern
A: | Strategy        | Complexity | Performance | Use Case                  |
   |-----------------|------------|-------------|---------------------------|
   | useMemo         | Low        | Medium      | Simple context, few consumers |
   | Context Split   | Medium     | High        | Independent state groups  |
   | Selector        | High       | Excellent   | Complex state, many consumers |

   useMemo: Dễ implement, chỉ fix object recreation.
   Splitting: Tách concerns, isolated re-renders, nhiều boilerplate.
   Selector: Fine-grained subscription, best performance, phức tạp nhất.
```

```
Q: State/Dispatch splitting hoạt động như thế nào?
A: Tách thành 2 contexts:
   - StateContext: Chứa state (thay đổi thường xuyên)
   - DispatchContext: Chứa dispatch function (stable, không bao giờ đổi)

   Components "write-only" (buttons) dùng useCartDispatch() → KHÔNG re-render.
   Components "read-only" (display) dùng useCartState() → Re-render khi cần.

   Dispatch từ useReducer là stable reference by design.
```

```
Q: Khi nào NÊN optimize Context?
A: CHỈ optimize khi có vấn đề THẬT:
   1. Profiler shows renders > 16ms (dưới 60fps)
   2. UI cảm thấy laggy
   3. User complaints

   KHÔNG optimize: "just in case", "best practice", app nhỏ không có issue.
   Premature optimization = evil: Code phức tạp hơn, khó maintain, không có benefit.
```

**Senior:**

```
Q: Design optimization strategy cho large app (100+ contexts)
A: 3-layer architecture:

   Layer 1 - Domain Splitting:
   - Mỗi domain 1 context (User, Cart, Analytics, Notification,...)
   - Không bao giờ có "God Context"

   Layer 2 - State/Dispatch trong mỗi domain:
   - DomainStateContext + DomainDispatchContext
   - Read-only components stable, write-only components stable

   Layer 3 - Selectors cho complex domains:
   - useDomainSelector(selector) cho fine-grained subscription
   - Derived state memoized với reselect-like pattern

   Infrastructure:
   - ESLint rule exhaustive-deps bắt buộc
   - React DevTools Profiler trong CI (performance regression tests)
   - Render counter components trong dev mode
   - Code review checklist: "Did you measure before optimizing?"
```

```
Q: Implement custom selector pattern không dùng useSyncExternalStore
A: Dùng useRef + useEffect để manual subscription:

   function useSelector(selector) {
     const { state, subscribe } = useContext(StoreContext);
     const [, forceRender] = useReducer(c => c + 1, 0);
     const selectedRef = useRef(selector(state));

     useEffect(() => {
       return subscribe(() => {
         const nextSelected = selector(state);
         if (!shallowEqual(selectedRef.current, nextSelected)) {
           selectedRef.current = nextSelected;
           forceRender();
         }
       });
     }, [subscribe, selector, state]);

     return selectedRef.current;
   }

   shallowEqual: Object.keys + === comparison cho từng key.
   Khó hơn useSyncExternalStore vì phải handle tearing, SSR, concurrent features.
```

```
Q: Debug re-render issues methodology
A: 5-step systematic approach:

   1. MEASURE: React DevTools Profiler → Record → Flamegraph
      - Identify WHICH components re-render
      - Measure render duration (ms)

   2. IDENTIFY TRIGGER: "Highlight updates" trong DevTools
      - Xem action nào gây re-render
      - Check console logs (console.log trong components)

   3. ROOT CAUSE ANALYSIS:
      - Context value mới mỗi render? → useMemo
      - Functions mới mỗi render? → useCallback
      - Single context quá lớn? → Split
      - Component dùng field không cần? → Selector

   4. APPLY MINIMAL FIX:
      - Đơn giản nhất trước (useMemo)
      - Chỉ split khi useMemo không đủ
      - Chỉ selector khi split không đủ

   5. VERIFY:
      - Profile lại
      - Compare render count & duration
      - Document improvement (before/after metrics)
```

```
Q: Measure performance improvements quantitatively
A: 4 metrics + 3 tools + reporting framework:

METRICS:
├─ Render Count: Số lần re-render per interaction (target: <5)
├─ Render Time: Thời gian render 1 component (target: <16ms = 60fps)
├─ Interaction Latency: Từ click đến UI update (target: <100ms)
└─ Memory Usage: Heap size growth (target: không leak)

TOOLS:
├─ React DevTools Profiler: Record flamegraph, export data
├─ Performance.mark/measure(): User Timing API cho custom metrics
└─ web-vitals library: CLS, FID, LCP cho real-user monitoring

METHODOLOGY (Before/After):
1. Baseline: Profile 10 interactions, lấy median render count + time
2. Optimize: Apply 1 strategy
3. Remeasure: Same 10 interactions, same device, same data
4. Calculate: % improvement = (before - after) / before × 100

EXAMPLE REPORT:
┌─────────────────┬─────────┬─────────┬────────────┐
│ Metric          │ Before  │ After   │ Improvement│
├─────────────────┼─────────┼─────────┼────────────┤
│ Re-renders/add  │ 47      │ 3       │ -93.6%     │
│ Render time     │ 45ms    │ 8ms     │ -82.2%     │
│ FPS (typing)    │ 22fps   │ 58fps   │ +163.6%    │
│ Interaction     │ 220ms   │ 45ms    │ -79.5%     │
└─────────────────┴─────────┴─────────┴────────────┘

CODE SNIPPET (measurement helper):
const measureRender = (name) => {
  performance.mark(`${name}-start`);
  useEffect(() => {
    performance.mark(`${name}-end`);
    performance.measure(name, `${name}-start`, `${name}-end`);
    const entries = performance.getEntriesByName(name);
    console.log(`${name}: ${entries[0].duration.toFixed(2)}ms`);
  });
};

RULE: Không claim "faster" nếu không có số. Baseline là bắt buộc.
```
