## 🤔 Kiểm tra đầu vào (5 phút)

1.  **Suspense boundaries hoạt động như thế nào với async data?**

- _Đáp án: Suspense boundary hiển thị fallback UI khi component con đang suspend (đang fetch data), sau đó tự động thay thế bằng nội dung thực khi data sẵn sàng — hỗ trợ streaming từng phần thay vì chờ toàn bộ._

2. **Error Boundaries catch errors ở đâu trong component tree?**
   - _Đáp án: Catch errors trong lifecycle methods và render phase của component con bên dưới nó trong tree — không catch errors trong event handlers, async code (useEffect, Promise), hoặc SSR._

3. **SSR (Server-Side Rendering) khác CSR (Client-Side Rendering) như thế nào?**
   - _Đáp án: SSR render HTML trên server rồi gửi về browser (SEO tốt, FCP nhanh), sau đó hydrate thành React app; CSR gửi HTML trống về browser rồi React render toàn bộ ở client (SEO kém, phải chờ download JS)._

### 1.1 Vấn Đề Thực Tế

**Traditional React = Mọi thứ chạy trên client**

```jsx
// ❌ Traditional React - ALL code runs on client browser
function ProductPage({ productId }) {
  const [product, setProduct] = useState(null);
  const [reviews, setReviews] = useState([]);
  const [recommendations, setRecommendations] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Client fetches everything
    Promise.all([
      fetch(`/api/products/${productId}`).then((r) => r.json()),
      fetch(`/api/reviews/${productId}`).then((r) => r.json()),
      fetch(`/api/recommendations/${productId}`).then((r) => r.json()),
    ]).then(([productData, reviewsData, recsData]) => {
      setProduct(productData);
      setReviews(reviewsData);
      setRecommendations(recsData);
      setLoading(false);
    });
  }, [productId]);

  if (loading) return <LoadingSpinner />;

  return (
    <div>
      <ProductInfo product={product} />
      <ReviewsList reviews={reviews} />
      <Recommendations items={recommendations} />
    </div>
  );
}

// Vấn đề:
// 1. 📦 Large JavaScript bundle - Cả app code gửi đến browser
// 2. ⏱️ Waterfall requests - Component mount → fetch → render
// 3. 🔄 Loading states everywhere - Poor UX
// 4. 🔍 SEO challenges - Content không có trong initial HTML
// 5. 🔐 Security risks - API keys có thể leak ra client
// 6. 💾 Duplicate code - Logic validation ở cả server & client
// 7. 🌐 Network overhead - Multiple round trips
```

**Real-world Impact:**

```
Typical E-commerce Page Load:
┌────────────────────────────────────────┐
│ 1. Server sends empty HTML (1KB)      │ → 50ms
│ 2. Client downloads React bundle      │ → 500ms
│ 3. Client downloads app code          │ → 800ms
│ 4. React hydrates                     │ → 200ms
│ 5. useEffect runs                     │ → 10ms
│ 6. Fetch product data                 │ → 300ms
│ 7. Fetch reviews                      │ → 250ms
│ 8. Fetch recommendations              │ → 200ms
│ 9. Final render                       │ → 50ms
└────────────────────────────────────────┘
Total: ~2.3 seconds before user sees content!

Bundle sizes:
- React: 40KB (gzipped)
- App code: 200KB (gzipped)
- Heavy libraries (charts, markdown): 100KB
→ Total: 340KB just for JavaScript
```

### 1.2 Giải Pháp: React Server Components

**RSC Payload là gì?**

**RSC Payload = Format text đặc biệt React dùng để stream UI tree từ server xuống client**, không phải HTML thuần cũng không phải JSON thường.

---

**Payload chứa gì?**

| Thành phần                        | Mô tả                                                  |
| --------------------------------- | ------------------------------------------------------ |
| **Rendered Server Components**    | UI structure cuối cùng của components chạy trên server |
| **Client Component Placeholders** | "Lỗ hổng" nơi cần chèn interactive components          |
| **JavaScript References**         | Đường dẫn đến JS chunks cần để hydrate                 |
| **Serialized Props**              | Data truyền từ Server Component → Client Component     |

---

**Cấu trúc thực tế**

```
1:I["./src/components/Counter.tsx",["client-chunk-id"],"Counter"]
2:{"name":"World","items":["Apple","Orange"]}
3:["$","div",null,{"children":[["$","h1",null,{"children":"Hello "}],["$","$L1",null,{"props":{"count":5}}]]}]
```

**Giải thích từng dòng:**

| Prefix     | Ý nghĩa                                                                                           |
| ---------- | ------------------------------------------------------------------------------------------------- |
| `1:I[...]` | **I** = Module Reference. Browser biết đâu để tải JS cho Client Component                         |
| `2:{...}`  | Props serializable truyền từ server xuống                                                         |
| `3:[...]`  | **$** = Element Definition (HTML tag hoặc React element). **$L1** = reference đến module ở dòng 1 |

---

**RSC = Components render trên server, gửi kết quả đến client**

```jsx
// ✅ Server Component - Runs on SERVER only
// ⚠️ CONCEPTUAL CODE - Chỉ chạy trong Next.js App Router
async function ProductPage({ productId }) {
  // NO useState! NO useEffect! NO loading states!
  // Direct data access on server
  const product = await db.products.findById(productId);
  const reviews = await db.reviews.findMany({ productId });
  const recommendations = await getRecommendations(productId);

  return (
    <div>
      <ProductInfo product={product} />
      <ReviewsList reviews={reviews} />
      <Recommendations items={recommendations} />
    </div>
  );
}

// Benefits:
// ✅ Zero JavaScript for this component
// ✅ Direct database/API access - No intermediate API layer
// ✅ Parallel data fetching - All await at once
// ✅ No loading states - Data ready when component renders
// ✅ Perfect SEO - HTML contains full content
// ✅ Smaller bundle - No data fetching code sent to client
// ✅ Better security - Secrets stay on server
```

**Flow Comparison:**

```
❌ REACT TRUYỀN THỐNG (Client-Side):
1. Browser gửi request trang
2. Server trả về HTML trống + JS bundles
3. Browser tải & parse JavaScript
4. React render (hiển thị loading)
5. useEffect kích hoạt gọi API
6. Chờ phản hồi từ server
7. Re-render với data
→ Thời gian tương tác được: 2-3 giây

✅ REACT SERVER COMPONENTS:
1. Browser gửi request trang
2. Server chạy React components
3. Server fetch toàn bộ data song song
4. Server render components thành RSC payload
5. Server stream HTML xuống browser
6. Browser hiển thị content ngay lập tức
7. Hydrate chỉ phần tương tác
→ Thời gian tương tác được: 0.5-1 giây
```

### 1.3 Mental Model

**RSC Architecture:**

Trong RSC (React Server Components) architecture, **"Hydrate only interactive parts"** là một trong những lợi ích cốt lõi so với SSR (Server-Side Rendering) truyền thống. Dưới đây là quá trình chi tiết:

**Hydrate = "Gắn hồn" JavaScript vào HTML tĩnh**

Browser nhận HTML từ server → hiển thị được ngay, nhưng chưa tương tác được (click không chạy, state không đổi). React cần **đọc lại** HTML đó, **gắn event listeners**, **khởi tạo hooks** — quá trình này gọi là hydration.

Ví dụ: Server gửi `<button>Add</button>` → hiện ra màn hình. Nhưng click chưa chạy. Sau hydration, React gắn `onClick` + `useState` vào → button sống.

RSC chỉ hydrate phần cần tương tác, còn lại để yên HTML tĩnh.

**Quy trình Hydrate trong RSC**

```
┌───────────────────────────────────────────────────────────────────────────┐
│  BƯỚC 1: Server render                                                    │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │  Server Components (default)                                        │  │
│  │  • Fetch data từ DB/API trực tiếp                                   │  │
│  │  • Render thành virtual nodes → RSC Payload (JSON-like)             │  │
│  │    Ví dụ: { type: 'h1', props: { children: 'Bài viết' } }           │  │
│  │  • Zero JS component code gửi đến client                            │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                    ↓                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │  Client Components (có 'use client')                                │  │
│  │  • Server thay bằng placeholder trong RSC Payload                   │  │
│  │    Ví dụ: { type: '$L1', props: {} }  ← $L1 = reference             │  │
│  │  • Gửi kèm JS bundle chứa component code để hydrate                 │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                    ↓                                      │
│  STREAM: RSC Payload + JS bundle xuống browser                            │
└───────────────────────────────────────────────────────────────────────────┘
                           ↓
┌───────────────────────────────────────────────────────────────────────────┐
│  BƯỚC 2: Browser nhận và xử lý RSC Payload                                │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │  Server Component nodes → React render thành HTML ngay              │  │
│  │  • <h1>Bài viết</h1>, <p>Nội dung...</p>                            │  │
│  │  • Hiển thị ngay lập tức, SEO-friendly                              │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                    +                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │  Client Component placeholders ($L1) → đợi JS bundle tải xong       │  │
│  │  • Hiển thị fallback hoặc skeleton nếu có <Suspense>                │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────────────────┘
                           ↓ JS bundle tải xong
┌───────────────────────────────────────────────────────────────────────────┐
│  BƯỚC 3: Hydrate CHỈ interactive parts                                    │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │  ❌ KHÔNG hydrate:                                                  │  │
│  │     • <h1>, <p>, <article> — HTML thuần, React không quản lý        │  │
│  │     • Không có VDOM, không có event listeners                       │  │
│  │     • Browser render như trang HTML thường                          │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                    ↓                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │  ✅ CHỈ hydrate placeholders ($L1 → LikeButton):                    │  │
│  │     • Parse JS bundle, tạo React component instance                 │  │
│  │     • Khởi tạo hooks (useState, useEffect...)                       │  │
│  │     • Attach event listeners (onClick, onChange...)                 │  │
│  │     • Thay placeholder bằng interactive component thực              │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────────────────┘
```

**So sánh Hydration: SSR vs RSC**

- SSR truyền thống (getServerSideProps / getStaticProps)
- RSC (App Router)

|                         | **SSR (Pages Router)**            | **RSC (App Router)**         |
| ----------------------- | --------------------------------- | ---------------------------- |
| **What gets hydrated**  | Toàn bộ React tree                | Chỉ Client Components        |
| **Hydration cost**      | Cao (parse + attach tất cả nodes) | Thấp (chỉ interactive nodes) |
| **Time to Interactive** | Chậm (phải hydrate hết)           | Nhanh (selective hydration)  |
| **Bundle JS**           | Full app code                     | Chỉ Client Components code   |
| **Tỷ lệ**               | 100% tree                         | ~10-20% tree (tùy app)       |

Để hình dung, mình dùng ví dụ đơn giản: **một trang blog với nút Like**.

---

#### SSR (Next.js Pages Router)

```jsx
// pages/blog.js — CẢ TRANG là React component
function BlogPage() {
  return (
    <article>
      <h1>Bài viết</h1> {/* ← React quản lý */}
      <p>Nội dung dài...</p> {/* ← React quản lý */}
      <LikeButton /> {/* ← React quản lý */}
    </article>
  );
}

function LikeButton() {
  const [liked, setLiked] = useState(false);
  return <button onClick={() => setLiked(!liked)}>❤️</button>;
}
```

**Server gửi xuống browser:**

```
1. HTML: <article><h1>...</h1><p>...</p><button>❤️</button></article>
2. JS bundle: toàn bộ code BlogPage + LikeButton
```

**Hydrate:** React đọc lại HTML, tạo VDOM cho `<h1>`, `<p>`, `<button>` — **dù h1 và p chẳng cần tương tác gì**. Vì React có cả `BlogPage` code, nó phải reconstruct toàn bộ tree.

---

#### RSC (Next.js App Router)

```jsx
// app/blog/page.js — Server Component
async function BlogPage() {
  const post = await db.posts.get();

  return (
    <article>
      <h1>{post.title}</h1> {/* ← RSC Payload: {type:'h1',...} */}
      <p>{post.content}</p> {/* ← RSC Payload: {type:'p',...} */}
      <LikeButton /> {/* ← RSC Payload: {type:'$L1',...} */}
    </article>
  );
}

// app/components/LikeButton.js — Client Component
('use client');
function LikeButton() {
  const [liked, setLiked] = useState(false);
  return <button onClick={() => setLiked(!liked)}>❤️</button>;
}
```

**Server gửi xuống browser:**

```
1. RSC Payload (stream):
   1:I["./components/LikeButton.js",["chunk-abc123"],"LikeButton"]
   2:{"liked":false}
   3:["$","article",null,{"children":[
        ["$","h1",null,{"children":"Bài viết"}],
        ["$","p",null,{"children":"Nội dung..."}],
        ["$","$L1",null,{"props":{"liked":false}}]
      ]}]

2. JS bundle: CHỈ code LikeButton (KHÔNG có BlogPage)
```

**Hydrate:** React render `h1`, `p` thành HTML rồi **quên luôn** — không tạo VDOM, không quản lý. Chỉ `$L1` được hydrate thành `LikeButton` với hooks và event listeners.

---

#### So sánh trực quan

```
SSR — React "ôm" cả nhà:

    ┌─────────────────┐
    │   <article>     │  ← React VDOM (có BlogPage code)
    │   <h1>          │  ← React VDOM
    │   <p>           │  ← React VDOM
    │   <LikeButton>  │  ← React VDOM + hydrate hooks
    └─────────────────┘
         ↑
    TẤT CẢ đều trong Virtual DOM
    Vì React có code của cả nhà

RSC — React chỉ "ôm" cái cần thiết:

    ┌─────────────────┐
    │   <article>     │  ← HTML thuần (React không có code)
    │   <h1>          │  ← HTML thuần (React không có code)
    │   <p>           │  ← HTML thuần (React không có code)
    │   <LikeButton>  │  ← React VDOM + hydrate hooks
    └─────────────────┘
         ↑
    React chỉ có LikeButton code
    Còn lại là HTML lạ hoắc, browser tự render
```

**Kết quả thực tế**

```
Traditional SSR hydration:
├─ Parse 1000 DOM nodes
├─ Attach 50 event listeners
├─ Khởi tạo 20 component instances
└─ Time: ~500ms

RSC selective hydration:
├─ Parse 1000 DOM nodes (skip, already HTML)
├─ Attach 3 event listeners (chỉ buttons)
├─ Khởi tạo 1 component instance
└─ Time: ~50ms (10x faster!)

```

**Tóm lại:** RSC gửi HTML sẵn cho static content → browser hiển thị ngay → React chỉ "đánh thức" (hydrate) những phần cần tương tác, giảm đáng kể thời gian và JavaScript cần thiết để page trở nên interactive.

---

#### Tại sao RSC Payload phức tạp vậy?

| Prefix     | Ý nghĩa                                                                        |
| ---------- | ------------------------------------------------------------------------------ |
| `1:I[...]` | **I** = Module Reference. Browser biết đâu để tải JS cho Client Component      |
| `2:{...}`  | Props serializable truyền từ server xuống                                      |
| `3:[...]`  | **$** = Element Definition (HTML tag). **$L1** = reference đến module ở dòng 1 |

Payload này cho phép:

- **Streaming**: Gửi từng chunk, browser render top trước khi bottom xong
- **Giữ state**: Không refresh HTML nguyên trang → modal mở, text đang gõ giữ nguyên
- **Giảm bundle**: Thư viện nặng (markdown parser) chỉ chạy server, không xuống client

**So sánh Fetching Data : SSR vs RSC**

**SSR truyền thống (getServerSideProps / getStaticProps)**

```jsx
// ❌ Sequential — chậm
export async function getServerSideProps() {
  K;
  const users = await fetch('/api/users'); // 300ms
  const posts = await fetch('/api/posts'); // 200ms
  const comments = await fetch('/api/comments'); // 250ms
  // Tổng: 750ms
  return { props: { users, posts, comments } };
}

// ✅ Song song — nhanh hơn
export async function getServerSideProps() {
  const [users, posts, comments] = await Promise.all([
    fetch('/api/users'),
    fetch('/api/posts'),
    fetch('/api/comments'),
  ]);
  // Tổng: ~300ms (bằng request chậm nhất)
  return { props: { users, posts, comments } };
}
```

**RSC (App Router)**

```jsx
// ✅ Song song tự nhiên với await
// Không streaming
async function Dashboard() {
  const [user, analytics, notifications] = await Promise.all([
    fetchUser(), // 100ms
    fetchAnalytics(), // 2000ms ← bottleneck
    fetchNotifications(), // 150ms
  ]);
  // Browser nhìn trắng 2 giây
  return (
    <div>
      <UserWidget user={user} />
      <AnalyticsWidget data={analytics} />
      <NotificationsWidget data={notifications} />
    </div>
  );
}

// Hoặc stream không cần Promise.all — render từng phần
async function Dashboard() {
  // Không await — trả về Promise ngay
  const userPromise = fetchUser();
  const analyticsPromise = fetchAnalytics();
  const notificationsPromise = fetchNotifications();

  // Browser thấy skeleton ngay → dần dần thay bằng content thật
  return (
    <div>
      <Suspense fallback={<UserSkeleton />}>
        <UserWidget promise={userPromise} />{' '}
        {/* Truyền promise xuống Server Components (async component)*/}
      </Suspense>

      <Suspense fallback={<AnalyticsSkeleton />}>
        <AnalyticsWidget promise={analyticsPromise} />
      </Suspense>

      <Suspense fallback={<NotificationsSkeleton />}>
        <NotificationsWidget promise={notificationsPromise} />
      </Suspense>
    </div>
  );
}

// app/components/UserWidget.js — Server Component
async function UserWidget({ promise }) {
  const user = await promise; // ← await trong component con

  return (
    <div>
      <h2>{user.name}</h2>
      <p>{user.email}</p>
    </div>
  );
}
// Pattern này chỉ hoạt động với Server Components (async function). Client Component không thể await trong render.
// Nếu cần dùng trong Client Component, phải dùng use hook (React 18+):
('use client');
import { use, Suspense } from 'react';

function UserWidget({ promise }) {
  const user = use(promise); // ← unwrap promise trong Client Component

  return <div>{user.name}</div>;
}

///------------------------------------

// RSC Payload khi streaming
// Chunk 1 (0ms):
// 1:["$","div",null,{"children":[
//   ["$","$L1",null,{"fallback":["$","UserSkeleton"]}],
//   ["$","$L2",null,{"fallback":["$","AnalyticsSkeleton"]}],
//   ["$","$L3",null,{"fallback":["$","NotificationsSkeleton"]}]
// ]}]

// Chunk 2 (100ms):
// 4:["$","UserWidget",null,{"name":"John","email":"john@..."}]

// Chunk 3 (150ms):
// 5:["$","NotificationsWidget",null,{"count":3,"items":[...]}]

// Chunk 4 (2000ms):
// 6:["$","AnalyticsWidget",null,{"views":1000,"clicks":50}]

///------------------------------------

// Server render Dashboard:
// ├─ UserWidget: promise chưa resolve
// │   → RSC Payload: { type: 'Suspense', fallback: UserSkeleton }
// │   → Stream xuống browser NGAY
// │
// ├─ 100ms sau: userPromise resolve
// │   → RSC Payload: { type: 'UserWidget', props: {...} }
// │   → Stream thêm chunk xuống browser
// │   → React thay UserSkeleton bằng UserWidget
// │
// ├─ AnalyticsWidget: promise chưa resolve
// │   → RSC Payload: { type: 'Suspense', fallback: AnalyticsSkeleton }
// │   → Stream xuống
// │
// ├─ 2000ms sau: analyticsPromise resolve
// │   → RSC Payload: { type: 'AnalyticsWidget', props: {...} }
// │   → Stream thêm chunk
// │   → React thay AnalyticsSkeleton
// │
// └─ NotificationsWidget tương tự (150ms)
```

**Lợi ích**

|                            | Không streaming                      | Có streaming                    |
| -------------------------- | ------------------------------------ | ------------------------------- |
| **Tính chất**              | Parent truyền data, cho children xài | Children nhận Promise, tự await |
| **First Contentful Paint** | Chậm (chờ tất cả)                    | Nhanh (skeleton hiện ngay)      |
| **Perceived performance**  | Kém (nhìn trắng lâu)                 | Tốt (luôn có gì để xem)         |
| **TTFB**                   | Chậm                                 | Nhanh                           |
| **UX**                     | Giật cục                             | Mượt, từng phần load dần        |

**So sánh**

|                   | SSR (Pages Router)               | RSC (App Router)                      |
| ----------------- | -------------------------------- | ------------------------------------- |
| **Song song**     | Phải dùng `Promise.all` thủ công | `Promise.all` hoặc Suspense streaming |
| **Data fetching** | Chỉ trong `getServerSideProps`   | Trực tiếp trong component             |
| **Streaming**     | Không — phải chờ toàn bộ data    | Có — render từng phần khi sẵn sàng    |
| **Place**         | Chỉ page level                   | Bất kỳ component nào                  |

**Component Tree Example:**

```jsx
// Server Component (default)
async function BlogPage() {
  const posts = await db.posts.getAll(); // Server-only

  return (
    <div>
      <Header /> {/* Server Component */}
      <Sidebar posts={posts} /> {/* Server Component */}
      {posts.map((post) => (
        // Client Component - needs interactivity
        <LikeButton
          key={post.id}
          postId={post.id}
        />
```

### 1.4 Hiểu Lầm Phổ Biến

**❌ Hiểu lầm 1: "RSC = Server-Side Rendering (SSR)"**

```jsx
// SSR (Cách cũ - Next.js Pages Router):
export async function getServerSideProps() {
  const data = await fetch('...');
  return { props: { data } };
}

function Page({ data }) {
  // Component chạy 2 lần: lần 1 server (render HTML), lần 2 client (hydrate)
  // Gửi toàn bộ React bundle xuống browser

  const [count, setCount] = useState(0);        // ✅ Chạy trên client sau hydrate
  useEffect(() => { console.log('mounted') }, []);  // ✅ Chạy trên client sau hydrate

  return (
    <div>
      <h1>{data.title}</h1>
      <button onClick={() => setCount(c => c + 1)}>
        Count: {count}  {/* ✅ State hoạt động vì chạy trên client */}
      </button>
    </div>
  );
}

// RSC (Cách mới - Next.js App Router):
async function Page() {
  const data = await fetch('...');

  // Component CHỈ chạy 1 lần trên server
  // KHÔNG gửi JavaScript xuống client cho component này

  // ❌ Không thể dùng useState
  // ❌ Không thể dùng useEffect
  // ❌ Không thể dùng onClick

  return (
    <div>
      <h1>{data.title}</h1>
      {/* Muốn tương tác → phải dùng Client Component */}
      <LikeButton />  {/* ← Client Component riêng */}
    </div>
  );
}

// Client Component - CHỈ phần này gửi JS xuống
'use client';
function LikeButton() {
  const [liked, setLiked] = useState(false);    // ✅ Được vì chạy trên client
  useEffect(() => { ... }, []);                 // ✅ Được vì chạy trên client

  return (
    <button onClick={() => setLiked(!liked)}>
      {liked ? '❤️' : '🤍'}
    </button>
  );
}

// Khác biệt:
// SSR: Component code gửi xuống client → hydrate → dùng hooks được
// RSC: Component code KHÔNG gửi xuống → chỉ chạy server → không dùng hooks
```

**So sánh:**

| Tính năng            | SSR                                     | RSC                          |
| -------------------- | --------------------------------------- | ---------------------------- |
| Code chạy ở đâu      | Server + Client                         | Chỉ Server                   |
| JavaScript gửi xuống | Toàn bộ component code                  | Zero (cho server components) |
| Dùng useState        | ✅ Có                                   | ❌ Không                     |
| Dùng useEffect       | ✅ Có                                   | ❌ Không                     |
| Truy cập database    | ❌ Không (chỉ trong getServerSideProps) | ✅ Có (trực tiếp)            |
| Ảnh hưởng bundle     | Lớn (toàn bộ code)                      | Nhỏ (chỉ client components)  |

|                         | Pages Router (SSR)   | App Router (RSC)                  |
| ----------------------- | -------------------- | --------------------------------- |

### Demo 2: Data Fetching Patterns ⭐⭐

```jsx
/**
 * Demo: Data Fetching - Traditional vs RSC
 *
 * So sánh cách fetch data
 */

// ============= TRADITIONAL REACT (Client-Side) =============

function TraditionalProductPage({ productId }) {
  const [product, setProduct] = useState(null);
  const [reviews, setReviews] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    // Waterfall: One after another
    async function fetchData() {
      try {
        setLoading(true);

        // Request 1
        const productRes = await fetch(`/api/products/${productId}`);
        const productData = await productRes.json();
        setProduct(productData);

        // Request 2 (depends on product)
        const reviewsRes = await fetch(`/api/reviews/${productId}`);
        const reviewsData = await reviewsRes.json();
        setReviews(reviewsData);
      } catch (err) {
        setError(err);
      } finally {
        setLoading(false);
      }
    }

    fetchData();
  }, [productId]);

  if (loading) return <Spinner />;
  if (error) return <Error message={error.message} />;

  return (
    <div>
      <h1>{product.name}</h1>
      <p>${product.price}</p>
      <ReviewsList reviews={reviews} />
    </div>
  );
}

// Problems:
// 1. Waterfall: product → reviews (sequential)
// 2. Loading states management
// 3. Error handling boilerplate
// 4. JavaScript bundle includes all fetch logic
// 5. API layer needed (/api/products, /api/reviews)

// ============= RSC APPROACH =============

// ⚠️ CONCEPTUAL - Only works in Next.js App Router
async function RSCProductPage({ productId }) {
  // Parallel fetching - both at once!
  const [product, reviews] = await Promise.all([
    db.products.findById(productId),
    db.reviews.findMany({ where: { productId } }),
  ]);

  // Direct database access - no API layer needed
  // No loading states - data ready before render
  // No useState/useEffect - simpler code

  return (
    <div>
      <h1>{product.name}</h1>
      <p>${product.price}</p>
      <ReviewsList reviews={reviews} />
    </div>
  );
}

// Benefits:
// ✅ Parallel fetching (faster)
// ✅ No loading states (simpler)
// ✅ Direct DB access (no API layer)
// ✅ Zero fetch code sent to client
// ✅ Better error handling (Error Boundaries)

// ============= STREAMING PATTERN =============

// Traditional: Wait for ALL data before showing anything
async function TraditionalDashboard() {
  const [user, analytics, notifications] = await Promise.all([
    fetchUser(),
    fetchAnalytics(), // Takes 2 seconds
    fetchNotifications(),
  ]);

  // User waits 2 seconds before seeing ANYTHING

      <AnalyticsWidget data={analytics} />

    </div>

}

// RSC: Stream components as data arrives

  const userPromise = fetchUser();
  const analyticsPromise = fetchAnalytics();
  const notificationsPromise = fetchNotifications();

  return (
    <div>
      {/* Show immediately */}
      <Suspense fallback={<UserSkeleton />}>
        <UserWidget promise={userPromise} />
      </Suspense>

      {/* Show when ready (2s) */}
      <Suspense fallback={<AnalyticsSkeleton />}>
        <AnalyticsWidget promise={analyticsPromise} />
      </Suspense>

      {/* Show when ready */}
      <Suspense fallback={<NotificationsSkeleton />}>
        <NotificationsWidget promise={notificationsPromise} />
      </Suspense>
    </div>
  );
}

// Flow:
// 0ms: User sees UserSkeleton + AnalyticsSkeleton + NotificationsSkeleton
// 100ms: UserWidget shows (user data ready)
// 150ms: NotificationsWidget shows
// 2000ms: AnalyticsWidget shows
// → Better perceived performance!

// Kết quả:
// Traditional: Waterfall, slow, complex
// RSC: Parallel, fast, simple
// Streaming: Progressive rendering, best UX
```

## 📊 PHẦN 4: SO SÁNH PATTERNS (30 phút)

### Bảng So Sánh Trade-offs

| Aspect                  | Traditional React (CSR) | SSR (Next.js Pages)           | RSC (Next.js App Router)           |
| ----------------------- | ----------------------- | ----------------------------- | ---------------------------------- |
| **Rendering Location**  | Client only             | Server + Client               | Server (default) + Client (opt-in) |
| **JavaScript Bundle**   | Large (all code)        | Large (all code)              | Small (only Client Components)     |
| **Data Fetching**       | Client (useEffect)      | getServerSideProps            | Direct in component (async)        |
| **SEO**                 | Poor (needs JS)         | Good (HTML ready)             | Excellent (HTML + streaming)       |
| **Time to Interactive** | Slow (2-3s)             | Medium (1-2s)                 | Fast (0.5-1s)                      |
| **Can use hooks**       | ✅ Yes                  | ✅ Yes                        | ⚠️ Only in Client Components       |
| **Direct DB access**    | ❌ No                   | ⚠️ Only in getServerSideProps | ✅ Yes (in Server Components)      |
| **Bundle size**         | ~300-500KB              | ~300-500KB                    | ~50-100KB                          |
| **Loading states**      | Required                | Required                      | Optional (use Suspense)            |
| **Complexity**          | Medium                  | High                          | Medium-High (new concepts)         |
| **Performance**         | Poor                    | Good                          | Excellent                          |
| **When to use**         | Simple SPAs             | SEO-critical pages            | Modern apps, best performance      |

### Decision Tree

```
START: Building a React feature?
    |
    ├─ Need interactivity (onClick, useState, forms)?
    │   |
    │   ├─ Yes → CLIENT COMPONENT ('use client')
    │   │   Examples:
    │   │   - Buttons with click handlers
    │   │   - Forms with controlled inputs
    │   │   - Interactive widgets (dropdowns, modals)
    │   │   - Browser API usage (localStorage, geolocation)
    │   │
    │   └─ No → Continue to next question
    │
    ├─ Need to fetch data?
    │   |
    │   ├─ Yes → SERVER COMPONENT (default)
    │   │   Examples:
    │   │   - Blog post content
    │   │   - Product details
    │   │   - User profiles
    │   │   - Dashboard data
    │   │
    │   └─ No → Continue to next question
    │
    ├─ Heavy computation (markdown, image processing)?
    │   |
    │   ├─ Yes → SERVER COMPONENT
    │   │   Benefits:
    │   │   - Computation happens on server
    │   │   - Zero cost to client bundle
    │   │   - Faster page loads
    │   │
    │   └─ No → Continue to next question
    │
    ├─ Static content display only?
    │   |
    │   ├─ Yes → SERVER COMPONENT (default)
    │   │   Examples:
    │   │   - Headers, footers
    │   │   - Static text blocks
    │   │   - Non-interactive cards
    │   │
    │   └─ No → Analyze specific needs
    │
    └─ MIXED: Use Server Component as wrapper,
       Client Components for interactive parts
```

**Composition Patterns:**

```
Pattern 1: Server wraps Client
✅ GOOD
async function Page() {
  const data = await fetchData();
  return (
    <>
      <ServerHeader data={data} />
      <ClientInteractive data={data} />
      <ServerFooter />
    </>
  );
}

Pattern 2: Client wraps Server (via children)
✅ GOOD (when necessary)
'use client';
function ClientWrapper({ children }) {
  return <div className={useTheme()}>{children}</div>;
}

// Usage
<ClientWrapper>
  <ServerContent />
</ClientWrapper>

Pattern 3: Client imports Server
❌ BAD - Will cause error
'use client';
import ServerComponent from './server';

function Client() {
  return <ServerComponent />; // Error!
}

Pattern 4: Server imports Client
✅ GOOD - Recommended
async function Server() {
  const data = await fetchData();
  return <ClientButton data={data} />;
}
```

```jsx
/**
 * GIẢI THÍCH:
 * Props từ Server → Client phải serializable (JSON.stringify compatible)
 *
 * Đây là một lỗi kinh điển và hầu như ai làm việc với React Server Components (RSC) cũng đều phải đối mặt ít nhất một lần.
 *
 * Lý do gốc rễ là vì Server Component chạy hoàn toàn trên Server, còn ClientButton chạy trên trình duyệt. Để gửi dữ liệu từ Server xuống Client, React * buộc phải chuyển dữ liệu đó thành một chuỗi văn bản (gọi là Serialization - tương tự như JSON.stringify).
 *
 * Hàm (handleClick): Không thể biến một hàm logic thành chuỗi để gửi qua Internet rồi bắt trình duyệt chạy được (vì vấn đề bảo mật và ngữ cảnh môi * trường chạy).
 *
 * Class Instances / Date Object: Khi đi qua "ranh giới" mạng, chúng sẽ bị mất hết các hàm prototype (như .toISOString(), .getDays()), khiến chúng * không còn là đối tượng ban đầu nữa.

 * ✅ SERIALIZABLE:
 * - Strings, numbers, booleans
 * - Plain objects, arrays
 * - null, undefined
 *
 * ❌ NOT SERIALIZABLE:
 * - Functions
 * - Class instances (Date, Map, Set, etc.)
 * - Symbols
 * - undefined (in objects)
 */

// ✅ FIX: Convert non-serializable to serializable

// Server Component
async function Page() {
  const user = await db.users.findById('123');

  // Convert Date to string
  const createdAt = user.createdAt.toISOString();

  return (
    <ClientButton
      user={user}
      createdAt={createdAt} // ✅ String is serializable
    />
  );
}

// Client Component
('use client');
function ClientButton({ user, createdAt }) {
  // Define handler in Client Component
  const handleClick = () => {
    console.log('Clicked!', user);
  };

  // Convert string back to Date if needed
  const date = new Date(createdAt);

  return (
    <button onClick={handleClick}>Created: {date.toLocaleDateString()}</button>
  );
}

// Server Component: Prepare data (fetch, transform)
// Client Component: Handle interactions (events, state)
// Props: Only serializable data
```

```jsx
'use client';

import { useState, useEffect } from 'react';

function ThemeToggleWrapper({ children }) {
  const [theme, setTheme] = useState('light');

  useEffect(() => {
    const saved = localStorage.getItem('theme') || 'light';
    setTheme(saved);
    document.documentElement.setAttribute('data-theme', saved);
  }, []);

  const toggle = () => {
    const next = theme === 'light' ? 'dark' : 'light';
    setTheme(next);
    localStorage.setItem('theme', next);
    document.documentElement.setAttribute('data-theme', next);
  };

  return (
    <button onClick={toggle} className="theme-toggle">
      {theme === 'light' ? '🌙' : '☀️'} {children}
    </button>
  );
}

function SearchBox() {
  const [query, setQuery] = useState('');
  const [isSearching, setIsSearching] = useState(false);
  const router = useRouter();

  const handleSearch = (e) => {
    e.preventDefault();
    if (query.trim()) {
      setIsSearching(true);
      router.push(`/blog/search?q=${encodeURIComponent(query)}`);
    }
  };

  return (
    <form onSubmit={handleSearch} className="search-box">
      <input
        type="text"
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="Tìm kiếm bài viết..."
        className="search-input"
      />
      <button type="submit" disabled={isSearching} className="search-button">
        {isSearching ? '🔍 Đang tìm...' : '🔍 Tìm kiếm'}
      </button>
    </form>
  );

'use client';

import { useState, useOptimistic } from 'react';

function LikeButton({ postId, initialLikes }) {
  const [liked, setLiked] = useState(false);
  const [optimisticLikes, addOptimisticLike] = useOptimistic(
    initialLikes,
    (state, increment) => state + increment
  );

  const handleLike = async () => {
    if (liked) return;

    addOptimisticLike(1);
    setLiked(true);

    try {
      await fetch(`/api/posts/${postId}/like`, { method: 'POST' });
      localStorage.setItem(`liked-${postId}`, 'true');
    } catch (error) {
      // Rollback nếu cần trong thực tế
      console.error('Like failed:', error);
    }
  };

  return (
    <button
      onClick={handleLike}
      className={`like-button ${liked ? 'liked' : ''}`}
      aria-label="Thích bài viết"
    >
      {liked ? '❤️' : '🤍'} {optimisticLikes}
    </button>
  );
}

'use client';

function ShareButton({ title, slug }) {
  const handleShare = async () => {
    const url = `${window.location.origin}/blog/${slug}`;

    if (navigator.share) {
      try {
        await navigator.share({ title, url });
      } catch (err) {
        console.log('Share cancelled');
      }
    } else {
      // Fallback: copy to clipboard
      await navigator.clipboard.writeText(url);
      alert('Đã sao chép link vào clipboard!');
    }
  };

  return (
    <button onClick={handleShare} className="share-button" aria-label="Chia sẻ bài viết">
      🔗 Chia sẻ
    </button>
  );
}

'use client';

import { useState } from 'react';

function NewsletterForm() {
  const [email, setEmail] = useState('');
  const [status, setStatus] = useState('idle'); // idle | loading | success | error

  const handleSubmit = async (e) => {
    e.preventDefault();
    if (!email.includes('@')) {
      setStatus('error');
      return;
    }

    setStatus('loading');
    try {
      await fetch('/api/newsletter', {
        method: 'POST',
        body: JSON.stringify({ email }),
        headers: { 'Content-Type': 'application/json' },
      });
      setStatus('success');
      setEmail('');
    } catch (err) {
      setStatus('error');
    }
  };

  return (
    <div className="sidebar-section newsletter">
      <h4>📬 Đăng ký Newsletter</h4>
      <p>Nhận thông báo khi có bài viết mới!</p>
      <form onSubmit={handleSubmit}>
        <input
          type="email"
          value={email}
          onChange={(e) => setEmail(e.target.value)}
          placeholder="email@example.com"
          disabled={status === 'loading'}
        />
        <button type="submit" disabled={status === 'loading'}>
          {status === 'loading' ? 'Đang gửi...' : 'Đăng ký'}
        </button>
        {status === 'success' && <p className="msg-success">🎉 Đăng ký thành công!</p>}
        {status === 'error' && <p className="msg-error">❌ Vui lòng kiểm tra lại email.</p>}
      </form>
    </div>
  );
}

'use client';

import { useState, useEffect } from 'react';

function ReadingProgressBar() {
  const [progress, setProgress] = useState(0);

  useEffect(() => {
    const updateProgress = () => {
      const scrollTop = window.scrollY;
      const docHeight = document.documentElement.scrollHeight - window.innerHeight;
      const scrollPercent = docHeight > 0 ? (scrollTop / docHeight) * 100 : 0;
      setProgress(scrollPercent);
    };

    window.addEventListener('scroll', updateProgress);
    return () => window.removeEventListener('scroll', updateProgress);
  }, []);

  return (
    <div className="progress-bar-container">
      <div
        className="progress-bar"
        style={{ width: `${progress}%` }}
        aria-hidden="true"
      />
    </div>
  );
}

// 📁 FILE: app/components/CommentSection.jsx
// 🖥️  CLIENT COMPONENT

'use client';

import { useState, useEffect } from 'react';

function CommentSection({ postId }) {
  const [comments, setComments] = useState([]);
  const [newComment, setNewComment] = useState('');
  const [isSubmitting, setIsSubmitting] = useState(false);

  useEffect(() => {
    fetch(`/api/posts/${postId}/comments`)
      .then((r) => r.json())
      .then((data) => setComments(data));
  }, [postId]);

  const handleSubmit = async (e) => {
    e.preventDefault();
    if (!newComment.trim()) return;

    setIsSubmitting(true);
    try {
      const res = await fetch(`/api/posts/${postId}/comments`, {
        method: 'POST',
        body: JSON.stringify({ content: newComment }),
        headers: { 'Content-Type': 'application/json' },
      });
      const saved = await res.json();
      setComments((prev) => [saved, ...prev]);
      setNewComment('');
    } finally {
      setIsSubmitting(false);
    }
  };

  return (
    <section className="comment-section">
      <h3>Bình luận ({comments.length})</h3>

      <form onSubmit={handleSubmit} className="comment-form">
        <textarea
          value={newComment}
          onChange={(e) => setNewComment(e.target.value)}
          placeholder="Viết bình luận của bạn..."
          rows={4}
        />
        <button type="submit" disabled={isSubmitting}>
          {isSubmitting ? 'Đang gửi...' : 'Gửi bình luận'}
        </button>
      </form>

      <div className="comment-list">
        {comments.map((comment) => (
          <div key={comment.id} className="comment-item">
            <strong>{comment.author}</strong>
            <p>{comment.content}</p>
            <span>{new Date(comment.createdAt).toLocaleDateString('vi-VN')}</span>
          </div>
        ))}
      </div>
    </section>
  );
}

/*
1. ❌ VẤN ĐỀ: Serialization Props từ Server → Client
   - Functions, Date objects, class instances không thể truyền qua props
   - Ví dụ: Không thể truyền `onClick` handler từ Server xuống Client

   ✅ GIẢI PHÁP:
   - Chuyển Date thành ISO string trên Server, parse lại ở Client nếu cần
   - Event handlers (onClick, onSubmit) chỉ định nghĩa trong Client Component
   - Chỉ truyền plain objects, arrays, strings, numbers, booleans

2. ❌ VẤN ĐỀ: Client Component KHÔNG THỂ import trực tiếp Server Component
   - Nếu CommentSection cố gắng import AuthorInfo → Lỗi build

   ✅ GIẢI PHÁP:
   - Dùng composition pattern: Truyền Server Component qua children prop
   - Hoặc fetch data ở Server rồi truyền plain data xuống Client

3. ❌ VẤN ĐỀ: "use client" ở đầu file làm toàn bộ file thành Client Bundle
   - Nếu để chung logic tĩnh vào file có 'use client', cả file sẽ xuống client

   ✅ GIẢI PHÁP:
   - Tách nhỏ components: Leaf architecture (Client ở lá cây)
   - Giữ 'use client' chỉ ở các file thực sự cần interactivity
   - Dùng Server Component wrapper bọc ngoài Client Component

4. ❌ VẤN ĐỀ: Third-party libraries không tương thích với RSC
   - Một số thư viện (ví dụ: charting, markdown editor) dùng window/document ngay khi import
   - Gây lỗi "window is not defined" khi Server Component render trên server

   ✅ GIẢI PHÁP:
   - Dùng dynamic import với ssr: false trong Client Component
   - Hoặc tìm thư viện có hỗ trợ Server Components (ví dụ: react-markdown cho RSC)
   - Cách import động: const Editor = dynamic(() => import('editor-lib'), { ssr: false })

5. ❌ VẤN ĐỀ: Context Providers là Client Component, làm children thành Client
   - Nếu bọc toàn bộ app trong ThemeProvider (use client), tất cả children đều hydrate

   ✅ GIẢI PHÁP:
   - Dùng pattern "Server Component as children của Client Provider"
   - Hoặc lưu theme vào Cookie để Server đọc được và render đúng class ngay từ đầu
   - Tránh dùng Context cho data có thể lấy từ Server

6. ❌ VẤN ĐỀ: Debugging RSC khó khăn hơn CSR
   - Lỗi trên server không hiện rõ trong browser console
   - RSC Payload khó đọc khi inspect network tab

   ✅ GIẢI PHÁP:
   - Dùng React DevTools phiên bản mới nhất hỗ trợ RSC
   - Log lỗi rõ ràng ở Server Component với try/catch
   - Dùng Error Boundaries (Client) để bắt lỗi render từ Server

7. ❌ VẤN ĐỀ: Caching và Revalidation phức tạp
   - Server Component cache mặc định, có thể trả về data cũ
   - User không thấy comments mới ngay lập tức nếu không cấu hình đúng

   ✅ GIẢI PHÁP:
   - Dùng `revalidatePath()` hoặc `revalidateTag()` khi có mutation
   - Hoặc dùng Client Component cho phần cần real-time (như CommentSection đã làm)
   - Cân nhắc `cache: 'no-store'` cho data nhạy cảm thời gian
*/
```

```jsx
// 'use client';
import { useState, useEffect } from 'react';

function GlobalSearch({ tenantId }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const [isOpen, setIsOpen] = useState(false);

  useEffect(() => {
    if (query.length < 2) {
      setResults([]);
      return;
    }

    const timer = setTimeout(async () => {
      const res = await fetch(`/api/search?tenantId=${tenantId}&q=${encodeURIComponent(query)}`);
      const data = await res.json();
      setResults(data);
    }, 300);

    return () => clearTimeout(timer);
  }, [query, tenantId]);

  return (
    <div className="global-search">
      <input
        type="text"
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        onFocus={() => setIsOpen(true)}
        placeholder="Tìm kiếm dữ liệu... (Ctrl+K)"
        className="search-input"

// 'use client';

function UserGrowthChart({ data }) {
  // 📌 Line chart giả lập với SVG
  const width = 400;
  const height = 200;
  const padding = 20;

  const maxTotal = Math.max(...data.map((d) => d.total));
  const points = data.map((d, i) => {
    const x = padding + (i / (data.length - 1)) * (width - 2 * padding);
    const y = height - padding - (d.total / maxTotal) * (height - 2 * padding);
    return `${x},${y}`;
  }).join(' ');

  return (
    <div className="chart-card">
      <h3>Tăng trưởng người dùng</h3>
      <svg viewBox={`0 0 ${width} ${height}`} className="line-chart">
        {/* Grid lines */}
        {[0, 25, 50, 75, 100].map((pct) => (
          <line
            key={pct}
            x1={padding}
            y1={height - padding - (pct / 100) * (height - 2 * padding)}
            x2={width - padding}
            y2={height - padding - (pct / 100) * (height - 2 * padding)}
            stroke="#eee"
          />
        ))}

        {/* Line */}
        <polyline
          fill="none"
          stroke="#4f46e5"
          strokeWidth="3"
          points={points}
        />

        {/* Data points */}
        {data.map((d, i) => {
          const x = padding + (i / (data.length - 1)) * (width - 2 * padding);
          const y = height - padding - (d.total / maxTotal) * (height - 2 * padding);
          return (
            <g key={d.month}>
              <circle cx={x} cy={y} r="5" fill="#4f46e5" />
              <text x={x} y={height - 5} textAnchor="middle" fontSize="12">{d.month}</text>
            </g>
          );
        })}
      </svg>
      <div className="chart-legend">
        <span className="legend-item">🔵 Tổng users</span>
        <span className="legend-item">🟢 Active users</span>
      </div>
    </div>
  );
}

import { useState } from 'react';

function TablePagination({ totalItems, pageSize }) {
  const [currentPage, setCurrentPage] = useState(1);
  const totalPages = Math.ceil(totalItems / pageSize);

  const pages = Array.from({ length: totalPages }, (_, i) => i + 1);

  return (
    <div className="table-pagination">
      <button
        onClick={() => setCurrentPage((p) => Math.max(1, p - 1))}
        disabled={currentPage === 1}
      >
        ← Trước
      </button>

      <div className="page-numbers">
        {pages.map((p) => (
          <button
            key={p}
            onClick={() => setCurrentPage(p)}
            className={p === currentPage ? 'active' : ''}
          >
            {p}
          </button>
        ))}
      </div>

      <button
        onClick={() => setCurrentPage((p) => Math.min(totalPages, p + 1))}
        disabled={currentPage === totalPages}
      >
        Sau →
      </button>

      <span className="pagination-info">
        Hiển thị {((currentPage - 1) * pageSize) + 1} - {Math.min(currentPage * pageSize, totalItems)} / {totalItems}
      </span>
    </div>
  );
}

// 'use client';

function ExportButton({ data, filename }) {
  const handleExport = () => {
    // 📌 Chỉ chạy trên browser: tạo CSV blob và download
    const headers = Object.keys(data[0] || {}).join(',');
    const rows = data.map((row) => Object.values(row).join(',')).join('\n');
    const csv = `${headers}\n${rows}`;

    const blob = new Blob([csv], { type: 'text/csv;charset=utf-8;' });
    const link = document.createElement('a');
    link.href = URL.createObjectURL(blob);
    link.download = filename;
    link.click();
    URL.revokeObjectURL(link.href);
  };

  return (
    <button onClick={handleExport} className="btn-export">
      📥 Xuất CSV
    </button>
  );
}

// 'use client';
import { useState, useEffect } from 'react';

function NotificationCenter({ tenantId, userId, initialAlerts }) {
  const [notifications, setNotifications] = useState(initialAlerts || []);
  const [isOpen, setIsOpen] = useState(false);
  const [unreadCount, setUnreadCount] = useState(initialAlerts?.length || 0);
  const [connectionStatus, setConnectionStatus] = useState('connecting');

  useEffect(() => {
    // 📌 Mở SSE connection để nhận real-time updates
    const eventSource = new EventSource(`/api/sse?tenantId=${tenantId}&userId=${userId}`);

    eventSource.onopen = () => setConnectionStatus('connected');

    eventSource.onmessage = (event) => {
      const data = JSON.parse(event.data);
      setNotifications((prev) => [data, ...prev]);
      setUnreadCount((c) => c + 1);
    };

    eventSource.onerror = () => {
      setConnectionStatus('error');
      eventSource.close();
      // 📌 Tự động reconnect sau 5 giây
      setTimeout(() => setConnectionStatus('reconnecting'), 5000);
    };

    return () => eventSource.close();
  }, [tenantId, userId]);

  return (
    <div className="notification-center">
      <button
        className="notification-bell"
        onClick={() => setIsOpen(!isOpen)}
        aria-label={`${unreadCount} thông báo chưa đọc`}
      >
        🔔
        {unreadCount > 0 && <span className="notification-badge">{unreadCount}</span>}
        <span className={`connection-status ${connectionStatus}`} aria-hidden="true" />
      </button>

      {isOpen && (
        <div className="notification-dropdown">
          <div className="notification-header">
            <h4>Thông báo</h4>
            <button onClick={() => setUnreadCount(0)}>Đánh dấu đã đọc</button>
          </div>

          {notifications.length === 0 ? (
            <p className="empty-state">Không có thông báo mới</p>
          ) : (
            <ul className="notification-list">
              {notifications.map((n, i) => (
                <li key={i} className={`notification-item ${n.severity}`}>
                  <span className="notification-dot" />
                  <p>{n.message || n.title}</p>
                  <time>{n.createdAt ? new Date(n.createdAt).toLocaleTimeString('vi-VN') : 'Vừa xong'}</time>
                </li>
              ))}
            </ul>
          )}
        </div>
      )}
    </div>
  );
}

function InviteMemberModal({ tenantId, invitedBy }) {

  const [isSubmitting, setIsSubmitting] = useState(false);
  const [result, setResult] = useState(null);

  const handleInvite = async (e) => {
    e.preventDefault();
    setIsSubmitting(true);

    try {
      const res = await fetch(`/api/tenants/${tenantId}/invitations`, {
        method: 'POST',
        body: JSON.stringify({ email, role, invitedBy }),
        headers: { 'Content-Type': 'application/json' },
      });

      if (res.ok) {
        setResult({ type: 'success', message: `Đã gửi lời mời đến ${email}` });
        setEmail('');
      } else {
        const err = await res.json();
        setResult({ type: 'error', message: err.message || 'Gửi lời mời thất bại' });
      }
    } catch (err) {
      setResult({ type: 'error', message: 'Lỗi mạng, vui lòng thử lại' });
    } finally {
      setIsSubmitting(false);
    }
  };

  return (
    <div className="invite-modal">
      <h3>Mời thành viên mới</h3>
      <form onSubmit={handleInvite}>
        <div className="form-group">
          <label>Email</label>
          <input
            type="email"
            value={email}
            onChange={(e) => setEmail(e.target.value)}
            placeholder="colleague@company.com"
            required
          />
        </div>

        <div className="form-group">
          <label>Vai trò</label>
          <select value={role} onChange={(e) => setRole(e.target.value)}>
            <option value="viewer">Viewer (Chỉ xem)</option>
            <option value="member">Member (Thành viên)</option>
            <option value="manager">Manager (Quản lý)</option>
          </select>
        </div>

        <button type="submit" disabled={isSubmitting} className="btn-primary">
          {isSubmitting ? 'Đang gửi...' : 'Gửi lời mời'}
        </button>

// 'use client';
import { Component } from 'react';

class WidgetErrorBoundary extends Component {
  constructor(props) {
    super(props);
    this.state = { hasError: false, error: null };
  }

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, info) {
    // 📌 Log lỗi đến monitoring service
    console.error('Widget Error:', error, info);
  }

  render() {
    if (this.state.hasError) {
      return (
        <div className="widget-error">
          <h4>⚠️ Không thể tải widget</h4>
          <p>Widget này gặp lỗi. Vui lòng refresh trang hoặc liên hệ support.</p>
          <button onClick={() => this.setState({ hasError: false })}>
            Thử lại
          </button>
        </div>
      );
    }
    return this.props.children;
  }
}
```

### Câu Hỏi Phỏng Vấn

**Junior:**

**1. RSC là gì?**  
React Server Components (RSC) là components React chỉ chạy trên server và không bao giờ gửi JavaScript xuống trình duyệt. Thay vì HTML, server stream một định dạng đặc biệt gọi là **RSC Payload** chứa UI tree, props serialized, và reference đến Client Components. Điều này cho phép truy cập database trực tiếp, giảm bundle size về zero cho phần tĩnh, và fetch data song song trên server trước khi gửi xuống client.

**2. Server Component khác Client Component như thế nào?**  
Server Component chạy **chỉ trên server** (có thể khai báo `async`), truy cập được DB/file-system/env variables, không dùng được hooks hay event handlers, và **không gửi JS xuống client**. Client Component (có directive `'use client'`) chạy trên trình duyệt, dùng được tất cả hooks (`useState`, `useEffect`), event handlers (`onClick`), và browser APIs (`localStorage`, `window`), nhưng JS của nó **được gửi xuống và hydrate**. Server Component là default; Client Component chỉ dùng khi thực sự cần tương tác.

**3. Khi nào dùng Server Component?**  
Dùng Server Component khi cần **fetch data**, thực hiện **tính toán nặng** (parse markdown, xử lý ảnh), hiển thị **nội dung tĩnh**, giữ **thông tin nhạy cảm** (API keys) trên server, hoặc **giảm kích thước bundle** client. Quy tắc nhớ: bắt đầu với Server Component cho mọi thứ, chỉ chuyển sang Client Component khi component đó cần interactivity (`useState`, `onClick`, `useEffect`, hoặc browser APIs).

---

**Mid:**

**1. Explain RSC architecture và rendering flow.**  
Kiến trúc RSC chia cây component thành **Server Components** (mặc định, chạy trên server) và **Client Components** (opt-in qua `'use client'`, chạy trên browser). Quá trình render: (1) Server chạy SC, fetch data song song từ DB/API, render ra **RSC Payload** — một stream text đặc biệt mô tả UI tree, props, và placeholder (`$L1`) cho CC. (2) Browser nhận payload, render SC thành HTML tĩnh ngay lập tức. (3) Browser tải JS chunks của CC theo reference trong payload. (4) React **selective hydration**: chỉ "gắn hồn" JavaScript vào các node CC, còn SC để nguyên HTML tĩnh. Kết quả là FCP nhanh, bundle nhỏ, và TTI thấp vì không hydrate toàn bộ tree.

**2. So sánh RSC vs SSR vs CSR.**  
**CSR (Client-Side Rendering)**: toàn bộ logic chạy trên browser, server gửi HTML trống + JS bundle lớn. SEO kém, TTI chậm. **SSR (Next.js Pages Router)**: server render HTML trước rồi gửi xuống, nhưng **toàn bộ component code** vẫn được gửi xuống để hydrate toàn bộ tree. Bundle vẫn lớn, hydration cost cao. **RSC (App Router)**: Server Components chỉ chạy server, zero JS bundle; chỉ Client Components được gửi JS xuống. Hỗ trợ **streaming** từng phần qua Suspense, **selective hydration**, và **parallel data fetching** trực tiếp trong component. RSC kết hợp ưu điểm SEO của SSR với bundle size nhỏ hơn cả CSR.

**3. Composition patterns trong RSC.**  
Có 5 quy tắc composition:

- (1) **SC import CC** ✅ — Server Component có thể import và render Client Component bình thường.
- (2) **SC import SC** ✅ — hoàn toàn được phép.
- (3) **CC import CC** ✅ — Client Component import Client Component khác.
- (4) **CC import SC trực tiếp** ❌ — không được phép vì SC không chạy trên client.
- (5) **CC nhận SC qua prop `children`** ✅ — đây là pattern cốt lõi: Client Component wrapper nhận Server Component đã render qua prop `children` hoặc slot prop.
  Best practice là **"push interactivity to the leaves"**: giữ SC ở root và layout, chỉ để CC ở các lá cây cần tương tác.

---

**Senior:**

**1. Design RSC architecture cho large-scale app.**  
Thiết kế theo **Leaf Architecture**:

- root và layout là Server Components (fetch data, auth, tenant resolution), chỉ để CC ở các lá cây cần tương tác (button, form, chart, modal).
- Sử dụng **parallel fetching** ở root page với `Promise.all` để giảm TTFB.
- Bọc từng section (charts, table, activity) trong **Suspense boundaries** riêng biệt để stream UI từng phần.
- Auth và role-based access phải resolve trên server trước khi render — filter sensitive columns và routes server-side.
- Mỗi widget phức tạp bọc trong **Error Boundary** riêng để tránh 1 lỗi làm crash trang.
- Dùng **Server Component làm data container**, truyền plain serialized props xuống CC leaves.

**2. Migration strategy từ traditional React sang RSC.**

- **Phase 1 — New features only**: các tính năng mới viết bằng RSC, giữ nguyên codebase cũ.
- **Phase 2 — High-traffic pages**: convert các trang quan trọng (product page, blog, dashboard) sang App Router, đo lường performance trước/sau.
- **Phase 3 — Gradual rollout**: từ từ chuyển các module còn lại, dùng composition để kết nối SC mới với CC cũ (truyền SC qua `children`).
- **Phase 4 — Complete migration**: loại bỏ `getServerSideProps`, `getStaticProps`, và các pattern cũ. Không bao giờ rewrite toàn bộ 1 lúc; luôn giữ khả năng rollback và đo lường bundle size, TTI, LCP sau từng phase.

**3. Performance optimization với RSC.**

- (1) **Song song hóa fetch**: dùng `Promise.all` ở SC thay vì waterfall `useEffect`.
- (2) **Streaming + Suspense**: bọc các phần chậm (charts, heavy data) trong `<Suspense>` để browser render skeleton và FCP nhanh.
- (3) **Leaf CC**: tránh đặt `'use client'` ở layout cao (như provider bao toàn app) vì sẽ buộc toàn bộ children thành client bundle; thay vào đó dùng pattern "Client wrapper + Server children".
- (4) **Serialize đúng cách**: chỉ truyền plain objects, chuyển `Date` → ISO string, tránh truyền functions qua boundary.
- (5) **Caching**: tận dụng `fetch` cache và `unstable_cache` của Next.js cho SC; dùng `revalidatePath`/`revalidateTag` sau mutation.
- (6) **Theme/Config**: lưu vào Cookie thay vì Context để server đọc được và render đúng class ngay từ đầu, tránh hydration mismatch.

**4. Trade-offs và khi nào KHÔNG nên dùng RSC.**

- **Trade-offs**: learning curve cao (phân biệt SC/CC, serialization, không dùng Context trong SC); debugging phức tạp (lỗi server không hiện trên browser console); giới hạn serialization (không truyền function, class instances qua boundary); nhiều third-party library chưa tương thích (dùng `window` ngay khi import); caching và revalidation phức tạp hơn CSR.
- **Khi không nên dùng**:
  - (a) **App đơn giản/SPA nội bộ** không cần SEO hoặc bundle đã đủ nhỏ;
  - (b) **Prototype/MVP cần ship nhanh** — RSC yêu cầu Next.js App Router, infra phức tạp hơn;
  - (c) **Heavy real-time collaboration** (như Google Docs, Figma) — cần toàn bộ app chạy trên client để sync state liên tục;
  - (d) **Thư viện UI chỉ hỗ trợ client** và không thể tách phần tĩnh ra SC (ví dụ: các form builder phức tạp dùng `ref` và `useEffect` dày đặc ở root). Trong những trường hợp đó, CSR hoặc SSR truyền thống vẫn là lựa chọn hợp lý hơn.

```jsx
import { useSyncExternalStore } from 'react';

// Ví dụ 1: Subscribe vào một custom store đơn giản
const createStore = (initialState) => {
  let state = initialState;
  const listeners = new Set();

  return {
    getState: () => state,
    setState: (next) => {
      state = typeof next === 'function' ? next(state) : next;
      listeners.forEach((l) => l());
    },
    subscribe: (listener) => {
      listeners.add(listener);
      return () => listeners.delete(listener);
    },
  };
};

const store = createStore({ count: 0 });

const useStore = (selector = (s) => s) => {
  // getSnapshot: lấy giá trị hiện tại (phải immutable hoặc memoized)
  // subscribe: đăng ký listener
  // getServerSnapshot: dùng cho SSR (optional)
  return useSyncExternalStore(
    store.subscribe,
    () => selector(store.getState()),
    () => selector(store.getState()), // getServerSnapshot
  );
};

const Counter = () => {
  const count = useStore((s) => s.count);
  return (

      <span>{count}</span>
      <button
        onClick={() => store.setState((prev) => ({ count: prev.count + 1 }))}
      >
        +
      </button>
    </div>
  );
};

 * Chỉ chạy trong development mode, không ảnh hưởng production.

import { useDebugValue, useState, useEffect } from 'react';

const useOnlineStatusWithDebug = () => {
  const [isOnline, setIsOnline] = useState(true);

  useEffect(() => {
    const onOnline = () => setIsOnline(true);
    const onOffline = () => setIsOnline(false);
    window.addEventListener('online', onOnline);
    window.addEventListener('offline', onOffline);
    return () => {
      window.removeEventListener('online', onOnline);
      window.removeEventListener('offline', onOffline);
    };

  useDebugValue(isOnline ? 'Online' : 'Offline');

  useDebugValue({ isOnline, timestamp: Date.now() }, (value) => {

| useId             | ID cho a11y, labels, forms             | Thường   |
| useImperativeHandle| Expose methods qua ref (cẩn thận)     | Hiếm     |
| useSyncExternalStore| Tích hợp external store/browser API  | Hiếm     |
| useDebugValue     | Debug custom hooks trong DevTools      | Rất hiếm |

Nguyên tắc:
- useId: Thay thế Math.random() / uuid() cho a11y attributes
- useImperativeHandle: Chỉ khi imperative API là bắt buộc (focus, scroll, play media)
- useSyncExternalStore: Thay thế useEffect + useState khi subscribe external data
- useDebugValue: Optional, chỉ khi team cần debug hook phức tạp
*/
```

**2. Actions API — Thay đổi lớn nhất về form & async**

React 19 giới thiệu **Actions** — pattern xử lý async mutations tự động, thay thế cách viết form thủ công bằng `useState` + `useEffect`.

|               | React 18                                      | React 19                            |
| ------------- | --------------------------------------------- | ----------------------------------- |
| Form handling | `useState` loading/error/success + `onSubmit` | `useActionState` hoặc `action` prop |
| Optimistic UI | Tự viết rollback logic                        | `useOptimistic` — rollback tự động  |
| Pending state | Tự quản lý `isLoading`                        | Tự động qua `isPending`             |

```jsx
// React 19: Form với Actions
import { useActionState } from 'react';

function UpdateName() {
  const [error, submitAction, isPending] = useActionState(
    async (previousState, formData) => {
      const name = formData.get('name');
      const result = await updateName(name);
      if (result.error) return result.error;
      return null;
    },
    null, // initial state
  );

  return (
    <form action={submitAction}>
      <input name='name' />
      <button disabled={isPending}>{isPending ? 'Saving...' : 'Save'}</button>
      {error && <p>{error}</p>}
    </form>
  );
}
```

```jsx
import { useFormStatus } from 'react-dom';

// Component con — tự động biết form cha đang submit hay không
function SubmitButton() {
```

**3. `use()` Hook — Đọc Promise & Context ở bất cứ đâu**

Hook mới `use()` cho phép **đọc Promise và Context trực tiếp trong component**, không cần `useEffect` + `useState` nữa.

```jsx
// React 19: Đơn giản hóa data fetching
import { use, Suspense } from 'react';

function Note({ id }) {
  const note = use(fetchNote(id)); // Đọc Promise trực tiếp
  const theme = use(ThemeContext); // Đọc Context (có thể dùng conditionally!)

  return (
    <section className={theme}>
      <h1>{note.title}</h1>
    </section>
  );
}

// React 18 tương đương: useEffect + useState + useContext (dài gấp 3-4 lần)
```

> 💡 `use()` có thể gọi **bên trong `if` hoặc loop** — khác với `useContext` bắt buộc top-level.

**4. useEffectEvent (React 19.2) — Tách logic Event khỏi Effect**

API mới trong React 19.2 giúp tách phần "event" ra khỏi `useEffect`, tránh re-run effect không cần thiết.

```jsx
import { useEffect, useEffectEvent } from 'react';

function ChatRoom({ roomId, theme }) {
  // theme thay đổi → KHÔNG re-run effect
  const onVisit = useEffectEvent((visitedRoomId) => {
    logVisit(visitedRoomId, theme); // theme luôn là giá trị mới nhất
  });

  useEffect(() => {
    const connection = createConnection(roomId);
    connection.connect();

    onVisit(roomId); // Gọi event khi connect

    return () => connection.disconnect();
  }, [roomId]); // Chỉ phụ thuộc roomId, không cần theme
}
```

**Vấn đề giải quyết:** Trước đây nếu `theme` thay đổi, effect phải re-run vì nó nằm trong dependency array. Với `useEffectEvent`, `theme` luôn là giá trị mới nhất mà không trigger re-run.
