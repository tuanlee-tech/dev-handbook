1.  **Suspense boundaries hoạt động như thế nào?**

- _Đáp án: Component throw promise khi cần data → React catch promise ở Suspense boundary gần nhất → Show fallback UI → Promise resolve → React re-render component với data._

2. **Thứ tự đúng: Error Boundary và Suspense?**
   - _Đáp án: Error Boundary bọc ngoài Suspense bọc Component. (`<ErrorBoundary><Suspense fallback={...}><Component /></Suspense></ErrorBoundary>`)_

3. **Tại sao Error Boundary bọc ngoài Suspense?**
   - _Đáp án: Nếu Suspense bọc ngoài, khi component throw error, Suspense sẽ suspend cả Error Boundary → Error Boundary không active để catch error được._

### 1.1 Vấn Đề Thực Tế

**Khi component throw error, điều gì xảy ra?**

```jsx
  const [products, setProducts] = useState([]);

  useEffect(() => {
    fetchProducts()
      .then((data) => setProducts(data.products)) // Assume data.products exists
      .catch((err) => console.error(err)); // Log error nhưng...

  // 💥 Nếu data structure khác (data.items instead of data.products)
  // products là undefined → .map() throws error → WHITE SCREEN OF DEATH!

          <ProductCard
            key={product.id}
```

### 1.2 Giải Pháp: Error Boundaries

**Error Boundary = try-catch cho React components**

```jsx
// ✅ Error Boundary giữ app hoạt động
<ErrorBoundary fallback={<div>Widget failed to load</div>}>
  <ChartWidget />
</ErrorBoundary>

// Flow:
// ChartWidget throws error
//     ↓
// ErrorBoundary catches
//     ↓
// Show fallback UI (Widget failed)
//     ↓
// Other widgets still work!
//     ↓
// User can continue using app
```

**Benefits:**

1. **Graceful degradation** - App continues working
2. **Isolated failures** - One component fails ≠ App fails
3. **Better UX** - Show error message instead of blank screen
4. **Recovery options** - Provide retry button
5. **Error logging** - Track errors in production

### 1.3 Mental Model

**Error Boundary như circuit breaker:**

```
Normal Flow:
Parent → Child → GrandChild (all render fine)

With Error:
Parent → Child → GrandChild 💥
    ↓
ErrorBoundary catches
    ↓
Show fallback instead of GrandChild
    ↓
Parent & Child still render fine
```

**Analogy: Building Fire Doors**

```
Without Error Boundary:
Fire in Room A → Spreads to entire building → Total loss

With Error Boundary:
Fire in Room A → Fire door closes → Rest of building safe
```

**Hierarchy:**

```jsx
<App>                              // ← Top-level error boundary
  <ErrorBoundary fallback={...}>
    <Dashboard>                    // ← Feature-level error boundary
      <ErrorBoundary fallback={...}>
        <Widget />                 // ← Component that might fail
      </ErrorBoundary>
    </Dashboard>
  </ErrorBoundary>
</App>
```

### Demo 1: Class-based Error Boundary (Legacy Pattern) ⭐

```jsx
/**
 * Demo: Basic Error Boundary với class component
 *
 * ⚠️ LƯU Ý: Hiện tại (2025), Error Boundary PHẢI là class component
 * React chưa có hooks cho error boundaries
 * Demo này để HIỂU cơ chế - Production nên dùng react-error-boundary library
 */

import { Component } from 'react';

// ✅ Basic Error Boundary
class ErrorBoundary extends Component {
  constructor(props) {
    super(props);
    this.state = { hasError: false };
  }

  // Called when child component throws error during render
  static getDerivedStateFromError(error) {
    // Update state để next render show fallback UI
    return { hasError: true };
  }

  // Called after error is caught - for logging
  componentDidCatch(error, errorInfo) {
    // errorInfo.componentStack: Stack trace của component tree
    console.error('Error caught by boundary:', error);
    console.error('Component stack:', errorInfo.componentStack);

    // TODO: Send to error tracking service (Sentry, LogRocket, etc.)
    // logErrorToService(error, errorInfo);
  }

  render() {
    if (this.state.hasError) {
      // Fallback UI
      return (
        <div style={{ padding: 20, border: '2px solid red', borderRadius: 8 }}>
          <h2>⚠️ Something went wrong</h2>
          <p>We're sorry for the inconvenience.</p>
        </div>
      );
    }

    return this.props.children;
  }
}

// Component that throws error
function BuggyComponent() {
  const [count, setCount] = useState(0);

  if (count === 3) {
    // Simulate error at specific count
    throw new Error('I crashed at count 3!');
  }

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Increment (crash at 3)
      </button>
    </div>
  );
}

// Usage
function App() {
  return (
    <div>
      <h1>Error Boundary Demo</h1>

      <ErrorBoundary>
        <BuggyComponent />
      </ErrorBoundary>

      <div style={{ marginTop: 20, padding: 10, backgroundColor: '#f0f0f0' }}>
        <p>This section is outside the error boundary</p>
        <p>It will still render even if BuggyComponent crashes!</p>
      </div>
    </div>
  );
}

// Kết quả:
// 1. Click button 3 lần
// 2. Component throws error
// 3. Error Boundary catches → Show fallback UI
// 4. Bottom section still renders normally
```

```jsx
 * Demo: Error Boundary với retry mechanism

class ErrorBoundaryWithRetry extends Component {
  constructor(props) {
    super(props);
    this.state = {
      hasError: false,
      error: null,
    };
  }

  static getDerivedStateFromError(error) {
    return {
      hasError: true,
      error,
    };
  }

  componentDidCatch(error, errorInfo) {
    console.error('Error:', error);
    console.error('Error Info:', errorInfo);
  }

  // Reset error state → Trigger re-render of children
  handleReset = () => {
    this.setState({
      hasError: false,
      error: null,
    });
  };

  render() {
    if (this.state.hasError) {
      return (
        <div
          style={{
            padding: 24,
            border: '2px solid #ef4444',
            borderRadius: 8,
            backgroundColor: '#fee',
            textAlign: 'center',
          }}
        >
          <h2 style={{ color: '#dc2626', marginBottom: 12 }}>
            ⚠️ Error Occurred
          </h2>
          <p style={{ color: '#991b1b', marginBottom: 8 }}>
            {this.state.error?.message || 'Unknown error'}
          </p>
          <button
            onClick={this.handleReset}
            style={{
              padding: '10px 20px',
              backgroundColor: '#ef4444',
              color: 'white',
              border: 'none',
```

```
## Rationale

### User Experience:
1. Navigation failures → Minimal nav fallback, dashboard vẫn usable
2. Widget failures → Isolated to that widget only
3. Settings failures → Error shown nhưng dashboard vẫn work
4. Help widget → Silent failure (not critical)

### Recovery:
- Auto-retry 1 lần cho transient errors (exponential backoff)
- Manual retry button cho persistent errors
- Fallback to cached data where applicable
- Clear error messages với actionable steps

## Consequences

**Positive:**
+ Excellent UX — isolated failures
+ Clear boundaries for debugging
+ Flexible recovery per component
+ Easy to add new widgets
+ Reduced support burden

**Negative:**
- More boundaries to maintain (4 levels)
- More boilerplate code
- Need consistent fallback UI design
- Complex error logging

**Trade-offs accepted:**
- Code complexity for UX reliability
```

```jsx
class ErrorLoggingService {
  constructor() {
    this.errors = [];
    this.listeners = [];
  }

  log(error, errorInfo, context = {}) {
    const errorEntry = {
      id: `${Date.now()}-${Math.random().toString(36).slice(2, 9)}`,
      timestamp: new Date().toISOString(),
      message: error.message,
      stack: error.stack,
      componentStack: errorInfo?.componentStack,
      context,
      resolved: false,
    };

    this.errors.push(errorEntry);
    this.notifyListeners();

    // Dev logging
    console.error('🔴 Production Error:', errorEntry);

    // TODO: Production — gửi lên Sentry/LogRocket/Datadog
    // if (context.priority === 'critical') {
    //   Sentry.captureException(error, { extra: errorEntry });
    // }

    return errorEntry.id;
  }

  getErrors() {
    return [...this.errors];
  }

  getErrorStats() {
    const total = this.errors.length;
    const byLevel = this.errors.reduce((acc, err) => {
      const level = err.context?.level || 'unknown';
      acc[level] = (acc[level] || 0) + 1;
      return acc;
    }, {});

    const byPriority = this.errors.reduce((acc, err) => {
      const priority = err.context?.priority || 'unknown';
      acc[priority] = (acc[priority] || 0) + 1;
      return acc;
    }, {});

    const byComponent = this.errors.reduce((acc, err) => {
      const comp = err.context?.component || 'unknown';
      acc[comp] = (acc[comp] || 0) + 1;
      return acc;
    }, {});

    return { total, byLevel, byPriority, byComponent };
  }

  subscribe(listener) {
    this.listeners.push(listener);
    return () => {
      this.listeners = this.listeners.filter((l) => l !== listener);
    };
  }

  notifyListeners() {
    this.listeners.forEach((listener) => listener(this.errors));
  }

  clearErrors() {
    this.errors = [];
    this.notifyListeners();
  }
}

class RetryService {
  async retryWithBackoff(fn, maxRetries = 3, baseDelayMs = 1000) {
    let lastError;

    for (let attempt = 1; attempt <= maxRetries; attempt++) {
      try {
        return await fn();
      } catch (error) {
        lastError = error;
        
        if (attempt === maxRetries) break;

        // Exponential backoff: 1s, 2s, 4s
        const delay = baseDelayMs * Math.pow(2, attempt - 1);
        console.log(`🔄 Retry ${attempt}/${maxRetries - 1} failed, waiting ${delay}ms...`);
        await new Promise((resolve) => setTimeout(resolve, delay));
      }
    }

    throw lastError;
  }
}

const retryService = new RetryService();

class CacheService {
  constructor() {
    this.cache = new Map();
  }

  set(key, value, ttlMs = 300000) { // Default 5 minutes
    this.cache.set(key, {
      value,
      timestamp: Date.now(),
      ttlMs,
    });
  }

  get(key) {
    const entry = this.cache.get(key);
    if (!entry) return null;

    const age = Date.now() - entry.timestamp;
    if (age > entry.ttlMs) {
      this.cache.delete(key);
      return null;
    }

    return entry.value;
  }

  invalidate(key) {
    this.cache.delete(key);
  }

  clear() {
    this.cache.clear();
  }

  getSize() {
    return this.cache.size;
  }
}

function NotificationProvider({ children }) {
  const [notifications, setNotifications] = useState([]);

  const addNotification = useCallback((message, type = 'error', title = '') => {
    const id = `${Date.now()}-${Math.random().toString(36).slice(2, 9)}`;
```

### Bảng So Sánh Trade-offs

| Pattern                          | Pros                                                                                | Cons                                                                  | When to use                                   |
| -------------------------------- | ----------------------------------------------------------------------------------- | --------------------------------------------------------------------- | --------------------------------------------- |
| **Class-based Error Boundary**   | - Full control<br>- No dependencies<br>- React native solution                      | - More boilerplate<br>- Class component syntax<br>- Harder to compose | ❌ Avoid (use library instead)                |
| **react-error-boundary Library** | - Function component API<br>- Reset keys<br>- Less boilerplate<br>- Well-maintained | - External dependency<br>- Learning curve                             | ✅ Production apps<br>✅ All new projects     |
| **Single Top-level Boundary**    | - Simple setup<br>- One place to manage                                             | - Bad UX<br>- No isolation                                            | ❌ Only for POCs                              |
| **Individual Boundaries**        | - Best isolation<br>- Best UX<br>- Independent failures                             | - More code<br>- More boundaries                                      | ✅ Production apps<br>✅ Independent features |
| **Strategic Multi-level**        | - Balance complexity/UX<br>- Clear hierarchy<br>- Flexible                          | - Requires planning<br>- Moderate complexity                          | ✅ Large applications<br>✅ Complex UIs       |

### Decision Tree

```jsx
function SmartErrorBoundary({ children, maxRetries = 3 }) {
  const [retryCount, setRetryCount] = useState(0);
  const [resetKey, setResetKey] = useState(0);

  const handleReset = () => {
    if (retryCount < maxRetries) {
      setRetryCount((c) => c + 1);
      setResetKey((k) => k + 1);
    }
  };

  const handleError = (error, errorInfo) => {
    console.error(`Error (attempt ${retryCount + 1}/${maxRetries}):`, error);
  };

  const FallbackWithRetries = ({ error, resetErrorBoundary }) => {
    const retriesLeft = maxRetries - retryCount;

    return (
      <div style={{ padding: 20, backgroundColor: '#fee', borderRadius: 8 }}>
        <h3 style={{ color: '#dc2626' }}>Error Occurred</h3>
        <p>{error.message}</p>

        {retriesLeft > 0 ? (
          <button
            onClick={() => {
              resetErrorBoundary();
              handleReset();
            }}
          >
            🔄 Try Again ({retriesLeft} attempts left)
          </button>
        ) : (
          <div>
            <p style={{ color: '#991b1b', fontWeight: 'bold' }}>
              ❌ Maximum retry attempts reached
            </p>
            <p style={{ fontSize: 14 }}>
              Please refresh the page or contact support.
            </p>
          </div>
        )}
      </div>
    );
  };

  return (
    <ErrorBoundary
      FallbackComponent={FallbackWithRetries}
      resetKeys={[resetKey]}
      onError={handleError}
    >
      {children}
    </ErrorBoundary>

}

// ✅ FIX 2: Automatic backoff delay
function BackoffErrorBoundary({ children }) {
  const [retryCount, setRetryCount] = useState(0);
  const [canRetry, setCanRetry] = useState(true);

  const handleReset = () => {
    const delay = Math.pow(2, retryCount) * 1000; // 1s, 2s, 4s, 8s...

    setCanRetry(false);

    setTimeout(() => {
      setRetryCount((c) => c + 1);
      setCanRetry(true);
    }, delay);
  };

  return (
    <ErrorBoundary
      fallbackRender={({ error, resetErrorBoundary }) => (
        <div>
          <p>Error: {error.message}</p>
          <button
            onClick={() => {
              resetErrorBoundary();
              handleReset();
            }}
            disabled={!canRetry}
          >
            {canRetry ? `Retry (attempt ${retryCount + 1})` : 'Waiting...'}
          </button>
        </div>

      {children}
```

```jsx
class FieldErrorBoundary extends Component {
  constructor(props) {
    super(props);

    this.state = {
      hasError: false,
      error: null,
    };
  }

  static getDerivedStateFromError(error) {
    return {
      hasError: true,
      error,

      boundary: this.props.boundaryName || 'FieldErrorBoundary',

    };
  }

  if (domain.split('.').some((part) => part.length < 2)) {
    return {
      valid: false,
      message: 'Email domain parts must be at least 2 characters.',
    };
  }

  if (value.length > 80) {
    return {
      valid: false,
      message: 'Email is too long.',
    };
  }

  return {
    valid: true,
    message: 'Email looks good.',
  };
}
```

```tsx
function wrapPromise<T>(promise: Promise<T>): Resource<T> {
  let status: 'pending' | 'success' | 'error' = 'pending';
  let result: T;
  let errorResult: Error;

  const suspender = promise.then(
    (data) => {
      status = 'success';
      result = data;
    },
    (error) => {
      status = 'error';
      errorResult = error instanceof Error ? error : new Error(String(error));
    },
  );

  return {
    read() {
      if (status === 'pending') {
        throw suspender;
      }

      if (status === 'error') {
        throw errorResult;
      }

      return result;
    },
  };
}

class ErrorRecoveryBoundary extends Component<
  ErrorRecoveryBoundaryProps,
  ErrorRecoveryBoundaryState
> {
  retryTimerId: number | null = null;

  constructor(props: ErrorRecoveryBoundaryProps) {
    super(props);

    this.state = {
      hasError: false,
      error: null,
      attempt: 0,
      isAutoRetrying: false,
      nextRetryInMs: 0,
    };
  }

  static getDerivedStateFromError(error: Error) {
    return {
      hasError: true,
      error,
    };
  }

  componentDidCatch(error: Error) {
    const cachedData = fallbackDataCache.get(this.props.widget.id);
    const attempt = this.state.attempt;
    const maxRetries = this.props.maxRetries ?? 3;

    const event: ErrorEvent = {
      id: `${this.props.widget.id}-${Date.now()}-${Math.random()
        .toString(16)
        .slice(2)}`,
      widgetId: this.props.widget.id,
      widgetTitle: this.props.widget.title,
      message: error.message,
      attempt,
      timestamp: new Date().toLocaleTimeString(),
      recoveredFromCache: Boolean(cachedData),
    };

    this.props.onErrorLogged(event);

    this.props.onToast({
      type: cachedData ? 'warning' : 'error',
      title: `${this.props.widget.title} failed`,
      message: cachedData
        ? 'Showing cached data while automatic retry runs.'
        : 'No cached data available. Automatic retry will run.',
    });

    if (attempt < maxRetries) {
      this.scheduleAutoRetry();
    }
  }

  componentWillUnmount() {
    if (this.retryTimerId !== null) {
      window.clearTimeout(this.retryTimerId);
    }
  }

  getRetryDelay() {
    const baseDelayMs = this.props.baseDelayMs ?? 800;

    /**
     * Exponential backoff:
     * attempt 0 → 800ms
     * attempt 1 → 1600ms
     * attempt 2 → 3200ms
     */
    return baseDelayMs * Math.pow(2, this.state.attempt);
  }

  scheduleAutoRetry = () => {
    const delayMs = this.getRetryDelay();

    this.setState({
      isAutoRetrying: true,
      nextRetryInMs: delayMs,
    });

    this.retryTimerId = window.setTimeout(() => {
      this.retryTimerId = null;
      this.retry();
    }, delayMs);
  };

  retry = () => {
    if (this.retryTimerId !== null) {
      window.clearTimeout(this.retryTimerId);
      this.retryTimerId = null;
    }

    this.setState((state) => ({
      hasError: false,
      error: null,
      attempt: state.attempt + 1,
      isAutoRetrying: false,
      nextRetryInMs: 0,
    }));
  };

  resetAttempts = () => {
    if (this.retryTimerId !== null) {
      window.clearTimeout(this.retryTimerId);
      this.retryTimerId = null;
    }

    this.setState({
      hasError: false,
      error: null,
      attempt: 0,
      isAutoRetrying: false,
      nextRetryInMs: 0,
    });
  };

  render() {
    if (this.state.hasError) {
      const cachedData = fallbackDataCache.get(this.props.widget.id);
      const maxRetries = this.props.maxRetries ?? 3;
      const canRetryMore = this.state.attempt < maxRetries;

      return (
        <WidgetErrorFallback
          widget={this.props.widget}
          error={this.state.error}
          cachedData={cachedData}
          attempt={this.state.attempt}
          maxRetries={maxRetries}
          canRetryMore={canRetryMore}
          isAutoRetrying={this.state.isAutoRetrying}
          nextRetryInMs={this.state.nextRetryInMs}
          onRetry={this.retry}
          onReset={this.resetAttempts}
        />
      );
    }

    return this.props.children({
      attempt: this.state.attempt,
    });
  }
}

export default function ErrorRecoverySystemHomework() {
  const [errorEvents, setErrorEvents] = useState<ErrorEvent[]>([]);
  const [toasts, setToasts] = useState<Toast[]>([]);
  const [globalResetKey, setGlobalResetKey] = useState(0);

    const id = `${Date.now()}-${Math.random().toString(16).slice(2)}`;
    const nextToast = { ...toast, id };

    setToasts((current) => [nextToast, ...current].slice(0, 5));

    window.setTimeout(() => {
      setToasts((current) => current.filter((item) => item.id !== id));
    }, 4200);

  const logErrorEvent = (event: ErrorEvent) => {
    console.group('🔴 Error analytics event');
    console.table(event);
    console.groupEnd();

    setErrorEvents((current) => [event, ...current].slice(0, 30));
  };

  const clearAnalytics = () => {
    setErrorEvents([]);
    addToast({
      type: 'info',
      title: 'Analytics cleared',
      message: 'Error analytics dashboard has been reset.',
    });
  };

  const clearCache = () => {
    fallbackDataCache.clear();
    resourceCache.clear();

    setGlobalResetKey((key) => key + 1);

    addToast({
      type: 'info',
      title: 'Cache cleared',
      message: 'Fallback cache and resource cache were cleared.',
    });
  };

  return (
    <main style={styles.app}>
      <ToastViewport
        toasts={toasts}
        onDismiss={(id) =>
          setToasts((current) => current.filter((toast) => toast.id !== id))

      />

      <section style={styles.container}>
        <header style={styles.header}>
          <p style={styles.eyebrow}>Ngày 50 · Error Boundaries</p>

          <h1 style={styles.title}>Error Recovery System</h1>

          <p style={styles.description}>
            Demo hệ thống phục hồi lỗi hoàn chỉnh: nhiều component có failure
            rate khác nhau, auto retry exponential backoff, cache fallback,
            analytics dashboard và toast notifications.

        </header>

        <section style={styles.actionBar}>
          <button
            type='button'
            onClick={() => setGlobalResetKey((key) => key + 1)}
            style={styles.primaryButton}
          >
            Reload all widgets
          </button>

          <button
            type='button'
            onClick={clearCache}
            style={styles.secondaryButton}
          >
            Clear cache
          </button>

          <button
            type='button'
            onClick={clearAnalytics}
            style={styles.secondaryButton}
          >
            Clear analytics
          </button>
        </section>

        <section style={styles.widgetGrid}>
          {WIDGETS.map((widget) => (
            <ErrorRecoveryBoundary
              key={`${widget.id}-${globalResetKey}`}
              widget={widget}
              maxRetries={3}
              baseDelayMs={800}
              onErrorLogged={logErrorEvent}
              onToast={addToast}

                <Suspense fallback={<WidgetSkeleton widget={widget} />}>
                  <RecoverableWidget
                    widget={widget}
                    attempt={attempt}

        </button>
      </div>

function ErrorAnalyticsDashboard({
  events,
  cachedCount,
}: {
  events: ErrorEvent[];
  cachedCount: number;
}) {
  const summary = useMemo(() => {
    const total = events.length;
    const recoveredFromCache = events.filter(
      (event) => event.recoveredFromCache,
    ).length;

    const byWidget = events.reduce<Record<string, number>>((acc, event) => {
      acc[event.widgetTitle] = (acc[event.widgetTitle] || 0) + 1;
      return acc;
    }, {});

    const mostFailingWidget =
      Object.entries(byWidget).sort((a, b) => b[1] - a[1])[0]?.[0] || 'None';

    return {
      total,
      recoveredFromCache,
      cachedCount,
      mostFailingWidget,
    };
  }, [events, cachedCount]);

  return (
    <section style={styles.analyticsCard}>
      <header style={styles.sectionHeader}>
        <div>
          <h2 style={styles.sectionTitle}>Error Analytics Dashboard</h2>
          <p style={styles.sectionDescription}>
            Theo dõi lỗi được Error Boundary catch, số lần recovery bằng cache
            và widget fail nhiều nhất.
          </p>
        </div>
      </header>

      <section style={styles.analyticsGrid}>
        <MetricCard
          label='Total Errors'
          value={summary.total}
        />
        <MetricCard
          label='Recovered From Cache'
          value={summary.recoveredFromCache}
        />
        <MetricCard
          label='Cached Widgets'
          value={summary.cachedCount}
        />
        <MetricCard
          label='Most Failing'
          value={summary.mostFailingWidget}
        />
      </section>

      <div style={styles.eventTableWrap}>
        <table style={styles.table}>
          <thead>
            <tr>
              <th style={styles.th}>Time</th>
              <th style={styles.th}>Widget</th>
              <th style={styles.th}>Attempt</th>
              <th style={styles.th}>Cache</th>
              <th style={styles.th}>Message</th>
            </tr>
          </thead>

          <tbody>
            {events.length === 0 ? (
              <tr>
                <td
                  colSpan={5}
                  style={styles.emptyCell}
                >
                  No error events yet. Reload widgets a few times to generate
                  failures.
                </td>
              </tr>
            ) : (
              events.map((event) => (
                <tr key={event.id}>
                  <td style={styles.td}>{event.timestamp}</td>
                  <td style={styles.td}>{event.widgetTitle}</td>
                  <td style={styles.td}>{event.attempt}</td>
                  <td style={styles.td}>
                    {event.recoveredFromCache ? '✅ Yes' : '❌ No'}
                  </td>
                  <td style={styles.td}>{event.message}</td>
                </tr>
              ))
            )}
          </tbody>
        </table>
      </div>
    </section>
  );
}
```

### Câu Hỏi Phỏng Vấn

**Junior**

**1. Error Boundary là gì?**  
Error Boundary là một React component dùng để bắt lỗi JavaScript xảy ra bất cứ đâu trong tree con của nó, ghi log lỗi đó và hiển thị UI dự phòng (fallback UI) thay vì để toàn bộ app crash thành white screen. Nó hoạt động như một `try-catch` cho React components.

**2. Error Boundary catch những errors nào?**  
Chỉ catch lỗi xảy trong quá trình render, trong lifecycle methods và trong constructor của các component con. Nó **không** catch lỗi trong:  
- Event handlers (`onClick`, `onSubmit`, v.v.)  
- Async code (`setTimeout`, `Promise`, `fetch`)  
- Server-side rendering (SSR) errors  
- Errors trong chính Error Boundary đó.

**3. Làm sao implement Error Boundary?**  
Cách 1 (native): Tạo class component implement `static getDerivedStateFromError(error)` để update state hiển thị fallback, và `componentDidCatch(error, errorInfo)` để log lỗi.  
Cách 2 (production): Dùng thư viện `react-error-boundary` để không phải viết class component thủ công. Ví dụ:  
```jsx
<<ErrorBoundary FallbackComponent={MyFallback}>
  <MyComponent />
</ErrorBoundary>
```

---

**Mid**

**1. Tại sao Error Boundary không catch event handler errors?**  
Vì event handlers chạy **bên ngoài React render cycle**. Khi user click một button, hàm `onClick` được gọi bởi browser event system, không phải bởi React reconciliation. Error Boundary chỉ can thiệp được vào các lỗi xảy ra trong quá trình React render tree (render phase). Để bắt lỗi trong event handler, phải dùng `try-catch` thủ công hoặc dùng `useErrorBoundary().showBoundary(error)` từ thư viện `react-error-boundary`.

**2. So sánh `getDerivedStateFromError` vs `componentDidCatch`**  

| | `static getDerivedStateFromError` | `componentDidCatch` |
|---|---|---|
| **Mục đích** | Update state để render fallback UI | Side effects: log lỗi, gửi monitoring |
| **Thời điểm chạy** | Render phase (trước khi commit) | Commit phase (sau khi render fallback) |
| **Cú pháp** | Static method | Instance method |
| **Truy cập context** | Không có `this`, chỉ nhận `error` | Có `this`, nhận cả `error` và `errorInfo` |
| **Dùng cho** | Quyết định UI hiển thị gì | Gửi lỗi lên Sentry, console, analytics |

Quy tắc: `getDerivedStateFromError` để **hiển thị**, `componentDidCatch` để **xử lý**.

**3. Strategic placement của Error Boundaries như thế nào?**  
Không đặt một boundary duy nhất ở root vì lỗi một widget sẽ crash cả app. Cũng không lạm dụng boundary ở mọi component vì tốn chi phí maintain. Chiến lược đúng là:  
- **Top-level (App)**: Bắt lỗi catastrophic, hiển thị "Something went wrong" toàn trang.  
- **Feature-level (route/section)**: Bắt lỗi ở từng trang lớn (Dashboard, Settings).  
- **Widget-level**: Bọc từng widget độc lập (chart, table, feed) để một widget lỗi không làm sập cả trang.  
- **Component-level (hiếm)**: Chỉ bọc các component có logic phức tạp, dễ fail (ví dụ: rich text editor, data visualization).  
Nguyên tắc: **isolate failures, preserve UX**.

---

**Senior**

**1. Thiết kế error handling architecture cho large-scale app**  
Tôi thiết kế theo **hierarchical boundary strategy** kết hợp với **graceful degradation**:  

- **Layer 1 — App Boundary**: Bắt lỗi không lường trước, hiển thị full-screen fallback với nút reload.  
- **Layer 2 — Feature Boundaries**: Mỗi major feature (Dashboard, Checkout, Admin) có boundary riêng. Nếu Checkout fail, user vẫn xem được Dashboard.  
- **Layer 3 — Widget Boundaries**: Mỗi widget độc lập (analytics card, notification bell) tự có boundary. Lỗi ở widget A không ảnh hưởng widget B.  
- **Layer 4 — Async Boundaries**: Kết hợp `ErrorBoundary` bọc ngoài `Suspense` để xử lý cả render error và data-fetching error.  

Kèm theo đó là **recovery mechanisms**:  
- `resetKeys` để auto-recover khi route/props thay đổi.  
- Retry với exponential backoff (1s → 2s → 4s) và giới hạn tối đa 3 lần.  
- Cache fallback: hiển thị stale data khi fresh data fail.  
- Kill switch: nếu một feature có error rate > 5%, tự động disable qua feature flag.  

**2. Error recovery strategies trong production**  

- **Immediate retry**: Dùng `resetErrorBoundary()` + `resetKeys` cho transient errors (network glitch).  
- **Exponential backoff**: Không retry liên tục mà delay tăng dần để tránh overload server và tạo UX mượt.  
- **Fallback to cache**: Nếu API fail, hiển thị data từ `localStorage`, `IndexedDB`, hoặc React Query cache thay vì white screen.  
- **Graceful degradation**: Với non-critical features (recommendations, help widget), silent failure — ẩn section thay vì hiển thị lỗi.  
- **User-initiated recovery**: Nút "Try Again" rõ ràng, nhưng sau 3 lần fail thì chuyển sang "Contact Support" hoặc "Reload Page".  
- **State preservation**: Khi reset boundary, dùng key prop để force re-mount component, đảm bảo state cũ (gây lỗi) bị xóa hoàn toàn.  

**3. Monitoring và alerting cho errors trong React app**  

- **Structured logging**: Mọi `componentDidCatch` / `onError` phải log object chứa: `error.message`, `error.stack`, `componentStack`, `userId`, `sessionId`, `timestamp`, `route`, `featureFlagState`.  
- **Integration với Sentry/Datadog/LogRocket**: Gửi error kèm breadcrumbs (user actions dẫn đến lỗi) và tags (release version, environment).  
- **Error analytics dashboard**: Theo dõi error rate theo component, theo thời gian, và tỷ lệ recovery từ cache. Phát hiện regression sau release.  
- **Alerting**:  
  - PagerDuty/Slack alert nếu error rate vượt ngưỡng (ví dụ: > 1% critical errors trong 5 phút).  
  - Alert riêng cho "new error" — lỗi chưa từng xuất hiện trong production.  
- **User feedback loop**: Khi hiển thị fallback UI, cho phép user gửi feedback/description ngay tại chỗ, attach vào error report.  
- **Source maps**: Upload source maps lên monitoring service để stack trace readable, dễ debug nhanh.

### War Stories

**Story 1: The Dashboard That Never Died**
"Ở startup cũ, dashboard có 20+ widgets. Ban đầu dùng single error boundary - một widget fail thì toàn bộ dashboard crash. Users phàn nàn nhiều.

Refactor sang individual boundaries cho mỗi widget. Error rate giảm 80% vì:

- Users vẫn thấy working widgets
- Có retry button rõ ràng
- Error messages specific hơn

Lesson: Isolation > Simplicity trong production apps."

**Story 2: The Infinite Retry Loop**
"Launch feature mới, có bug khiến component always fail. Users spam retry button → Server overload → Cascade failure.

Fix bằng cách:

- Limit retry attempts (max 3)
- Exponential backoff (1s, 2s, 4s)
- Rate limiting ở client-side

Lesson: Always limit retry mechanisms."
