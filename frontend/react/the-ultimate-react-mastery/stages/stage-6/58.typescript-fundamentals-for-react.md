### 1.1 Vấn Đề Thực Tế

Hãy tưởng tượng bạn đang làm việc trong một team 5 người. Sau 3 tháng, codebase có 200+ components. Một buổi sáng thứ Hai, bạn nhận được Slack từ đồng nghiệp:

Đây chính là giá trị cốt lõi của TypeScript: **bắt lỗi tại thời điểm viết code (compile time), không phải lúc chạy (runtime).**

### 1.2 TypeScript là gì?

TypeScript (TS) là **superset của JavaScript** — nghĩa là mọi code JS hợp lệ đều là TS hợp lệ. TS thêm vào JS một hệ thống **static type** (kiểu tĩnh):

```
JavaScript + Type System = TypeScript
                ↓
           TypeScript Compiler (tsc)
                ↓
           JavaScript thuần (chạy trên browser)
```

**Browser không hiểu TypeScript.** TS phải được compile sang JS. Với React (Vite/CRA), quá trình này xảy ra tự động trong background.

---

### 1.3 Mental Model: TypeScript như "Hợp Đồng"

Hãy nghĩ TypeScript như một **hợp đồng** giữa các phần của codebase:

```
Component A (người dùng) ←→ Interface/Type ←→ Component B (người cung cấp)
         "Tôi cam kết                              "Tôi cam kết
          truyền đúng                               nhận đúng
          kiểu dữ liệu"                             kiểu này"
```

Khi hợp đồng bị vi phạm → TypeScript báo lỗi ngay lập tức.

```
Analogy thực tế:
TypeScript giống như "form đặt hàng" trong nhà hàng.
- Mục "số lượng": bắt buộc phải là số, không thể ghi "hai"
- Mục "ghi chú": tùy chọn, có thể để trống
- Nếu điền sai → form không submit được (compile error)
- Nếu không có form → nhân viên phải đoán ý khách (runtime bug)
```

### 1.5 `interface` vs `type` — Khi nào dùng cái nào?

Đây là câu hỏi mà **100% senior** sẽ hỏi bạn trong phỏng vấn. Hãy hiểu rõ:

```typescript
// Cách 1: interface
interface UserProps {
  name: string;
  age: number;
}

// Cách 2: type alias
type UserProps = {
  name: string;
  age: number;
};
```

**Khác biệt chính:**

| Tính năng             | `interface`                 | `type`             |
| --------------------- | --------------------------- | ------------------ |
| Khai báo lại để merge | ✅ Có (Declaration merging) | ❌ Không           |
| Extend / Kế thừa      | `extends`                   | `&` (intersection) |
| Union types           | ❌ Không                    | ✅ Có              |
| Computed properties   | ❌ Không                    | ✅ Có              |
| Tuple types           | ❌ Không                    | ✅ Có              |

**Quy tắc thực tế trong React (theo React team và phần lớn senior devs):**

```typescript
// ✅ Dùng interface cho Props và State của component
interface ButtonProps {
  label: string;
  onClick: () => void;
}

// ✅ Dùng type cho union, intersection, hoặc type phức tạp
type Status = 'loading' | 'success' | 'error';
type ID = string | number;
```

> **💡 Senior Insight:** Trong thực tế, cả hai đều hoạt động tốt cho props. Quan trọng là **nhất quán trong codebase**. Nhiều team dùng `interface` cho objects, `type` cho primitives/unions. Nếu không có convention, dùng `interface` cho props — đây là convention phổ biến hơn.

### 1.6 Hiểu Lầm Phổ Biến

**❌ Hiểu lầm 1: "TypeScript làm code chạy an toàn hơn"**

```typescript
// TypeScript chỉ kiểm tra ở COMPILE TIME
// Nếu dữ liệu đến từ API (runtime), TS không bảo vệ bạn:
const data = await fetch('/api/user').then((r) => r.json());
// data có type là `any` → TS không biết gì về structure của nó
// → Vẫn cần validate runtime data!
```

**❌ Hiểu lầm 2: "Dùng `any` tạm thời cho nhanh"**

```typescript
// ❌ `any` vô hiệu hóa toàn bộ type checking
function processUser(user: any) {
  return user.name.toUpperCase(); // TS không báo lỗi nếu user là null
}

// ✅ Dùng `unknown` nếu thực sự không biết kiểu
function processUser(user: unknown) {
  if (typeof user === 'object' && user !== null && 'name' in user) {
    return (user as { name: string }).name.toUpperCase();
  }
}
```

**❌ Hiểu lầm 3: "Type là documentation, không cần đọc"**

Types trong TypeScript **là code**, không chỉ là comment. Chúng được compiler enforce.

### Demo 1: Typing Props Cơ Bản ⭐

Bắt đầu với component đơn giản nhất:

```tsx
// ❌ Cách SAI: Không có type
function Greeting({ name, age }) {
  // TypeScript sẽ báo: Parameter 'name' implicitly has an 'any' type
  return <h1>Xin chào {name}, {age} tuổi!</h1>;
}

// ✅ Cách ĐÚNG: Định nghĩa interface cho props
interface GreetingProps {
  name: string;
  age: number;
}

function Greeting({ name, age }: GreetingProps) {
  return <h1>Xin chào {name}, {age} tuổi!</h1>;
}

// Khi dùng:
<Greeting name="Tuân" age={29} />      // ✅ OK
<Greeting name="Tuân" age="29" />      // ❌ Lỗi: age phải là number
<Greeting name="Tuân" />               // ❌ Lỗi: age là bắt buộc
```

**Optional Props — Dùng `?`:**

```tsx
interface UserCardProps {
  name: string;          // Bắt buộc
  email: string;         // Bắt buộc
  avatar?: string;       // Tùy chọn (string | undefined)

function UserCard({ name, email, avatar, bio }: UserCardProps) {
  return (
    <div>
      {/* avatar có thể undefined → cần fallback */}
      <img src={avatar ?? '/default-avatar.png'} alt={name} />
      <h2>{name}</h2>
      <p>{email}</p>
      {/* Render có điều kiện cho optional prop */}
      {bio && <p className="bio">{bio}</p>}
    </div>
  );

// Sử dụng:
<UserCard name="Tuân" email="tuan@example.com" />                    // ✅ OK
<UserCard name="Tuân" email="tuan@example.com" bio="Frontend Dev" /> // ✅ OK
```

### Demo 2: Typing Events ⭐⭐

Events là nơi **nhiều dev React mới học TS bị vấp nhất**. Hãy master phần này:

```tsx
// ❌ Cách SAI: Dùng any hoặc không type event
function SearchInput() {
  const handleChange = (event: any) => {
    // Mất toàn bộ type safety!
    console.log(event.target.value);
  };

  return <input onChange={handleChange} />;
}

// ✅ Cách ĐÚNG: Type đúng event
function SearchInput() {
  // React.ChangeEvent<HTMLInputElement> = event từ <input>
  const handleChange = (event: React.ChangeEvent<HTMLInputElement>) => {
    const value: string = event.target.value; // TypeScript biết đây là string
    console.log(value);
  };

  return <input onChange={handleChange} />;
}
```

**Bảng tham chiếu nhanh — DOM Events phổ biến:**

```tsx
// onClick — dùng cho button, div, etc.
const handleClick = (event: React.MouseEvent<HTMLButtonElement>) => {

// onChange — dùng cho input, select, textarea
const handleInputChange = (event: React.ChangeEvent<HTMLInputElement>) => {
  const value = event.target.value; // string

const handleSelectChange = (event: React.ChangeEvent<HTMLSelectElement>) => {

};

const handleCheckboxChange = (event: React.ChangeEvent<HTMLInputElement>) => {

const handleSubmit = (event: React.FormEvent<HTMLFormElement>) => {
  event.preventDefault();
  // Xử lý form...

// onKeyDown/onKeyUp/onKeyPress
const handleKeyDown = (event: React.KeyboardEvent<HTMLInputElement>) => {
  if (event.key === 'Enter') {
    // ...
  }
};

// onFocus / onBlur
const handleFocus = (event: React.FocusEvent<HTMLInputElement>) => {
  // ...
```

**Ví dụ thực tế — Contact Form đầy đủ:**

```tsx
interface FormData {
  name: string;
  email: string;
  message: string;
}

function ContactForm() {
  const [formData, setFormData] = React.useState<FormData>({
    name: '',
    email: '',
    message: '',
  });
  const [status, setStatus] = React.useState<
    'idle' | 'loading' | 'success' | 'error'
  >('idle');

  // ✅ Handler dùng chung cho input và textarea
  const handleChange = (
    event: React.ChangeEvent<HTMLInputElement | HTMLTextAreaElement>,
  ) => {
    const { name, value } = event.target;
    setFormData((prev) => ({
      ...prev,
      [name]: value, // Computed property name
    }));
  };

  const handleSubmit = async (event: React.FormEvent<HTMLFormElement>) => {
    event.preventDefault();
    setStatus('loading');

    try {
      // Giả lập API call
      await new Promise((resolve) => setTimeout(resolve, 1000));
      setStatus('success');
    } catch {
      setStatus('error');
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        name='name'
        value={formData.name}
        onChange={handleChange}
        placeholder='Họ tên'
        required
      />
      <input
        name='email'
        type='email'
        value={formData.email}
        onChange={handleChange}
        placeholder='Email'
        required
      />
      <textarea
        name='message'
        value={formData.message}
        onChange={handleChange}
        placeholder='Nội dung'
        required
      />
      <button
        type='submit'
        disabled={status === 'loading'}
      >
        {status === 'loading' ? 'Đang gửi...' : 'Gửi'}
      </button>
      {status === 'success' && <p>✅ Gửi thành công!</p>}
      {status === 'error' && <p>❌ Có lỗi xảy ra, thử lại!</p>}
    </form>
  );
}
```

### Demo 3: Typing Children ⭐⭐⭐

`children` prop là một trong những phần **phức tạp nhất khi mới học TS với React**. Có nhiều cách type, mỗi cách dùng cho trường hợp khác nhau:

```tsx
// ❌ Cách SAI phổ biến nhất: Dùng React.FC
// React.FC tự động thêm children?: ReactNode nhưng gây nhiều vấn đề khác
// → Đây là anti-pattern từ React 18+, KHÔNG nên dùng

const Card: React.FC<{ title: string }> = ({ title, children }) => {
  // TS không báo lỗi vì React.FC ngầm thêm children
  // Nhưng bạn không kiểm soát được kiểu của children
  return (
    <div>
      <h2>{title}</h2>
      {children}
    </div>
  );
};
```

**✅ Cách ĐÚNG — 4 patterns cho children:**

```tsx
import { ReactNode, ReactElement, PropsWithChildren } from 'react';

// Pattern 1: ReactNode — Linh hoạt nhất (string, number, JSX, null, array...)
interface CardProps {
  title: string;
  children: ReactNode; // Chấp nhận hầu hết mọi thứ
}

function Card({ title, children }: CardProps) {
  return (
    <div className="card">
      <h2>{title}</h2>
      <div className="content">{children}</div>
    </div>
  );
}

// Dùng được với:
<Card title="Test">
  <p>Text</p>           {/* ✅ JSX element */}
</Card>
<Card title="Test">
  Chỉ là string         {/* ✅ String */}
</Card>
<Card title="Test">
  {null}                {/* ✅ null (không render gì) */}
</Card>

// ---

// Pattern 2: ReactElement — Chỉ chấp nhận JSX element (không cho phép string thuần)
interface StrictWrapperProps {
  children: ReactElement; // Phải là JSX
}

function StrictWrapper({ children }: StrictWrapperProps) {
  return <div className="wrapper">{children}</div>;
}

// Dùng được với:
<StrictWrapper><Button>Click</Button></StrictWrapper>  // ✅
<StrictWrapper>Just a string</StrictWrapper>           // ❌ Lỗi!

// ---

// Pattern 3: PropsWithChildren — Helper type của React (ngắn gọn hơn)
// PropsWithChildren<P> = P & { children?: ReactNode }
interface LayoutProps {
  sidebar: ReactNode;
}

function Layout({ sidebar, children }: PropsWithChildren<LayoutProps>) {
  return (
    <div className="layout">
      <aside>{sidebar}</aside>
      <main>{children}</main>
    </div>
  );
}

// ---

// Pattern 4: Render Props (function as children)
interface DataProviderProps {
  userId: string;
  children: (data: User | null, loading: boolean) => ReactNode;
}

interface User {
  id: string;
  name: string;
  email: string;
}

function DataProvider({ userId, children }: DataProviderProps) {
  const [user, setUser] = React.useState<User | null>(null);
  const [loading, setLoading] = React.useState(true);

  React.useEffect(() => {
    // Giả lập fetch
    setTimeout(() => {
      setUser({ id: userId, name: 'Tuân', email: 'tuan@example.com' });
      setLoading(false);
    }, 1000);
  }, [userId]);

  return <>{children(user, loading)}</>;
}

// Sử dụng:
<DataProvider userId="123">
  {(user, loading) => (
    loading ? <Spinner /> : <UserProfile user={user} />
  )}
</DataProvider>
```

## 📊 PHẦN 4: SO SÁNH PATTERNS (30 phút)

### Bảng So Sánh — Cách Type Props

| Cách         | Ví dụ                                    | Pros                                          | Cons                                      | Khi dùng                          |
| ------------ | ---------------------------------------- | --------------------------------------------- | ----------------------------------------- | --------------------------------- |
| `interface`  | `interface Props { name: string }`       | Extendable, clean syntax, declaration merging | Không support union                       | Props của component (khuyến nghị) |
| `type` alias | `type Props = { name: string }`          | Support union/intersection, flexible          | Không có declaration merging              | Union types, complex types        |
| Inline type  | `function F({ name }: { name: string })` | Nhanh, không cần đặt tên                      | Không reusable, khó đọc                   | Props 1-2 fields, internal        |
| `React.FC`   | `const F: React.FC<Props>`               | Auto-include displayName                      | Children type implicit, hides return type | **TRÁNH dùng** từ React 18+       |

| --------- | ------------ | -------------------------------------------------------- |

## 💡 SENIOR INSIGHTS

### Cân Nhắc Production

**1. Strict Mode là bắt buộc trong team:**

```json
// tsconfig.json
{
  "compilerOptions": {
    "strict": true, // Bật tất cả strict checks
    "noImplicitAny": true, // Không cho phép implicit any
    "strictNullChecks": true, // Phân biệt null và undefined
    "noUncheckedIndexedAccess": true // Array access có thể undefined
  }
}
```

**2. Tách types ra file riêng khi cần chia sẻ:**

```
src/
  types/
    api.ts         ← Types từ API responses
    components.ts  ← Shared component types
    store.ts       ← State management types
  components/
    Button/
      index.tsx
      Button.types.ts  ← Component-specific types
```

**3. Đừng over-engineer types:**

```tsx
// ❌ Over-engineered
type ButtonVariant = 'primary' | 'secondary';
type ButtonSize = 'sm' | 'md' | 'lg';
type ButtonState = 'default' | 'hover' | 'active' | 'disabled';
type ButtonConfig = {
  variant: ButtonVariant;
  size: ButtonSize;
  state: ButtonState;
};

// ✅ Vừa đủ — Types phục vụ code, không phải ngược lại
interface ButtonProps {
  variant?: 'primary' | 'secondary';
  size?: 'sm' | 'md' | 'lg';
  disabled?: boolean;
}
```

### Câu Hỏi Phỏng Vấn

**Junior:**

> "Sự khác biệt giữa `interface` và `type` trong TypeScript là gì?"

_Expected answer:_ Interface hỗ trợ declaration merging và thường được dùng cho object shapes. Type alias linh hoạt hơn, hỗ trợ unions, intersections, tuples. Trong thực tế, cả hai đều dùng được cho props — quan trọng là nhất quán.

**Mid-level:**

> "Tại sao `React.FC` không được khuyến nghị trong React 18+? Bạn thay thế bằng gì?"

_Expected answer:_ `React.FC` tự động thêm `children?: ReactNode` vào mọi component (không cần thiết), ẩn return type (có thể return `null` mà TS không bắt), và gây khó debug. Thay thế: function declaration bình thường với explicit return type `JSX.Element` hoặc `ReactNode`.

**Senior:**

> "Hãy thiết kế type system cho một polymorphic component (component render element HTML khác nhau tùy theo prop, như `as` prop trong Chakra UI)."

_Expected answer:_ Đây là bài toán về generic components với discriminated unions và `React.ElementType`. Cần xử lý: props của element target phải merge với component props, `ref` forwarding, và TypeScript inference. (Đây chính là content của Ngày 59!)

### War Stories

**Câu chuyện thật từ production:**

_"Chúng tôi có component `<DataTable>` đã chạy production 6 tháng. Một ngày, một dev thêm column mới bằng cách truyền `columns={newColumns}` — nhưng `newColumns` có field `sortKey` là `number` trong khi codebase cũ expect `string`. Bug xuất hiện ở runtime khi user click sort._

_Sau sự cố, team quyết định migrate toàn bộ `<DataTable>` và các components liên quan sang TypeScript strict mode. Quá trình migration tìm ra thêm 12 type mismatch bugs tiềm ẩn khác — chưa có bug nào được report nhưng tất cả đều là time bomb._

_Lesson learned: TypeScript không chỉ là 'nice to have' cho developer experience. Nó là safety net ngăn những bug như thế này reach production."_
