### 1.2 Mental Model — Generics như "Khuôn Đúc"

```
Analogy: Generic như khuôn đúc bánh.

Khuôn            → Generic Type Parameter <T>
Nguyên liệu      → Type cụ thể (string, number, User...)
Bánh thành phẩm  → Component/function đã được type

Cùng 1 khuôn (generic) → nhiều loại bánh (string select, number select...)
Không cần viết lại khuôn cho từng loại nguyên liệu!
```

```typescript
// Generic function — T là "placeholder" cho type thực tế
function identity<T>(value: T): T {
  return value;
}

identity<string>('hello'); // T = string → returns string
identity<number>(42); // T = number → returns number
identity('hello'); // TypeScript TỰ SUY RA T = string (type inference!)

// Generic với constraint — T phải có property 'id'
function getById<T extends { id: string }>(
  items: T[],
  id: string,
): T | undefined {
  return items.find((item) => item.id === id);
}
```

---

### 1.3 Mental Model — Utility Types như "Bộ Công Cụ Biến Đổi"

```
Utility Types = Công cụ biến đổi type có sẵn → type mới

Type gốc: User { id, name, email, password, role, createdAt, bio? }
                               ↓
Partial<User>              → Tất cả fields thành optional   (dùng cho update form)
Required<User>             → Tất cả fields thành required
Pick<User, 'id' | 'name'>  → Chỉ giữ lại id và name
Omit<User, 'password'>     → Bỏ password, giữ lại phần còn lại
Readonly<User>             → Không thể thay đổi sau khi tạo
Record<string, User>       → Object với key là string, value là User
```

---

### 1.4 Hiểu Lầm Phổ Biến

**❌ Hiểu lầm 1: "Generic quá phức tạp, không cần thiết"**

Sai. Bất cứ khi nào bạn copy-paste một component/function chỉ để đổi type → đó là dấu hiệu cần generic.

**❌ Hiểu lầm 2: "Utility types chỉ là syntactic sugar"**

Không đúng. `Omit<User, 'password'>` tạo ra một **type mới** được TypeScript track riêng biệt. Khi `User` thay đổi, `Omit<User, 'password'>` tự động cập nhật theo — không cần sửa tay.

**❌ Hiểu lầm 3: "Cứ dùng `Partial<T>` cho mọi thứ"**

`Partial<T>` làm tất cả fields optional → mất type safety. Chỉ dùng khi thực sự cần (update forms, PATCH operations).

**Generic Select Component — Ví dụ thực tế nhất:**

```tsx
interface Option<T> {
  value: T;
  label: string;
  disabled?: boolean;
}

interface SelectProps<T> {
  label: string;
  options: Option<T>[];
  value: T | null;
  onChange: (value: T) => void;
  placeholder?: string;
  disabled?: boolean;
  error?: string;
  required?: boolean;
  id: string;
}

// ⚠️ Lưu ý cú pháp: <T,> có dấu phẩy trong .tsx
// để TypeScript không nhầm với JSX opening tag
function Select<T,>({
  label,
  options,
  value,
  onChange,
  placeholder = 'Chọn một giá trị...',
  disabled = false,
  error,
  required = false,
  id,
}: SelectProps<T>) {
  const selectedIndex = options.findIndex(opt => opt.value === value);

  return (
    <div className={`select-wrapper ${error ? 'has-error' : ''}`}>
      <label htmlFor={id}>
        {label}
        {required && (
          <span aria-hidden="true" className="required-mark"> *</span>
        )}
      </label>

      <select
        id={id}
        disabled={disabled}
        aria-required={required}
        aria-invalid={!!error}
        aria-describedby={error ? `${id}-error` : undefined}
        // HTML <select> chỉ làm việc với string — ta map index → value
        value={selectedIndex >= 0 ? selectedIndex.toString() : ''}
        onChange={(e) => {
          const index = parseInt(e.target.value, 10);
          if (!isNaN(index) && options[index] && !options[index].disabled) {
            onChange(options[index].value);
          }
        }}
      >
        <option value="" disabled>{placeholder}</option>
        {options.map((option, index) => (
          <option
            key={index}
            value={index.toString()}
            disabled={option.disabled}
          >
            {option.label}
          </option>
        ))}
      </select>

      {error && (
        <span id={`${id}-error`} className="error-message" role="alert">
          {error}
        </span>
      )}
    </div>
  );
}

// --- Sử dụng với các value types khác nhau ---
```

### Demo 2: Utility Types Thực Tế ⭐⭐

**6 Utility types bạn sẽ dùng hàng ngày:**

```tsx
// --- Base type ---
interface User {
  id: string;
  name: string;
  email: string;
  password: string;
  role: 'admin' | 'user' | 'seller';
  createdAt: Date;
  bio?: string;
}

// ─────────────────────────────────────────────
// 1. Partial<T> — Mọi fields thành optional
//    Use case: Update/PATCH payload
// ─────────────────────────────────────────────
async function updateUser(
  id: string,
  updates: Partial<Omit<User, 'id' | 'createdAt'>>, // Kết hợp 2 utility types!
): Promise<User> {
  const response = await fetch(`/api/users/${id}`, {
    method: 'PATCH',
    body: JSON.stringify(updates),
    headers: { 'Content-Type': 'application/json' },
  });
  return response.json();
}

// Chỉ update những gì cần — không cần truyền đủ fields
await updateUser('123', { name: 'Tuân Mới', bio: 'Frontend Dev' }); // ✅

// ─────────────────────────────────────────────
// 2. Required<T> — Mọi fields thành bắt buộc
//    Use case: Sau validation, đảm bảo tất cả có giá trị
// ─────────────────────────────────────────────
type ValidatedUser = Required<User>;
// bio không còn optional → bắt buộc phải có

// ─────────────────────────────────────────────
// 3. Pick<T, K> — Chỉ lấy một số fields
//    Use case: Public profile (ẩn password, createdAt)
// ─────────────────────────────────────────────
type UserPublicProfile = Pick<User, 'id' | 'name' | 'bio'>;
// { id: string; name: string; bio?: string }

function PublicProfileCard({ user }: { user: UserPublicProfile }) {
  // TypeScript biết user.password KHÔNG TỒN TẠI ở đây
  // → Không thể vô tình expose password!
  return (
    <div>
      <h2>{user.name}</h2>
      {user.bio && <p>{user.bio}</p>}
    </div>
  );
}

// ─────────────────────────────────────────────
// 4. Omit<T, K> — Bỏ một số fields
//    Use case: Create payload (server tự generate id, createdAt)
// ─────────────────────────────────────────────
type CreateUserPayload = Omit<User, 'id' | 'createdAt'>;
// { name, email, password, role, bio? }

async function createUser(payload: CreateUserPayload): Promise<User> {
  const response = await fetch('/api/users', {
    method: 'POST',
    body: JSON.stringify(payload),
    headers: { 'Content-Type': 'application/json' },
  });
  return response.json();
}

// ─────────────────────────────────────────────
// 5. Record<K, V> — Object với key type K, value type V
//    Use case: Lookup table, cache, dictionary
// ─────────────────────────────────────────────
type UserRole = 'admin' | 'user' | 'seller';
type PermissionMap = Record<UserRole, string[]>;

// TypeScript đảm bảo tất cả keys đều có — không thể bỏ sót
const permissions: PermissionMap = {
  admin: ['read', 'write', 'delete', 'manage-users'],
  user: ['read'],
  seller: ['read', 'write', 'manage-products'],
};

// ─────────────────────────────────────────────
// 6. Readonly<T> — Không thể thay đổi sau khi tạo
//    Use case: Config objects, constants
// ─────────────────────────────────────────────
type AppConfig = Readonly<{
  apiUrl: string;
  timeout: number;
  maxRetries: number;
}>;

const config: AppConfig = {
  apiUrl: 'https://api.example.com',
  timeout: 5000,
  maxRetries: 3,
};

config.apiUrl = 'https://other.com'; // ❌ Lỗi: Cannot assign to 'apiUrl' (readonly)
```

### Demo 3: Typing Hooks Nâng Cao ⭐⭐

**useRef — 3 trường hợp khác nhau:**

```tsx
import { useRef, useEffect } from 'react';

// ─────────────────────────────────────────────
// Trường hợp 1: Ref cho DOM element
// Initial value: null (element chưa mount)
// ─────────────────────────────────────────────
function AutoFocusInput() {
  const inputRef = useRef<HTMLInputElement>(null);

  useEffect(() => {
    inputRef.current?.focus(); // Optional chaining — safe khi chưa mount
  }, []);

  return (
    <input
      ref={inputRef}
      placeholder='Auto-focused!'
    />
  );
}

// ─────────────────────────────────────────────
// Trường hợp 2: Ref lưu mutable value (không trigger re-render)
// ─────────────────────────────────────────────
function Timer() {
  const [seconds, setSeconds] = React.useState(0);
  const intervalRef = useRef<ReturnType<typeof setInterval> | null>(null);

  const start = () => {
    if (intervalRef.current) clearInterval(intervalRef.current);
    intervalRef.current = setInterval(() => {
      setSeconds((s) => s + 1);
    }, 1000);
  };

  const stop = () => {
    if (intervalRef.current) {
      clearInterval(intervalRef.current);
      intervalRef.current = null;
    }
  };

  useEffect(() => {
    return () => {
      if (intervalRef.current) clearInterval(intervalRef.current);
    };
  }, []);

  return (
    <div>
      <p>Thời gian: {seconds}s</p>
      <button onClick={start}>Bắt đầu</button>
      <button onClick={stop}>Dừng</button>
    </div>
  );
}

// ─────────────────────────────────────────────
// Trường hợp 3: Generic custom hook — usePrevious<T>
// ─────────────────────────────────────────────
function usePrevious<T>(value: T): T | undefined {
  const previousRef = useRef<T | undefined>(undefined);

  useEffect(() => {
    previousRef.current = value;
  });

  return previousRef.current;
}

// Sử dụng:
function Counter() {
  const [count, setCount] = React.useState(0);
  const previousCount = usePrevious(count); // TypeScript suy ra: number | undefined

  return (
    <div>
      <p>Hiện tại: {count}</p>
      <p>Trước đó: {previousCount ?? 'N/A'}</p>
      <button onClick={() => setCount((c) => c + 1)}>+1</button>
    </div>
  );
}
```

**useFetch — Custom hook generic cho mọi API response:**

```tsx
// Discriminated Union cho fetch state (xem thêm ở Demo 5)
type FetchState<T> =
  | { status: 'idle'; data: null; error: null }
  | { status: 'loading'; data: null; error: null }
  | { status: 'success'; data: T; error: null }
  | { status: 'error'; data: null; error: string };

function useFetch<T>(url: string): FetchState<T> {
  const [state, setState] = React.useState<FetchState<T>>({
    status: 'idle',
    data: null,
    error: null,
  });

  React.useEffect(() => {
    if (!url) return;

    let cancelled = false; // Tránh setState sau unmount

    setState({ status: 'loading', data: null, error: null });

    fetch(url)
      .then((res) => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json() as Promise<T>;
      })
      .then((data) => {
        if (!cancelled) setState({ status: 'success', data, error: null });
      })
      .catch((err) => {
        if (!cancelled)
          setState({
            status: 'error',
            data: null,
            error: err instanceof Error ? err.message : 'Unknown error',
          });
      });

    return () => {
      cancelled = true;
    };
  }, [url]);

  return state;
}

// --- Sử dụng ---
interface Product {
  id: string;
  name: string;
  price: number;
}

function ProductList() {
  const state = useFetch<Product[]>('/api/products');
  // TypeScript biết:
  // status === 'success' → data là Product[] (guaranteed, không phải null)
  // status === 'error'   → error là string (guaranteed, không phải null)

  if (state.status === 'loading') return <p>Đang tải...</p>;
  if (state.status === 'error') return <p>Lỗi: {state.error}</p>;
  if (state.status === 'idle') return null;

  // ✅ Ở đây TypeScript biết chắc state.data là Product[]
  return (
    <ul>
      {state.data.map((product) => (
        <li key={product.id}>{product.name}</li>
      ))}
    </ul>
  );
}
```

### Demo 4: Type-safe Context API ⭐⭐⭐

Đây là pattern mà **hầu hết junior devs làm sai** khi kết hợp Context với TypeScript:

```tsx
// ❌ Cách SAI — Buộc phải dùng non-null assertion (!)
interface ThemeContextType {
  theme: 'light' | 'dark';
  toggleTheme: () => void;
}

const ThemeContext = React.createContext<ThemeContextType | undefined>(
  undefined,
);

function useTheme() {
  const context = React.useContext(ThemeContext);
  return context!; // ❌ Nguy hiểm! Nếu dùng ngoài Provider → runtime crash!
}
```

```tsx
// ✅ Cách ĐÚNG — Type-safe với custom error message rõ ràng

interface ThemeContextType {
  theme: 'light' | 'dark';
  toggleTheme: () => void;
}

const ThemeContext = React.createContext<ThemeContextType | undefined>(
  undefined,
);

function useTheme(): ThemeContextType {
  const context = React.useContext(ThemeContext);

  if (context === undefined) {
    // Lỗi này xuất hiện ngay khi dev dùng sai → dễ debug
    throw new Error(
      'useTheme phải được dùng bên trong <ThemeProvider>.\n' +
        'Kiểm tra component tree của bạn.',
    );
  }

  return context; // context: ThemeContextType ✅ (không cần !)
}

// Provider với useMemo + useCallback để tránh re-render không cần thiết
interface ThemeProviderProps {
  children: React.ReactNode;
  defaultTheme?: 'light' | 'dark';
}

function ThemeProvider({
  children,
  defaultTheme = 'light',
}: ThemeProviderProps) {
  const [theme, setTheme] = React.useState<'light' | 'dark'>(defaultTheme);

  const toggleTheme = React.useCallback(() => {
    setTheme((prev) => (prev === 'light' ? 'dark' : 'light'));
  }, []);

  // ⚠️ Quan trọng: memoize value để tránh mọi consumer re-render
  // khi Provider re-render vì lý do khác
  const value = React.useMemo(
    () => ({ theme, toggleTheme }),
    [theme, toggleTheme],
  );

  return (
    <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>
  );
}
```

**Pattern tổng quát — `createSafeContext` helper dùng cho toàn bộ project:**

```tsx
// Viết 1 lần, dùng cho mọi Context

function createSafeContext<T>(displayName: string) {
  const Context = React.createContext<T | undefined>(undefined);
  Context.displayName = displayName; // Hiển thị đẹp trong React DevTools

  function useSafeContext(): T {
    const value = React.useContext(Context);
    if (value === undefined) {
      throw new Error(
        `use${displayName} phải được dùng bên trong <${displayName}Provider>`,
      );
    }
    return value;
  }

  // `as const` → tuple type [Context, hook], không phải array union
  return [Context, useSafeContext] as const;
}

// --- Sử dụng ---
interface AuthContextType {
  user: User | null;
  login: (credentials: { email: string; password: string }) => Promise<void>;
  logout: () => void;
}

const [AuthContext, useAuth] = createSafeContext<AuthContextType>('Auth');

function AuthProvider({ children }: { children: React.ReactNode }) {
```

## 🧪 PHẦN 5: DEBUG LAB (20 phút)

### Bug 1: Generic Inference Thất Bại — Array vs Tuple

```tsx
// ❌ Code có bug — TypeScript không infer đúng:

function createPair<T, U>(first: T, second: U) {
  return [first, second];
  // TypeScript infer: (T | U)[]
  // Thay vì:          [T, U]
}

const pair = createPair('hello', 42);
// pair[0] → string | number ❌ (phải là string)
// pair[1] → string | number ❌ (phải là number)

// ❓ Tại sao? Có bao nhiêu cách sửa?
```

**Giải thích và fix:**

```tsx
// 🔍 VẤN ĐỀ:
// TypeScript infer array literal là (T | U)[]
// vì arrays có thể có bất kỳ độ dài nào với bất kỳ thứ tự nào
// TypeScript không tự biết bạn muốn fixed-length tuple [T, U]

// ✅ Fix 1: Explicit return type
function createPair<T, U>(first: T, second: U): [T, U] {
  return [first, second];
}

// ✅ Fix 2: `as const` — return readonly tuple
function createPair<T, U>(first: T, second: U) {
  return [first, second] as const; // readonly [T, U]
}

const pair = createPair('hello', 42);
const s = pair[0]; // string ✅
const n = pair[1]; // number ✅
```

### Bug 2: Context Value Tạo Mới Mỗi Render

```tsx
// ❌ Performance bug — Tìm nguyên nhân:

interface CounterContextType {
  count: number;
  increment: () => void;
}

const CounterContext = React.createContext<CounterContextType | undefined>(
  undefined,
);

function CounterProvider({ children }: { children: React.ReactNode }) {
  const [count, setCount] = React.useState(0);

  return (
    <CounterContext.Provider
      value={{
        count,
        increment: () => setCount((c) => c + 1),
        // ❌ Object literal và function tạo mới mỗi render!
      }}
    >
      {children}
    </CounterContext.Provider>
  );
}

// ❓ Vấn đề cụ thể là gì?
// ❓ Cách sửa đúng với TypeScript?
```

**Fix:**

```tsx
// 🔍 VẤN ĐỀ:
// Mỗi render của CounterProvider → {} object mới → mọi consumer re-render
// ngay cả khi count không thay đổi (ví dụ parent re-render vì lý do khác)

// ✅ Fix: useMemo + useCallback
function CounterProvider({ children }: { children: React.ReactNode }) {
  const [count, setCount] = React.useState(0);

  const increment = React.useCallback(
    () => setCount((c) => c + 1),
    [], // increment stable — không bao giờ thay đổi
  );

  const value = React.useMemo<CounterContextType>(
    () => ({ count, increment }),
    [count, increment],
    // value thay đổi CHỈ khi count thay đổi
  );

  return (
    <CounterContext.Provider value={value}>{children}</CounterContext.Provider>
  );
}
```

### Bug 3: Quên Handle Case Mới — Exhaustive Check

```tsx
// ❌ Thiếu exhaustive check — Nguy hiểm khi extend union type:

type Status = 'pending' | 'active' | 'suspended';

function getStatusColor(status: Status): string {
  if (status === 'pending') return 'yellow';
  if (status === 'active') return 'green';
  // Quên 'suspended'! TypeScript KHÔNG báo lỗi
  // → Runtime: trả về undefined → lỗi style!
}

// ❓ Làm sao để TypeScript BẮT BUỘC handle mọi case?
```

**Fix với `assertNever`:**

```tsx
// ✅ assertNever — compile error nếu thiếu case

function assertNever(value: never, message?: string): never {
  throw new Error(message ?? `Unhandled case: ${JSON.stringify(value)}`);
}

function getStatusColor(status: Status): string {
  if (status === 'pending') return 'yellow';
  if (status === 'active') return 'green';
  if (status === 'suspended') return 'red';
  return assertNever(status); // status là `never` ở đây ✅
}

// Khi thêm 'blocked' vào Status:
type Status = 'pending' | 'active' | 'suspended' | 'blocked';
// → TypeScript báo lỗi ngay tại assertNever(status)
// → Buộc dev phải handle 'blocked' trước khi compile!

// ✅ Dùng với switch:
function StatusBadge({ status }: { status: Status }) {
  switch (status) {
    case 'pending':
      return <span className='yellow'>Chờ duyệt</span>;
    case 'active':
      return <span className='green'>Hoạt động</span>;
    case 'suspended':
      return <span className='red'>Đình chỉ</span>;
    default:
      return assertNever(status); // Compile error nếu thiếu ✅
  }
}
```

## 💡 SENIOR INSIGHTS

### Cân Nhắc Production

**1. Đừng over-generic — biết khi nào dừng:**

```tsx
// ❌ Over-engineered: Generic không cần thiết
function DisplayName<T extends { name: string }>({ item }: { item: T }) {
  return <span>{item.name}</span>;
}

// ✅ Đủ dùng: Interface đơn giản
interface WithName {
  name: string;
}
function DisplayName({ item }: { item: WithName }) {
  return <span>{item.name}</span>;
}

// Quy tắc: Chỉ dùng generic khi return type
// hoặc behavior phụ thuộc vào T
```

**2. `satisfies` operator — TypeScript 4.9+:**

```tsx
// satisfies: Validate type mà không làm mất literal type inference
const config = {
  theme: 'dark', // type là 'dark', không bị widening thành 'string'
  fontSize: 16,
} satisfies Partial<AppConfig>;

config.theme; // 'dark' ✅ (literal type được giữ nguyên)
// Khác với `as AppConfig` — satisfies không cast, chỉ validate
```

**3. Tổ chức file types trong monorepo:**

```
src/
  types/
    api.ts          ← Types từ API responses (User, Product, Order...)
    components.ts   ← Shared component types (WithId, WithChildren...)
    store.ts        ← State management types
  components/
    Button/
      index.tsx
      Button.types.ts  ← Component-specific types (ButtonProps, variants...)
```

### Câu Hỏi Phỏng Vấn

**Junior:**

> "Utility types như `Partial<T>` và `Omit<T, K>` dùng khi nào?"

_Expected answer:_ `Partial<T>` khi cần form update payload — chỉ truyền fields muốn thay đổi, thường dùng với PATCH. `Omit<T, K>` khi có fields nhạy cảm hoặc server-managed không muốn expose (id, password, createdAt). Ưu điểm lớn: khi base type thay đổi, derived types tự cập nhật — không cần sửa tay nhiều nơi.

**Mid-level:**

> "Giải thích Discriminated Union. Ví dụ thực tế bạn đã áp dụng?"

_Expected answer:_ Discriminated Union là union type trong đó mỗi variant có một field chung (discriminant) với literal value riêng. TypeScript dùng discriminant để tự động narrow type — không cần cast. Ví dụ thực tế: API fetch state (`status: 'idle'|'loading'|'success'|'error'`) — khi check `status === 'success'`, TypeScript biết chắc `data` tồn tại và không phải null. Tôi đã dùng cho shopping cart state machine và alert component variants với props khác nhau theo type.

**Senior:**

> "Tại sao bạn cần `useMemo` và `useCallback` trong Context Provider? Không có thì sao?"

_Expected answer:_ Không có `useMemo` cho value object: mỗi render của Provider tạo `{}` mới → reference change → tất cả consumer re-render kể cả khi data không đổi. Không có `useCallback` cho functions: mỗi render tạo function mới → `useMemo` dependencies thay đổi → value object mới → lại trigger consumer re-render. Kết quả: chain re-renders không cần thiết, performance degradation trong large component trees. Fix: `useCallback` trước cho tất cả functions, rồi `useMemo` bao value object với dependencies là các callback đã stable.

### War Stories

_"Chúng tôi có Notification component dùng boolean props: `isSuccess`, `isError`, `isWarning`, `isInfo`. Một ngày xuất hiện bug kỳ lạ: dev vô tình truyền `isSuccess={true} isError={true}` cùng lúc — component render màu xanh lá với icon ❌, không ai hiểu tại sao._

_Thay vì thêm validation runtime, tôi refactor sang Discriminated Union:_

```tsx
type NotificationProps =
  | { type: 'success'; message: string; onDismiss?: () => void }
  | { type: 'error'; message: string; onRetry: () => void }
  | { type: 'warning'; message: string; onConfirm: () => void }
  | { type: 'info'; message: string };
```

_Sau đó, toàn bộ class bug 'invalid state combination' biến mất. Compiler không cho phép truyền sai — không cần thêm test, không cần documentation, không cần code review checklist._

_Lesson: Thiết kế type system tốt không chỉ giúp DX — nó loại bỏ hoàn toàn một category bugs. Tốt hơn là defensive programming, tốt hơn là unit tests, vì nó hoạt động 100% thời gian ở compile time thay vì catch bugs sau khi đã xảy ra."_

- **Focus management** — focus trap trong modals, skip links, visible focus indicator
- **Keyboard navigation** — Tab, Enter, Space, Arrow keys, Escape
- **TypeScript + A11y** — TS enforce aria attributes ở compile time
