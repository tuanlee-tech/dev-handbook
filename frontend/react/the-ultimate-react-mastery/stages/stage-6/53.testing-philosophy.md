### 1.1 Vấn Đề Thực Tế

**Kịch bản 1: The "Works on My Machine" Problem**

```jsx
// Component bạn vừa viết
function SearchBar({ onSearch }) {
  const [query, setQuery] = useState('');

  const handleSubmit = (e) => {
    e.preventDefault();
    onSearch(query.trim());
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
      />
      <button>Search</button>
    </form>
  );
}

// Bạn test thủ công:
// ✅ Type "react" → click Search → works!
// ✅ Type "  react  " (with spaces) → works!

// Deploy lên production...
// 💥 User nhấn Enter thay vì click button → KHÔNG HOẠT ĐỘNG!
// 💥 User paste text có newlines → UI vỡ!
// 💥 User spam click button → duplicate searches!
```

**Vấn đề:**

- Manual testing chỉ test "happy path"
- Không test edge cases
- Mỗi lần code thay đổi, phải test lại TOÀN BỘ
- Không scale khi app lớn

**Kịch bản 2: The Regression Bug**

```jsx
// Week 1: Bạn viết function
function calculateDiscount(price, couponCode) {
  if (couponCode === 'SAVE10') {
    return price * 0.9;
  }
  return price;
}

// Week 5: PM yêu cầu thêm coupon mới
function calculateDiscount(price, couponCode) {
  if (couponCode === 'SAVE10') {
    return price * 0.9;
  }
  if (couponCode === 'SAVE20') {
    return price * 0.8; // BUG: Forgot to handle case-insensitive!
  }
  return price;
}

// Production bug:
// User dùng "save20" (lowercase) → không được discount
// Worse: Không ai biết 'SAVE10' có còn hoạt động không!
```

**Vấn đề:**

- Thay đổi code → có thể break existing functionality
- Không có safety net
- Fear of refactoring

### 1.2 Giải Pháp: Automated Testing

**Testing là gì?**

```markdown
❌ TESTING KHÔNG PHẢI LÀ:

- Chạy app và click xung quanh
- "Console.log() để check"
- "Nó chạy được là ok rồi"
- Việc làm của QA team (developers không cần test)

✅ TESTING THỰC SỰ LÀ:

- Automated code kiểm tra code của bạn
- Safety net khi refactor
- Living documentation
- Confidence để ship
```

**Value của Testing:**

```jsx
// ❌ KHÔNG CÓ TESTS
function addToCart(item) {
  cart.push(item);
  updateTotal();
}

// Để biết nó hoạt động, bạn phải:
// 1. Chạy app
// 2. Click qua nhiều pages
// 3. Add item
// 4. Check cart
// 5. Verify total
// ⏱️ Mất 2-3 phút mỗi lần test

  addToCart(item);

  expect(getTotal()).toBe(100);

// ⏱️ Test chạy trong < 10ms
// 🔄 Tự động chạy mỗi khi code thay đổi
// 💰 Tiết kiệm hàng giờ đồng hồ
```

**ROI (Return on Investment) của Testing:**

```markdown
Initial Cost:

- ⏱️ 20-30% thời gian viết tests
- 📚 Learning curve
- 🛠️ Setup infrastructure

Long-term Benefits:

- 🐛 Catch bugs sớm (cheaper to fix)
- 🔄 Refactor fearlessly
- 📖 Documentation (tests show usage)
- 🚀 Ship với confidence
- ⏰ Save debugging time
- 👥 Onboard new developers faster

Break-even point: Usually after 2-3 months
```

### 1.3 Mental Model: The Testing Pyramid

**Cấu trúc Testing:**

```
        /\
       /  \
      / E2E \         ← 10% tests (slow, expensive, brittle)
     /--------\
    /          \
   / Integration \    ← 20% tests (medium speed, medium cost)
  /--------------\
 /                \
/   Unit Tests     \  ← 70% tests (fast, cheap, stable)
--------------------

BASE RULE: More unit tests, fewer E2E tests
```

**Chi tiết từng loại:**

```markdown
1️⃣ UNIT TESTS (70%)
What: Test 1 function/component in isolation
Speed: ⚡ Very fast (< 10ms)
Cost: 💰 Cheap to write & maintain
Stability: 🎯 Very stable
When to use: Mọi business logic, utilities, pure functions

Example:

- Test calculateDiscount() function
- Test custom hook useToggle()
- Test utility formatCurrency()

2️⃣ INTEGRATION TESTS (20%)
What: Test multiple components working together
Speed: 🐢 Medium (100-500ms)
Cost: 💰💰 Medium cost
Stability: ⚠️ Moderately stable
When to use: Critical user flows

Example:

- Test form validation + submission + success message
- Test search input + filter + results list
- Test authentication flow

3️⃣ E2E TESTS (10%)
What: Test entire app like a real user
Speed: 🐌 Slow (seconds to minutes)
Cost: 💰💰💰 Expensive
Stability: 💥 Can be flaky
When to use: Critical business paths only

Example:

- Test complete checkout flow
- Test user registration → login → dashboard
- Test payment processing
```

**Anti-pattern: Inverted Pyramid**

```
❌ BAD: Ice Cream Cone Anti-pattern

        /\
       /  \
      /    \
     /  E2E \       ← 70% tests here (DISASTER!)
    /--------\
   /          \
  / Integration \   ← 20% tests
 /--------------\
/   Unit Tests   \  ← 10% tests
------------------

Problems:
- Slow test suite (30+ minutes)
- Flaky tests (random failures)
- Expensive to maintain
- Developers skip running tests
- Low confidence despite many tests
```

### 1.4 Testing Philosophy: Test Behavior, Not Implementation

**❌ BAD: Testing Implementation Details**

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <span data-testid='count'>{count}</span>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

// ❌ Implementation-focused test
test('BAD: uses useState', () => {
  const { result } = renderHook(() => useState(0));
  // Testing React's implementation!
});

test('BAD: calls setCount with correct value', () => {
  const mockSetCount = jest.fn();
  // Testing internal function calls
});

// Why bad?
// - Refactor useState → useReducer → tests break
// - Testing React, not YOUR code
// - Brittle, breaks on every refactor
```

**✅ GOOD: Testing Behavior**

```jsx
// ✅ Behavior-focused test
test('GOOD: increments count when button clicked', () => {
  render(<Counter />);

  const button = screen.getByRole('button', { name: /increment/i });
  const count = screen.getByText('0');

  fireEvent.click(button);

  expect(screen.getByText('1')).toBeInTheDocument();
});

// Why good?
// - Tests what USER sees
// - Can refactor implementation freely
// - Stable, only breaks when behavior changes
// - Documents how component should work
```

**The Golden Rule:**

```markdown
🎯 TEST OUTPUTS, NOT INTERNALS

Ask yourself:
"If I refactor this code completely but keep the same behavior,
should my tests still pass?"

✅ YES → Good test (behavior-focused)
❌ NO → Bad test (implementation-focused)

Examples:
✅ "Clicking submit shows success message"
✅ "Invalid email shows error"
✅ "Total price updates when quantity changes"

❌ "Component uses useState"
❌ "Function calls API with correct params"
❌ "State updates to expected value"
```

## 💻 PHẦN 2: TESTING PATTERNS & STRATEGIES (45 phút)

### Demo 1: AAA Pattern (Arrange, Act, Assert) ⭐

**Pattern cơ bản của mọi test:**

```jsx
test('descriptive test name', () => {
  // 🏗️ ARRANGE: Setup the world
  // - Create test data
  // - Setup mocks
  // - Render components
  // 🎬 ACT: Do the thing
  // - Click button
  // - Call function
  // - Trigger event
  // ✅ ASSERT: Check the result
  // - Verify output
  // - Check side effects
  // - Validate state
});
```

**Ví dụ cụ thể:**

```jsx
// ❌ BAD: Unclear structure
test('discount works', () => {
  expect(calculateDiscount(100, 'SAVE10')).toBe(90);
  const price = 200;
  const result = calculateDiscount(price, 'SAVE20');
  expect(result).toBe(160);
});

// ✅ GOOD: Clear AAA structure
test('applies 10% discount for SAVE10 coupon', () => {
  // Arrange
  const originalPrice = 100;
  const couponCode = 'SAVE10';
  const expectedPrice = 90;

  // Act
  const discountedPrice = calculateDiscount(originalPrice, couponCode);

  // Assert
  expect(discountedPrice).toBe(expectedPrice);
});

test('applies 20% discount for SAVE20 coupon', () => {
  // Arrange
  const originalPrice = 200;
  const couponCode = 'SAVE20';
  const expectedPrice = 160;

  // Act
  const discountedPrice = calculateDiscount(originalPrice, couponCode);

  // Assert
  expect(discountedPrice).toBe(expectedPrice);
});
```

**AAA với React Components:**

```jsx
// ✅ GOOD: AAA for component test
test('shows error message when email is invalid', () => {
  // Arrange
  render(<LoginForm />);
  const emailInput = screen.getByLabelText(/email/i);
  const submitButton = screen.getByRole('button', { name: /submit/i });

  // Act
  fireEvent.change(emailInput, { target: { value: 'invalid-email' } });
  fireEvent.click(submitButton);

  // Assert
  expect(screen.getByText(/invalid email/i)).toBeInTheDocument();
});
```

### Demo 2: Test Naming - Self-Documenting Tests ⭐⭐

**❌ BAD Test Names:**

```jsx
test('test 1', () => {
  /* ... */
});
test('it works', () => {
  /* ... */
});
test('button', () => {
  /* ... */
});
test('should return true', () => {
  /* ... */
});
```

**✅ GOOD Test Names:**

```jsx
// Pattern: "should [expected behavior] when [condition]"
test('should show success message when form submitted with valid data', () => {});
test('should disable submit button when form is invalid', () => {});
test('should clear errors when user starts typing', () => {});

// Pattern: "[Action] [expected result]"
test('clicking delete button removes item from cart', () => {});
test('entering invalid email displays error message', () => {});
test('loading products shows skeleton placeholders', () => {});

// Pattern for edge cases: "[Function] [edge case] [result]"
test('calculateDiscount handles negative prices by returning 0', () => {});
test('formatCurrency handles null by returning empty string', () => {});
test('searchProducts handles empty query by returning all products', () => {});
```

**Template:**

```markdown
TEMPLATE FOR TEST NAMES:

Format 1: "should [expected behavior] when [condition]"
Format 2: "[action] [expected result]"
Format 3: "[feature] with [scenario] [expected outcome]"

Good test name checklist:
✅ Describes WHAT is being tested
✅ Describes EXPECTED behavior
✅ Describes WHEN/WHY it happens
✅ Readable as a sentence
✅ No need to read test code to understand

Bad test name checklist:
❌ Vague (test 1, it works)
❌ Implementation detail (uses useState)
❌ Too short (button)
❌ Too long (> 100 characters)
```

### Demo 3: What to Test vs What NOT to Test ⭐⭐⭐

**🎯 WHAT TO TEST:**

```jsx
// ✅ 1. Business Logic
function calculateShippingCost(items, country) {
  const baseRate = country === 'US' ? 5 : 10;
  const itemCount = items.length;
  const total = baseRate + itemCount * 2;

  return itemCount > 10 ? total * 0.9 : total;
}

// TEST THIS! Critical business rules

// ✅ 2. User Interactions
function TodoApp() {
  // Test: Adding todo
  // Test: Completing todo
  // Test: Filtering todos
}

// ✅ 3. Edge Cases
function divide(a, b) {
  if (b === 0) return 0; // Edge case!
  return a / b;
}
// TEST: divide by zero
// TEST: negative numbers
// TEST: very large numbers

// ✅ 4. Conditional Rendering
function UserProfile({ user }) {
  if (!user) return <LoginPrompt />;
  if (user.isPremium) return <PremiumProfile />;
  return <BasicProfile />;
}
// TEST: Each branch

// ✅ 5. Custom Hooks
function useLocalStorage(key, initialValue) {
  // Complex logic
  return [value, setValue];
}
// TEST THIS! Reusable logic
```

**🚫 WHAT NOT TO TEST:**

```jsx
// ❌ 1. Third-party Libraries
test('useState updates state', () => {
  // Don't test React's implementation!
});

test('fetch makes HTTP request', () => {
  // Don't test browser APIs!
});

// ❌ 2. Implementation Details
function Component() {
  const [isOpen, setIsOpen] = useState(false);
  // ...
}

test('uses useState hook', () => {
  // Don't care HOW it stores state
});

// ❌ 3. Trivial Code
function add(a, b) {
  return a + b;
}
// Too simple, not worth testing

function HelloWorld() {
  return <div>Hello World</div>;
}
// Static output, no logic

// ❌ 4. Styles/CSS
test('button has blue background', () => {
  // CSS is visual, use visual regression testing
});

// ❌ 5. Console logs / Debug code
function Component() {
  console.log('rendered'); // Don't test this
  useEffect(() => {
    console.log('effect ran'); // Don't test this
  }, []);
}
```

**Decision Tree: Should I Test This?**

```markdown
                    START
                      |
           Does it have logic?
                 /        \
               NO          YES
                |           |
         DON'T TEST    Is it YOUR code?
                          /        \
                        NO          YES
                         |           |
                    DON'T TEST   Can users see/use it?
                                   /        \
                                 NO          YES
                                  |           |
                          Is it critical?    TEST IT!
                              /        \
                            NO          YES
                             |           |
                        MAYBE TEST    TEST IT!
```

**Coverage Guidelines:**

```markdown
🎯 TARGET COVERAGE:

Critical Code (business logic, payments, auth):
→ 100% coverage required

Important Features (forms, dashboards):
→ 80-90% coverage

Nice-to-have Features:
→ 60-70% coverage

Utilities/Helpers:
→ 90-100% coverage (they're reused everywhere!)

⚠️ WARNING:
"100% coverage" ≠ "bug-free code"
"0% coverage" ≠ "broken code"

Quality > Quantity
Better: 50% coverage of RIGHT things
Worse: 100% coverage of WRONG things
```

```jsx
// CODE TO ANALYZE:
function SearchableProductList({ products, categories }) {
  const [searchTerm, setSearchTerm] = useState('');
  const [selectedCategory, setSelectedCategory] = useState('all');
  const [sortBy, setSortBy] = useState('name');

  const filteredProducts = useMemo(() => {
    let result = products;

    // Filter by search term
    if (searchTerm) {
      result = result.filter((p) =>
        p.name.toLowerCase().includes(searchTerm.toLowerCase()),
      );
    }

    // Filter by category
    if (selectedCategory !== 'all') {
      result = result.filter((p) => p.category === selectedCategory);
    }

    // Sort
    result.sort((a, b) => {
      if (sortBy === 'name') return a.name.localeCompare(b.name);
      if (sortBy === 'price') return a.price - b.price;
      return 0;
    });

    return result;
  }, [products, searchTerm, selectedCategory, sortBy]);

  if (products.length === 0) {
    return <EmptyState message='No products available' />;
  }

  return (
    <div>
      <input
        placeholder='Search products...'
        value={searchTerm}
        onChange={(e) => setSearchTerm(e.target.value)}
      />

      <select
        value={selectedCategory}
        onChange={(e) => setSelectedCategory(e.target.value)}
      >
        <option value='all'>All Categories</option>
        {categories.map((cat) => (
          <option
            key={cat}
            value={cat}
          >
            {cat}
          </option>
        ))}
      </select>

      <select
        value={sortBy}
        onChange={(e) => setSortBy(e.target.value)}
      >
        <option value='name'>Sort by Name</option>
        <option value='price'>Sort by Price</option>
      </select>

      <div>
        {filteredProducts.length === 0 ? (
          <p>No products match your filters</p>
        ) : (
          filteredProducts.map((product) => (
            <ProductCard
              key={product.id}
              product={product}
            />
          ))
        )}
      </div>
    </div>
  );
}
```

## 🧪 PHẦN 5: DEBUG LAB (20 phút)

### Bug 1: Flaky Test

**Problematic Test:**

```jsx
test('shows notification after 2 seconds', () => {
  render(<NotificationComponent />);

  // Wait 2 seconds
  setTimeout(() => {
    expect(screen.getByText('Notification!')).toBeInTheDocument();
  }, 2000);
});

// ❌ PROBLEM: Test sometimes passes, sometimes fails
// Why? setTimeout in test doesn't wait!
```

**Questions:**

1. Tại sao test này flaky?
2. Làm sao fix?
3. Best practice cho async testing?

❌ VẤN ĐỀ:

Code test không thực sự chờ setTimeout!

- Test kết thúc ngay lập tức
- Assertion (hàm kiểm tra) chạy trước khi notification xuất hiện
- Xảy ra tình trạng tranh chấp dữ liệu (Race condition)

✅ CÁCH FIX:

Cách 1: Sử dụng các hàm async tiện ích của testing library

```jsx
test('shows notification after delay', async () => {
  render(<NotificationComponent />);

  // Chờ element xuất hiện
  const notification = await screen.findByText('Notification!');
  expect(notification).toBeInTheDocument();
});
```

Cách 2: Mock timers (Giả lập thời gian)

```jsx
test('shows notification after 2 seconds', () => {
  jest.useFakeTimers();
  render(<NotificationComponent />);

  // Tua nhanh thời gian
  jest.advanceTimersByTime(2000);

  expect(screen.getByText('Notification!')).toBeInTheDocument();

  jest.useRealTimers();
});
```

🎯 BEST PRACTICES:

1. Không bao giờ sử dụng setTimeout thật trong các bài test
2. Sử dụng findBy\* cho các element bất đồng bộ (async elements)
3. Sử dụng waitFor cho các trường hợp chờ đợi phức tạp
4. Mock timers khi test các đoạn code dựa trên thời gian
5. Luôn luôn dọn dẹp sau khi test (useRealTimers)

### Bug 2: False Positive Test

```jsx
function addToCart(item) {
  // BUG: Function does nothing!
  console.log('Adding item:', item);
}

test('adds item to cart', () => {
  const item = { id: 1, name: 'Product' };
  addToCart(item);

  expect(true).toBe(true); // ✅ Test passes!
});

// ❌ PROBLEM: Test passes but function is broken!
```

**Questions:**

1. Tại sao test pass nhưng code broken?
2. Làm sao fix test này?
3. How to prevent false positives?

❌ VẤN ĐỀ:

Test không thực sự xác thực bất kỳ điều gì!

- Kỳ vọng true === true (luôn luôn đúng)
- Không kiểm tra xem item đã được thêm vào hay chưa
- Lỗi False positive (Kết quả đúng giả tạo)

✅ CÁCH FIX:

```jsx
// Đầu tiên, sửa lại hàm:
let cart = [];

function addToCart(item) {
  cart.push(item);
}

// Sau đó, sửa lại bài test:
test('adds item to cart', () => {
  const item = { id: 1, name: 'Product' };

  // Xóa trống giỏ hàng trước khi test
  cart = [];

  // Act (Thực hiện)
  addToCart(item);

  // Assert (Xác thực) hành vi THỰC TẾ
  expect(cart).toContain(item);
  expect(cart.length).toBe(1);
});
```

🎯 CÁCH PHÒNG TRÁNH:

1. Luôn xác thực (assert) những thứ có ý nghĩa
2. Tránh sử dụng `expect(true).toBe(true)`
3. Kiểm tra kết quả đầu ra thực tế
4. Sử dụng các hàm so khớp (matchers) cụ thể
5. Hãy tự hỏi: "Điều gì sẽ hỏng nếu bài test này thất bại?"

Các pattern gây ra false positive phổ biến:

- ❌ `expect(true).toBeTruthy()`
- ❌ `expect(result).toBeDefined()` (mà không kiểm tra giá trị cụ thể)
- ❌ `expect(mockFn).toHaveBeenCalled()` (mà không kiểm tra đối số truyền vào)

Cách tốt hơn:

- ✅ `expect(cart).toContain(item)`
- ✅ `expect(result).toBe(expectedValue)`
- ✅ `expect(mockFn).toHaveBeenCalledWith(specificArgs)`

### Bug 3: Over-mocking

```jsx
function fetchUserProfile(userId) {
  return fetch(`/api/users/${userId}`)
    .then((res) => res.json())
    .then((data) => formatUserData(data));
}

test('fetches user profile', () => {
  // Mock EVERYTHING
  global.fetch = jest.fn(() =>
    Promise.resolve({
      json: () => Promise.resolve(mockData),
    }),
  );

  const formatUserData = jest.fn((data) => data);

  const result = fetchUserProfile(1);

  expect(result).toBeDefined();
});

// ❌ PROBLEM: Mocking too much, not testing real behavior
```

**Questions:**

1. What's wrong with this test?
2. What's NOT being tested?
3. How to improve?

````markdown
❌ VẤN ĐỀ:

1. Mock hàm `formatUserData`
   - Không kiểm tra xem nó có thực sự được gọi hay không!
   - Không test logic bên trong của nó!
2. Lạm dụng mock hàm `fetch` (Over-mocking)
   - Phải thiết lập (setup) quá nhiều dòng code phức tạp
   - Khó bảo trì
3. Assertion (hàm kiểm tra) quá yếu
   - `expect(result).toBeDefined()` không mang lại giá trị xác thực nào cả

✅ CÁCH TIẾP CẬN TỐT HƠN:

Cách 1: Mock tại ranh giới API (Sử dụng MSW)

```jsx
import { rest } from 'msw';
import { server } from './mocks/server';

test('fetches and formats user profile', async () => {
  // Mock response của API
  server.use(
    rest.get('/api/users/1', (req, res, ctx) => {
      return res(
        ctx.json({
          id: 1,
          first_name: 'John',
          last_name: 'Doe',
        }),
      );
    }),
  );

  // Test hàm thực tế
  const result = await fetchUserProfile(1);

  // Xác thực kết quả đầu ra thực tế
  expect(result).toEqual({
    id: 1,
    fullName: 'John Doe', // Hàm formatUserData được chạy thật!
  });
});
```
````

Cách 2: Tách biệt các mối bận tâm (Separate concerns)

```jsx
// Test riêng phần fetch
test('fetchUser calls correct endpoint', async () => {
  // Mock fetch (tại ranh giới)
  // Test URL, method, headers
});

// Test riêng phần định dạng dữ liệu (Unit test)
test('formatUserData formats correctly', () => {
  const input = { first_name: 'John', last_name: 'Doe' };
  const output = formatUserData(input);

  expect(output).toEqual({ fullName: 'John Doe' });
});
```

🎯 HƯỚNG DẪN (GUIDELINES):

Nên Mock:

- ✅ Các cuộc gọi mạng (fetch, axios)
- ✅ Các dịch vụ bên ngoài (analytics, thanh toán)
- ✅ Hệ thống tệp (File system)
- ✅ Ngày/giờ (để đảm bảo kết quả cố định - determinism)

Không nên mock:

- ❌ Các hàm do chính bạn viết (hãy test chúng!)
- ❌ Các React hooks
- ❌ Các hàm tiện ích (Utilities) bạn tự phát triển
- ❌ Mock mọi thứ "chỉ vì thích/thói quen"

QUY TẮC: Mock tại ranh giới (boundaries), test hành vi thực tế (real behavior)

## 💡 SENIOR INSIGHTS

### Cân Nhắc Production

**1. The Coverage Trap:**

````markdown
❌ MYTH: "100% coverage = không có bug"

✅ THỰC TẾ:

- Coverage đo lường số dòng code được thực thi, chứ không đo lường tính đúng đắn
- Bạn vẫn có thể đạt 100% coverage với những bài test tồi
- Bug vẫn có thể tồn tại trong những đoạn code đã được cover 100%

Ví dụ:

```javascript
function divide(a, b) {
  return a / b; // ✅ Đạt 100% covered
}

test('divides numbers', () => {
  expect(divide(10, 2)).toBe(5); // ✅ Test pass
});
// 🐛 BUG: `divide(10, 0) = Infinity` (không được phát hiện!)
```
````

Mục tiêu tốt hơn: "Kiểm thử có ý nghĩa cho các luồng quan trọng"

**2. Testing in Agile:**

```markdown
KỊCH BẢN: Product muốn có tính năng mới vào ngày mai

Option 1: Bỏ qua test, ship nhanh

- ✅ Ship ngay hôm nay
- ❌ Lỗi trên production
- ❌ Sợ refactor code
- ❌ Nợ kỹ thuật (Technical debt)

Option 2: Viết toàn bộ test, ship chậm

- ❌ Trễ deadline
- ✅ Chất lượng cao
- ❌ Chi phí cơ hội (Opportunity cost)

Option 3: Kiểm thử chiến lược (KHUYẾN NGHỊ)

- ✅ Ship đúng hạn
- ✅ Các luồng quan trọng (critical paths) đã được cover
- ✅ Có thể refactor an toàn
- ⚠️ Chấp nhận một vài rủi ro ở các case đặc biệt (edge cases)

QUY TẮC: Test luồng quan trọng, ship, rồi cải thiện dần (iterate)
```

**3. Test Maintenance Cost:**

```markdown
REALITY: Test cũng là code và cần được bảo trì

Chi phí (Costs):

- ⏱️ Thời gian viết (Chiếm 20-30% thời gian phát triển)
- 🔧 Thời gian bảo trì (Dễ gãy khi thay đổi cấu trúc code/refactors)
- 🏃 Thời gian chạy (Làm chậm vòng phản hồi/feedback loop)
- 🧠 Khối lượng nhận thức (Phải đọc hiểu cả code test)

Công thức tính ROI (Hiệu quả đầu tư):

- Chi phí viết test: 30 phút
- Chi phí test thủ công (manual): 5 phút mỗi lần
- Số lần chạy test: 100 lần

Điểm hòa vốn: Sau 6 lần test thủ công
Sau cột mốc đó: Sinh lời hoàn toàn!

GÓC NHÌN: Nhiều test hơn ≠ tốt hơn
Test đúng chỗ = tốt hơn
```

### Câu Hỏi Phỏng Vấn

**Junior Level:**

**Q1:** "Giải thích sự khác biệt giữa Unit test và Integration test?"

**Expected Answer:**

```markdown
Unit Test:

- Test 1 hàm/component một cách độc lập (isolation)
- Tốc độ nhanh (< 10ms)
- Không có phụ thuộc (No dependencies)
- Ví dụ: Test hàm calculateTotal()

Integration Test:

- Test nhiều thành phần (components) phối hợp hoạt động với nhau
- Tốc độ trung bình (100-500ms)
- Có các phụ thuộc (có thể sử dụng API thật kết hợp với mock)
- Ví dụ: Test luồng gửi form → gọi API → hiển thị thông báo thành công

When to use:

- Unit: Logic nghiệp vụ (business logic), hàm tiện ích (utilities), hàm thuần túy (pure functions)
- Integration: Luồng công việc của người dùng (user workflows), các luồng quan trọng (critical paths)
```

```markdown
- Các phụ thuộc phức tạp (Complex dependencies)

STRATEGY:

1. Cấp độ Dịch vụ / Service Level (70%):

- Các bài test unit (logic nghiệp vụ)
- Các bài test tích hợp (DB, message queue)
- Các bài test hợp đồng / Contract tests (API contracts)
- Không mock các dịch vụ khác

2. Cấp độ Tích hợp / Integration Level (20%):

- Kiểm thử theo từng cặp dịch vụ
- Mock các dịch vụ phía hạ nguồn (downstream services)
- Kiểm thử các kịch bản lỗi (dịch vụ bị sập)

3. Cấp độ End-to-End (10%):

- Chỉ áp dụng cho các luồng nghiệp vụ quan trọng
- Toàn bộ hệ thống (Full stack), tất cả dịch vụ
- Tốn kém chi phí, chạy với tần suất ít hơn

CONTRACT TESTING:

- Sử dụng Pact hoặc công cụ tương tự
- Provider tests (Xác thực API khớp với hợp đồng)
- Consumer tests (Xác thực việc sử dụng API chính xác)
- Ngăn ngừa các vấn đề khi tích hợp

MONITORING:

- Test chỉ chứng minh hệ thống đã hoạt động ĐÚNG MỘT LẦN
- Giám sát trên production mới chứng minh hệ thống hoạt động NGAY LÚC NÀY
- Kết hợp: Kiểm thử (Tests) + Khả năng quan sát (Observability)
```

**Q6:** "Cách thuyết phục team đầu tư vào việc kiểm thử?"

**Expected Answer:**

```markdown
PITCH TO STAKEHOLDERS ( TRÌNH BÀY Ý TƯỞNG CHO CÁC BÊN LIÊN QUAN ):

Dành cho Doanh nghiệp (Business):
"Kiểm thử giúp giảm 10 lần chi phí sửa lỗi:

- Lỗi trong lúc phát triển: Mất 1 giờ để sửa
- Lỗi trên production: Mất 10 giờ để sửa + ảnh hưởng tới khách hàng

ROI: Sau 2-3 tháng, việc viết test giúp TIẾT KIỆM thời gian"

Dành cho Lập trình viên (Developers):
"Kiểm thử giúp mang lại:

- Khả năng refactor code không sợ hãi
- Gỡ lỗi (debugging) nhanh hơn (test chỉ rõ vị trí gặp lỗi)
- Thiết kế code tốt hơn (code viết test được = code có tính module cao)
- Tài liệu hướng dẫn (các bài test cho thấy cách sử dụng code)"

Dành cho Quản lý Sản phẩm (Product):
"Kiểm thử giúp mang lại:

- Ship tính năng nhanh hơn (tự tin khi triển khai/deploy)
- Trải nghiệm người dùng (UX) tốt hơn (ít lỗi hơn)
- Có nhiều tính năng hơn (tốn ít thời gian sửa lỗi hơn)"

STRATEGY:

- Bắt đầu từ quy mô nhỏ (chỉ tập trung vào các luồng quan trọng)
- Chứng minh kết quả (những lỗi đã bắt được)
- Xây dựng văn hóa từ từ
- Tuyên dương các thành quả (khi test giúp phát hiện ra bug)

WRONG APPROACH:
❌ "Chúng ta cần đạt 100% coverage"
❌ "Đây là tiêu chuẩn của ngành"
❌ Ép buộc áp dụng khi chưa có sự đồng thuận

RIGHT APPROACH:
✅ "Test giúp chúng ta tiết kiệm thời gian"
✅ "Test đã giúp phát hiện ra lỗi"
✅ "Test giúp chúng ta phát triển tính năng thuận lợi hơn"
```

### War Stories

**Story 1: The 100% Coverage Disaster (Thảm họa)**

````markdown
🔥 TÌNH HUỐNG THỰC TẾ:
CTO đưa ra quy định: "Yêu cầu 100% coverage, nếu không sẽ từ chối PR"

ĐIỀU GÌ ĐÃ XẢY RA:

- Các lập trình viên viết những bài test vô nghĩa

```javascript
test('component exists', () => {
  expect(wrapper.exists()).toBe(true);
});
```

- Coverage: 100% ✅
- Chất lượng: Tồi tệ ❌
- Bug vẫn lọt lên production
- Tinh thần của đội ngũ đi xuống

💡 BÀI HỌC rút ra:
"Chỉ số coverage không có ý nghĩa gì cả.
Chất lượng bài test mới là điều quan trọng hơn.
Tốt hơn là: Đạt 60% coverage nhưng đúng những phần CẦN THIẾT."

CÁCH KHẮC PHỤC:

- Thay đổi quy định thành: "Các luồng quan trọng phải được kiểm thử"
- Quy trình code review bao gồm cả việc đánh giá chất lượng bài test
- Các bài test phải thất bại (fail) nếu tính năng bị hỏng
````

**Story 2: The Flaky E2E Hell**

```markdown
🔥 TÌNH HUỐNG THỰC TẾ:
Bộ test E2E: Chạy mất 45 phút, tỷ lệ thất bại (fail) là 30%

HÀNH VI CỦA LẬP TRÌNH VIÊN:

- "Cứ chạy lại (re-run) xem sao"
- Bỏ qua test để merge nhanh hơn
- Lòng tin bị xói mòn
- Các bài test trở nên vô dụng

NGUYÊN NHÂN GỐC RỄ:

- Hết thời gian chờ mạng (Network timeouts)
- Tình trạng tranh chấp dữ liệu (Race conditions)
- Dùng chung cơ sở dữ liệu test (Shared test database)
- Cài đặt cứng thời gian chờ (Hardcoded waits)

💡 BÀI HỌC rút ra:
"Chỉ một bài test chập chờn (flaky test) cũng có thể phá hủy lòng tin vào toàn bộ bộ test.
Hãy sửa ngay lập tức hoặc xóa bỏ nó đi."

CÁCH KHẮC PHỤC:

- Chuyển đổi 70% bài test E2E → Test tích hợp (Integration tests)
- Sửa các lỗi race conditions (sử dụng các cơ chế chờ phù hợp)
- Cô lập dữ liệu test (Isolated test data)
- Bộ test hiện tại: Chạy mất 8 phút, độ ổn định đạt 99%
```

**Story 3: The Mocking Madness**

````markdown
🔥 TÌNH HUỐNG THỰC TẾ:
Bài test chứa 100 dòng code mock, nhưng chỉ có 2 dòng xác thực (assertion)

```javascript
const mockFetch = jest.fn();
const mockUseState = jest.fn();
const mockUseEffect = jest.fn();
const mockComponent = jest.fn();
// ... và 90 dòng mock khác
```

💥 VẤN ĐỀ:

- Test bị gãy (fail) sau mỗi lần refactor code
- Tốn nhiều thời gian để sửa bài test hơn là sửa code tính năng
- Dữ liệu mock bị xa rời thực tế (drifted from reality)
- Sự tự tin ảo (False confidence)

💡 BÀI HỌC rút ra:
"Lạm dụng mock (Over-mocking) là bạn đang test cấu trúc code (implementation) chứ không phải hành vi (behavior).
Hãy mock tại ranh giới hệ thống, và test code thật."

CÁCH KHẮC PHỤC:

- Chuyển sang dùng MSW (mock API ở tầng mạng, không mock React)
- Sử dụng các component thật để test
- Bộ test hiện tại: Chỉ còn 10 dòng, nhưng mang lại giá trị cao hơn nhiều
````
